Objects In Kubernetes

이 페이지는 Kubernetes 객체가 Kubernetes API에서 어떻게 표현되는지, 그리고 .yaml 형식으로 어떻게 표현할 수 있는지에 대해 설명합니다.

Understanding Kubernetes objects

Kubernetes 객체는 Kubernetes 시스템의 지속적인 엔터티입니다. Kubernetes는 이러한 엔터티를 사용하여 클러스터의 상태를 나타냅니다. 구체적으로, 이들은 다음을 설명할 수 있습니다:

  • 어떤 컨테이너화된 애플리케이션이 실행 중인지(그리고 어떤 노드에서 실행 중인지)
  • 해당 애플리케이션에 사용 가능한 리소스
  • 애플리케이션의 동작에 대한 정책, 예를 들어 다시 시작 정책, 업그레이드 및 내결함성

Kubernetes 객체는 “의도의 기록”입니다. 한 번 객체를 생성하면 Kubernetes 시스템은 해당 객체가 존재하는지 계속 확인합니다. 객체를 생성함으로써 실제로는 Kubernetes 시스템에 클러스터 작업 부하가 어떻게 보이길 원하는지 알려주는 것입니다. 이것이 클러스터의 원하는 상태입니다.

Kubernetes 객체를 사용하려면(생성, 수정 또는 삭제) Kubernetes API를 사용해야 합니다. 예를 들어 kubectl 명령줄 인터페이스를 사용할 때 CLI는 필요한 Kubernetes API 호출을 대신 수행합니다. 또한 Client Libraries 중 하나를 사용하여 직접 프로그램에서 Kubernetes API를 사용할 수도 있습니다.

Object spec and status

거의 모든 Kubernetes 객체에는 객체의 구성을 관리하는 두 개의 중첩된 객체 필드가 포함되어 있습니다: 객체 spec 및 객체 status. spec을 가진 객체에 대해서는 객체를 생성할 때 이를 설정해야 하며, 리소스가 가져야 하는 특성에 대한 설명을 제공하여 원하는 상태를 나타냅니다.

status는 Kubernetes 시스템 및 그 구성 요소에 의해 제공 및 업데이트되는 객체의 현재 상태를 설명합니다. Kubernetes 제어 플레인은 지속적으로 모든 객체의 실제 상태를 활발하게 관리하여 제공한 원하는 상태와 일치하도록 합니다.

예를 들어, Kubernetes에서 Deployment는 클러스터에서 실행되는 애플리케이션을 나타낼 수 있는 객체입니다. Deployment를 생성할 때, Deployment spec을 설정하여 해당 애플리케이션의 세 개의 복제본이 실행되도록 할 수 있습니다. Kubernetes 시스템은 Deployment spec을 읽고 원하는 애플리케이션의 세 개의 인스턴스를 시작하여 상태를 귀하의 spec과 일치하도록 업데이트합니다. 이러한 인스턴스 중 하나라도 실패하면(상태 변경), Kubernetes 시스템은 spec과 상태 간의 차이에 응답하여 교정을 수행합니다. 이 경우, 대체 인스턴스를 시작합니다.

객체의 spec, status 및 metadata에 대한 자세한 정보는 Kubernetes API Conventions를 참조하십시오.

Describing a Kubernetes object

Kubernetes에서 객체를 생성할 때, 원하는 상태를 설명하는 객체 spec과 객체에 대한 몇 가지 기본 정보(예: 이름)를 제공해야 합니다. Kubernetes API를 사용하여 객체를 생성할 때(직접 또는 kubectl을 통해), 해당 API 요청은 요청 본문에서 JSON으로 이러한 정보를 포함해야 합니다. 대부분의 경우, kubectl에 정보를 제공하는 파일로서 manifest라고 하는 것이 일반적입니다. 관례적으로, manifest는 YAML로 작성됩니다(JSON 형식을 사용할 수도 있음). kubectl과 같은 도구는 HTTP를 통해 API 요청을 할 때 manifest에서 정보를 JSON 또는 다른 지원되는 직렬화 형식으로 변환합니다.

다음은 Kubernetes 배포(Deployment)의 필수 필드와 객체 spec을 보여주는 예제 manifest입니다.

application/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 2 # tells deployment to run 2 pods matching the template
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80

위와 같은 manifest 파일을 사용하여 배포(Deployment)를 생성하는 한 가지 방법은 kubectl 명령줄 인터페이스에서 kubectl apply 명령을 사용하여 .yaml 파일을 인자로 전달하는 것입니다. 다음은 예시입니다:

kubectl apply -f https://k8s.io/examples/application/deployment.yaml

The output is similar to this:

deployment.apps/nginx-deployment created

Required fields

생성하려는 Kubernetes 객체의 manifest(YAML 또는 JSON 파일)에서 다음 필드의 값을 설정해야 합니다:

  • apiVersion - 이 객체를 생성하는 데 사용하는 Kubernetes API의 버전
  • kind - 생성하려는 객체의 종류
  • metadata - 객체를 고유하게 식별하는 데 도움이 되는 데이터로, 이름 문자열, UID 및 선택적인 namespace를 포함합니다.
  • spec - 객체에 대한 원하는 상태

각 Kubernetes 객체마다 객체 spec의 정확한 형식은 다르며 해당 객체에 특정한 중첩된 필드를 포함합니다. Kubernetes API Reference를 통해 Kubernetes를 사용하여 생성할 수 있는 모든 객체의 spec 형식을 찾을 수 있습니다.

예를 들어, Pod API 참조에서 spec field를 살펴보세요. 각 Pod에 대해 .spec 필드는 해당 Pod와 그 원하는 상태(예: 해당 Pod 내의 각 컨테이너에 대한 컨테이너 이미지 이름)를 지정합니다. 또 다른 예로는 StatefulSet API의 spec field가 있습니다. StatefulSet에 대해 .spec 필드는 StatefulSet 및 해당 원하는 상태를 지정합니다. StatefulSet의 .spec 내부에는 Pod 객체에 대한 template이 포함되어 있습니다. 이 템플릿은 StatefulSet 컨트롤러가 StatefulSet 사양을 충족하기 위해 생성할 Pod를 설명합니다. 서로 다른 종류의 객체는 서로 다른 .status를 가질 수도 있습니다. 다시 말해, API 참조 페이지에서 각 다른 유형의 객체에 대한 .status 필드의 구조와 내용이 상세히 설명되어 있습니다.

참고: YAML 구성 파일 작성에 대한 추가 정보는 Configuration Best Practices를 참조하십시오.

Server side field validation

Kubernetes v1.25부터 API 서버는 객체에서 인식되지 않거나 중복된 필드를 감지하는 서버 측 field validation를 제공합니다. 이는 서버 측에서 kubectl --validate의 모든 기능을 제공합니다.

kubectl 도구는 필드 유효성 검사의 수준을 설정하기 위해 --validate 플래그를 사용합니다. 이 플래그는 ignore, warn 및 strict 값을 허용하며, true (strict와 동일) 및 false (ignore와 동일) 값을도 허용합니다. kubectl의 기본 검증 설정은 --validate=true입니다.

Strict

Strict 필드 유효성 검사는 유효성 검사 실패 시 오류를 발생시킵니다.

Warn

필드 유효성 검사가 수행되지만 오류는 요청을 실패시키는 대신 경고로 노출됩니다.

Ignore

서버 측 필드 유효성 검사가 수행되지 않습니다.

kubectl이 필드 유효성 검사를 지원하는 API 서버에 연결할 수 없는 경우 클라이언트 측 유효성 검사를 사용하도록 되돌아갑니다. Kubernetes 1.27 이상의 버전은 항상 필드 유효성 검사를 제공합니다. 이전 버전의 Kubernetes는 제공하지 않을 수 있습니다. 클러스터가 v1.27보다 오래된 경우 해당 Kubernetes 버전의 문서를 확인하십시오.
