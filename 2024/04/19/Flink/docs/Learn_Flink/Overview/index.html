Learn Flink: Hands-On Training

Goals and Scope of this Training

이 교육은 Apache Flink의 소개를 제공하여 확장 가능한 스트리밍 ETL, 분석 및 이벤트 기반 응용 프로그램을 작성하는 데 충분한 내용을 포함하고 있으며, 매우 중요한 많은 세부 정보를 생략합니다. 핵심은 Flink의 상태 및 시간을 관리하기 위한 API에 대한 직관적인 소개를 제공하는 것으로, 이 기본 사항을 마스터한 후에는 더 자세한 참조 문서에서 필요한 내용을 더 잘 이해할 수 있을 것으로 기대됩니다. 각 섹션의 끝에 있는 링크는 더 자세히 학습할 수 있는 곳으로 이어집니다.

구체적으로 다음을 배우게 될 것입니다:

  • 스트리밍 데이터 처리 파이프라인을 구현하는 방법
  • Flink가 상태를 관리하는 방법과 이유
  • 이벤트 시간을 사용하여 정확한 분석을 일관되게 계산하는 방법
  • 지속적인 스트림에서 이벤트 기반 응용 프로그램을 구축하는 방법
  • Flink가 정확히 한 번 의미론을 가진 내결함성 상태 스트림 처리를 제공하는 방법

이 교육은 스트리밍 데이터의 연속 처리, 이벤트 시간, 상태를 가진 스트림 처리, 상태 스냅샷이라는 네 가지 핵심 개념에 중점을 두고 있습니다. 이 페이지에서는 이러한 개념을 소개합니다.

Note: 이 교육에는 제시된 개념을 학습하는 데 안내하는 실습 연습 세트가 함께 제공됩니다. 각 섹션의 끝에 해당하는 실습 링크가 제공됩니다.

Stream Processing

스트림은 데이터의 자연적인 환경입니다. 웹 서버의 이벤트, 주식 거래, 또는 공장 바닥의 기계에서의 센서 읽기와 같은 데이터는 스트림의 일부로 생성됩니다. 그러나 데이터를 분석할 때, 유계 또는 무계 스트림을 중심으로 처리를 구성할 수 있으며, 선택한 패러다임에는 깊은 영향이 있습니다.

Batch processing는 유한 데이터 스트림을 처리할 때 사용되는 패러다임입니다. 이 작업 모드에서는 결과를 생성하기 전에 전체 데이터 집합을 수용할 수 있으므로, 예를 들어 데이터를 정렬하거나 전역 통계를 계산하거나 모든 입력을 요약하는 최종 보고서를 생성하는 등의 작업이 가능합니다.

한편, Stream processing는 무계 데이터 스트림을 포함합니다. 개념적으로, 최소한 입력이 끝나지 않을 수 있으며, 그러므로 데이터가 도착하는 대로 지속적으로 처리해야 합니다.

Flink에서 애플리케이션은 사용자 정의 operators에 의해 변환될 수 있는 streaming dataflows로 구성됩니다. 이러한 데이터 플로우는 하나 이상의 sources에서 시작하여 하나 이상의 sinks로 끝나는 방향성 그래프를 형성합니다.

프로그램의 변환과 데이터 플로우의 연산자 사이에는 종종 일대일 대응이 있습니다. 그러나 때로는 하나의 변환에 여러 연산자가 포함될 수도 있습니다.

애플리케이션은 Apache Kafka 또는 Kinesis와 같은 메시지 대기열이나 분산 로그와 같은 스트리밍 소스에서 실시간 데이터를 소비할 수 있습니다. 그러나 Flink는 다양한 데이터 소스에서 유계, 과거 데이터를 소비할 수도 있습니다. 마찬가지로, Flink 애플리케이션에서 생성되는 결과 스트림은 다양한 시스템으로 전송되어 싱크로 연결될 수 있습니다.

Parallel Dataflows

Flink의 프로그램들은 본질적으로 병렬적이고 분산됩니다. 실행 중에는 스트림에 하나 이상의 stream partitions이 있으며, 각 연산자에는 하나 이상의 operator subtasks이 있습니다. 연산자 하위 작업은 서로 독립적이며, 다른 스레드에서 실행되며 가능하면 다른 컴퓨터나 컨테이너에서 실행될 수 있습니다.

각 연산자 하위 작업의 수는 해당 연산자의 parallelism을 나타냅니다. 동일한 프로그램의 다른 연산자는 서로 다른 병렬성 수준을 가질 수 있습니다.

스트림은 두 연산자 간에 데이터를 전송할 때 일대일(또는 전달) 패턴으로 또는 재분배 패턴으로 전송할 수 있습니다.

One-to-one 스트림(예: 위의 그림에서 소스 및 map() 연산자 간)은 요소의 파티셔닝과 순서를 유지합니다. 즉, map() 연산자의 하위 작업[1]은 소스 연산자의 하위 작업[1]이 생성한 것과 동일한 순서로 동일한 요소를 볼 것입니다.

Redistributing 스트림(예: 위의 map() 및 keyBy/window 사이, 그리고 keyBy/window 및 Sink 사이)은 스트림의 파티셔닝을 변경합니다. 각 연산자 하위 작업은 선택한 변환에 따라 다른 대상 하위 작업으로 데이터를 보냅니다. 예를 들어, keyBy()는 키를 해싱하여 재분할하는 것이 있습니다. broadcast()나 rebalance()와 같은 것들이 재분할하는 예시입니다. 재분배 스트림에서는 요소 간의 순서가 송신 및 수신 하위 작업 쌍 내에서만 보존됩니다(예: map()의 하위 작업[1] 및 keyBy/window의 하위 작업[2] 사이). 따라서 예를 들어, 위의 keyBy/window 및 Sink 연산자 사이의 재분배는 서로 다른 키에 대한 집계 결과가 Sink에 도착하는 순서에 대한 결정론적이지 않은 요소를 도입합니다.

Timely Stream Processing

대부분의 스트리밍 응용 프로그램에서는 실시간 데이터를 처리하는 데 사용되는 동일한 코드로 과거 데이터를 다시 처리하고, 일관된 결과를 생성하는 것이 매우 가치 있습니다.

이벤트가 처리될 때 전달된 순서가 아니라 발생한 순서에 주의를 기울이고, 일련의 이벤트가 완료되었는지 (또는 완료되어야 하는지) 추론할 수 있는 것이 중요할 수 있습니다. 예를 들어, 전자 상거래 거래 또는 금융 거래와 관련된 이벤트 집합을 고려해보세요.

이러한 적시의 스트림 처리 요구사항은 데이터 스트림에 기록된 이벤트 시간 타임스탬프를 사용하여 충족시킬 수 있습니다. 이는 데이터를 처리하는 기계의 시계를 사용하는 대신에 데이터 스트림에 기록된 이벤트 시간을 사용하는 것입니다.

Stateful Stream Processing

Flink의 연산은 상태를 가질 수 있습니다. 이는 하나의 이벤트가 처리되는 방식이 그 이전에 발생한 모든 이벤트의 누적 효과에 따라 달라질 수 있다는 것을 의미합니다. 상태는 간단한 것으로는 대시보드에 표시할 분당 이벤트 수를 계산하는 것과 같은 것부터, 사기 탐지 모델의 특징을 계산하는 등 더 복잡한 작업에 사용될 수 있습니다.

Flink 애플리케이션은 분산 클러스터에서 병렬로 실행됩니다. 특정 연산자의 다양한 병렬 인스턴스는 독립적으로 실행되며, 별도의 스레드에서 실행되며 일반적으로 서로 다른 기기에서 실행됩니다.

상태를 가진 연산자의 병렬 인스턴스 집합은 사실상 샤딩된 키-값 저장소입니다. 각 병렬 인스턴스는 특정 키 그룹에 대한 이벤트를 처리하는 책임이 있으며 해당 키의 상태는 로컬에서 유지됩니다.

아래 다이어그램은 작업 그래프의 처음 세 연산자를 가로로 두 병렬로 실행되는 작업을 보여주고, 병렬로 하나인 싱크로 끝나는 작업을 보여줍니다. 세 번째 연산자는 상태를 가지며, 두 번째와 세 번째 연산자 사이에서 완전히 연결된 네트워크 셔플이 발생하는 것을 볼 수 있습니다. 이는 스트림을 특정 키로 분할하여 함께 처리해야 하는 모든 이벤트를 처리하기 위해 수행됩니다.

상태는 항상 로컬로 액세스되며, 이는 Flink 애플리케이션이 높은 처리량과 낮은 대기 시간을 달성하는 데 도움이 됩니다. 상태를 JVM 힙에 유지하거나, 상태가 너무 큰 경우 효율적으로 구성된 디스크 상의 데이터 구조에 유지할 수 있습니다.

Fault Tolerance via State Snapshots

Flink는 상태 스냅숏과 스트림 재생을 결합하여 장애 내성과 정확한 한 번만 처리 의미론을 제공할 수 있습니다. 이러한 스냅숏은 분산 파이프라인의 전체 상태를 캡처하며, 입력 대기열의 오프셋과 함께 해당 시점까지의 데이터를 처리하여 발생한 작업 그래프 전체 상태를 기록합니다. 장애가 발생하면 소스가 되감기되고 상태가 복원되며 처리가 다시 시작됩니다. 위에서 설명한대로 이러한 상태 스냅숏은 끊임없는 처리를 방해하지 않고 비동기적으로 캡처됩니다.
