Data Pipelines & ETL

아파치 Flink의 매우 일반적인 사용 사례 중 하나는 데이터를 하나 이상의 소스에서 추출하여 일부 변환 및/또는 보강을 수행 한 다음 결과를 어딘가에 저장하는 ETL (추출, 변환, 로드) 파이프라인을 구현하는 것입니다. 이 섹션에서는 Flink의 DataStream API를 사용하여 이 유형의 응용 프로그램을 구현하는 방법에 대해 살펴 보겠습니다.

Flink의 Table and SQL APIs는 많은 ETL 사용 사례에 적합합니다. 그러나 DataStream API를 직접 사용하든지 여부에 관계없이 여기에 제시된 기본 사항에 대한 견고한 이해는 가치가 있습니다.

Stateless Transformations

이 섹션은 상태를 가지지 않는 변환을 구현하는 데 사용되는 기본 작업인 map() 및 flatmap()을 다룹니다. 이 섹션의 예제는 flink-training-repo의 실습에서 사용되는 택시 운행 데이터에 익숙하다고 가정합니다.

map()

첫 번째 연습에서는 택시 운행 이벤트의 스트림을 필터링했습니다. 동일한 코드 기반에서 위치 (경도, 위도)를 약 100x100 미터 크기의 영역에 대응하는 그리드 셀로 매핑하는 정적 메서드 GeoUtils.mapToGridCell(float lon, float lat)를 제공하는 GeoUtils 클래스가 있습니다.

지금은 택시 운행 객체의 스트림을 startCell 및 endCell 필드를 각 이벤트에 추가하여 보강해 보겠습니다. 이러한 필드를 추가하는 EnrichedRide 객체를 생성할 수 있습니다. 이 객체는 TaxiRide를 확장하여 작성됩니다:

public static class EnrichedRide extends TaxiRide {
    public int startCell;
    public int endCell;

    public EnrichedRide() {}

    public EnrichedRide(TaxiRide ride) {
        this.rideId = ride.rideId;
        this.isStart = ride.isStart;
        ...
        this.startCell = GeoUtils.mapToGridCell(ride.startLon, ride.startLat);
        this.endCell = GeoUtils.mapToGridCell(ride.endLon, ride.endLat);
    }

    public String toString() {
        return super.toString() + "," +
            Integer.toString(this.startCell) + "," +
            Integer.toString(this.endCell);
    }
}

그런 다음 스트림을 변환하는 애플리케이션을 작성할 수 있습니다.

DataStream<TaxiRide> rides = env.addSource(new TaxiRideSource(...));

DataStream<EnrichedRide> enrichedNYCRides = rides
    .filter(new RideCleansingSolution.NYCFilter())
    .map(new Enrichment());

enrichedNYCRides.print();

다음과 같은 MapFunction을 사용하여 이를 수행할 수 있습니다:

public static class Enrichment implements MapFunction<TaxiRide, EnrichedRide> {

    @Override
    public EnrichedRide map(TaxiRide taxiRide) throws Exception {
        return new EnrichedRide(taxiRide);
    }
}

flatmap()

MapFunction은 하나의 요소를 다른 하나로 변환할 때 적합합니다. 따라서 들어오는 각 스트림 요소에 대해 map()이 하나의 변환된 요소를 발행합니다. 그렇지 않은 경우 flatmap()을 사용해야 합니다.

DataStream<TaxiRide> rides = env.addSource(new TaxiRideSource(...));

DataStream<EnrichedRide> enrichedNYCRides = rides
    .flatMap(new NYCEnrichment());

enrichedNYCRides.print();

FlatMapFunction과 함께 사용합니다:

public static class NYCEnrichment implements FlatMapFunction<TaxiRide, EnrichedRide> {

    @Override
    public void flatMap(TaxiRide taxiRide, Collector<EnrichedRide> out) throws Exception {
        FilterFunction<TaxiRide> valid = new RideCleansing.NYCFilter();
        if (valid.filter(taxiRide)) {
            out.collect(new EnrichedRide(taxiRide));
        }
    }
}

이 인터페이스에서 제공된 Collector를 사용하여 flatmap() 메서드는 원하는 만큼 많은 스트림 요소를 발행할 수 있으며, 아예 발행하지 않을 수도 있습니다.

Keyed Streams

keyBy()

스트림을 그 속성 주변에 파티션 할 수 있는 기능은 매우 유용합니다. 이렇게 함으로써 해당 속성의 동일한 값을 갖는 모든 이벤트가 함께 그룹화됩니다. 예를 들어, 각 그리드 셀에서 시작하는 가장 긴 택시 운행을 찾고자 한다고 가정해보십시오. SQL 쿼리로 생각하면 startCell을 기준으로 어떤 형태의 GROUP BY를 수행하는 것을 의미하며, Flink에서는 keyBy(KeySelector)를 사용하여 이 작업을 수행합니다.

rides
    .flatMap(new NYCEnrichment())
    .keyBy(enrichedRide -> enrichedRide.startCell);

keyBy는 스트림을 리파티션하는 네트워크 셔플을 유발합니다. 일반적으로 이는 상당히 비용이 많이 듭니다. 왜냐하면 네트워크 통신과 함께 직렬화 및 역직렬화가 포함되기 때문입니다.

Keys are computed

KeySelector는 이벤트에서 키를 추출하는 데 제한되지 않습니다. 대신, 결과 키가 결정론적이며 hashCode() 및 equals()의 유효한 구현이 있는 한 원하는 방식으로 키를 계산할 수 있습니다. 이 제한으로 인해 무작위 숫자를 생성하는 KeySelector나 배열 또는 열거형을 반환하는 KeySelector는 사용할 수 없지만, 예를 들어 Tuples 또는 POJOs를 사용하여 구성된 키를 사용할 수 있습니다. 단, 그 요소들이 같은 규칙을 따르는 한입니다.

키는 스트림 레코드에 첨부되는 것이 아니라 필요할 때마다 다시 계산되므로 결정론적인 방식으로 생성되어야 합니다.

예를 들어, 새로운 EnrichedRide 클래스를 만드는 대신에 startCell 필드를 사용하여 키를 생성하는 경우,

keyBy(enrichedRide -> enrichedRide.startCell);

we could do this, instead:

keyBy(ride -> GeoUtils.mapToGridCell(ride.startLon, ride.startLat));

Aggregations on Keyed Streams

다음 코드는 각 종료 이벤트마다 startCell 및 지속 시간(분)을 포함하는 튜플의 새로운 스트림을 생성합니다:

import org.joda.time.Interval;

DataStream<Tuple2<Integer, Minutes>> minutesByStartCell = enrichedNYCRides
    .flatMap(new FlatMapFunction<EnrichedRide, Tuple2<Integer, Minutes>>() {

        @Override
        public void flatMap(EnrichedRide ride,
                            Collector<Tuple2<Integer, Minutes>> out) throws Exception {
            if (!ride.isStart) {
                Interval rideInterval = new Interval(ride.startTime, ride.endTime);
                Minutes duration = rideInterval.toDuration().toStandardMinutes();
                out.collect(new Tuple2<>(ride.startCell, duration));
            }
        }
    });

이제 각 startCell에 대해 그동안 관찰된 가장 긴 운행만 포함하는 스트림을 생성할 수 있습니다.

키로 사용할 필드를 표현하는 방법에는 여러 가지가 있습니다. 이전에 EnrichedRide POJO를 사용한 예제를 보았는데, 여기서 키로 사용할 필드는 해당 이름으로 지정되었습니다. 이 경우 Tuple2 객체가 사용되며 튜플 내의 인덱스(0부터 시작)가 키를 지정하는 데 사용됩니다.

minutesByStartCell
  .keyBy(value -> value.f0) // .keyBy(value -> value.startCell)
  .maxBy(1) // duration
  .print();

출력 스트림에는 지속 시간이 새로운 최대값에 도달할 때마다 각 키에 대한 레코드가 포함됩니다. 여기에서는 셀 50797의 경우를 보여줍니다:

(Implicit) State

이것은 상태를 가진 스트리밍을 포함하는 이 교육에서의 첫 번째 예제입니다. 상태는 투명하게 처리되지만 Flink는 각 고유한 키에 대한 최대 지속 시간을 추적해야 합니다.

응용 프로그램에서 상태가 관련될 때 상태가 얼마나 커질 수 있는지를 고려해야 합니다. 키 공간이 무한하면 Flink가 필요로 하는 상태도 무한합니다.

스트림 작업 시 over the entire stream보다는 over finite windows에 대한 집계를 고려하는 것이 일반적으로 더 합리적입니다.

reduce() and other aggregators

위에서 사용된 maxBy()는 Flink의 KeyedStreams에서 사용할 수 있는 여러 집계 함수 중 하나에 불과합니다. 또한 자체 사용자 정의 집계를 구현하는 데 사용할 수 있는 보다 일반적인 용도의 reduce() 함수도 있습니다.

Stateful Transformations

Why is Flink Involved in Managing State?

당신의 응용 프로그램은 Flink의 관리 없이 상태를 사용할 수 있습니다. 그러나 Flink는 관리하는 상태에 대해 몇 가지 매력적인 기능을 제공합니다:

  • local: Flink 상태는 처리하는 머신에 로컬로 유지되며 메모리 속도로 액세스할 수 있습니다.
  • durable: Flink 상태는 내결함성이 있으며, 즉, 정기적인 간격으로 자동으로 체크포인트가 생성되고 장애 발생 시 복원됩니다.
  • vertically scalable: Flink 상태는 추가적인 로컬 디스크를 추가함으로써 확장 가능한 내장 RocksDB 인스턴스에 유지될 수 있습니다.
  • horizontally scalable: Flink 상태는 클러스터가 커지고 줄어들 때 상태가 재분배됩니다.

이 섹션에서는 keyed state를 관리하는 Flink의 API를 사용하는 방법을 배우게 될 것입니다.

Rich Functions

지금까지 FilterFunction, MapFunction 및 FlatMapFunction과 같은 Flink의 여러 함수 인터페이스를 보았습니다. 이들은 모두 단일 추상 메서드 패턴의 예입니다.

각각의 인터페이스에 대해, Flink는 “rich” 변형이라고 불리는 것을 제공합니다. 예를 들어, RichFlatMapFunction은 추가적인 메서드를 포함합니다. 그 예로는 다음이 있습니다:

  • open(Configuration c)
  • close()
  • getRuntimeContext()

open() 메서드는 연산자 초기화 중에 한 번 호출됩니다. 이는 정적 데이터를 로드하거나 외부 서비스에 연결하는 기회입니다.

getRuntimeContext()는 잠재적으로 흥미로운 다양한 요소에 액세스할 수 있지만, 가장 중요한 것은 Flink에서 관리하는 상태를 생성하고 액세스하는 방법입니다.

An Example with Keyed State

이 예에서는 각 키별로 첫 번째 이벤트만 유지하도록 이벤트 스트림을 중복 제거하고자 합니다. 이를 수행하는 애플리케이션인 Deduplicator라는 RichFlatMapFunction을 사용한 애플리케이션을 살펴보겠습니다.

private static class Event {
    public final String key;
    public final long timestamp;
    ...
}

public static void main(String[] args) throws Exception {
    StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
  
    env.addSource(new EventSource())
        .keyBy(e -> e.key)
        .flatMap(new Deduplicator())
        .print();
  
    env.execute();
}

이를 수행하기 위해 Deduplicator는 각 키마다 해당 키에 대한 이벤트가 이미 있는지 여부를 어떻게든 기억해야 합니다. 이를 위해 Flink의 keyed state 인터페이스를 사용할 것입니다.

이와 같은 keyed stream을 다룰 때 Flink는 관리되는 각 상태 항목에 대해 키/값 저장소를 유지합니다.

Flink는 여러 종류의 keyed state를 지원하며, 이 예제에서는 가장 간단한 유형인 ValueState를 사용합니다. 이는 각 키마다 Flink가 단일 객체를 저장한다는 것을 의미합니다. 이 경우에는 Boolean 타입의 객체입니다.

우리의 Deduplicator 클래스에는 두 개의 메서드가 있습니다: open()과 flatMap(). open() 메서드는 ValueStateDescriptor<Boolean>을 정의함으로써 관리 상태의 사용을 설정합니다. 생성자의 인수는 이러한 keyed state 항목에 대한 이름을 지정하고 이러한 객체를 직렬화하는 데 사용할 수 있는 정보를 제공합니다(이 경우 Types.BOOLEAN).

public static class Deduplicator extends RichFlatMapFunction<Event, Event> {
    ValueState<Boolean> keyHasBeenSeen;

    @Override
    public void open(Configuration conf) {
        ValueStateDescriptor<Boolean> desc = new ValueStateDescriptor<>("keyHasBeenSeen", Types.BOOLEAN);
        keyHasBeenSeen = getRuntimeContext().getState(desc);
    }

    @Override
    public void flatMap(Event event, Collector<Event> out) throws Exception {
        if (keyHasBeenSeen.value() == null) {
            out.collect(event);
            keyHasBeenSeen.update(true);
        }
    }
}

flatMap 메서드가 keyHasBeenSeen.value()를 호출하면 Flink의 런타임은 컨텍스트의 키에 대한 이 상태의 값을 찾아보고, 값이 null인 경우에만 이벤트를 출력에 수집합니다. 이 경우에는 keyHasBeenSeen도 true로 업데이트됩니다.

키-파티션 상태를 액세스하고 업데이트하는 이 메커니즘은 마법 같을 수 있습니다. 왜냐하면 Deduplicator의 구현에서 키가 명시적으로 표시되지 않기 때문입니다. Flink의 런타임이 RichFlatMapFunction의 open 메서드를 호출할 때는 이벤트가 없으므로 해당 시점에는 컨텍스트에 키가 없습니다. 그러나 flatMap 메서드를 호출할 때 처리 중인 이벤트의 키가 런타임에 사용 가능하며, Flink의 상태 백엔드에서 어떤 항목이 작동되는지를 결정하는 데 사용됩니다.

분산 클러스터에 배포되면 이 Deduplicator의 많은 인스턴스가 있을 것이며, 각 인스턴스는 전체 키 공간의 겹치지 않는 하위 집합을 담당할 것입니다. 따라서 다음과 같은 단일한 ValueState 항목을 볼 때,

ValueState<Boolean> keyHasBeenSeen;

이것은 단순히 단일 Boolean이 아니라 분산되고 샤딩된 키/값 저장소를 나타내는 것이므로 이해하십시오.

Clearing State

위의 예제에는 잠재적인 문제가 있습니다: 키 공간이 무한한 경우 어떻게 될까요? Flink는 사용되는 각 고유한 키에 대해 Boolean 인스턴스를 어딘가에 저장하고 있습니다. 키의 집합이 유한하면 이는 괜찮지만, 키의 집합이 무한하게 커지는 응용 프로그램에서는 더 이상 필요하지 않은 키에 대한 상태를 지우는 것이 필요합니다. 이를 위해 상태 객체에서 clear()를 호출합니다:

keyHasBeenSeen.clear();

예를 들어, 특정 키에 대한 비활동 기간 후에 이 작업을 수행할 수 있습니다. 이벤트 기반 응용 프로그램 섹션에서 ProcessFunction에 대해 배울 때 타이머를 사용하는 방법을 살펴볼 것입니다.

또한 상태 설명자로 구성할 수 있는 상태의 유효 기간 (TTL) 옵션이 있습니다. 이를 사용하여 오래된 키에 대한 상태가 자동으로 지워지는 시기를 지정할 수 있습니다.

Non-keyed State

non-keyed 컨텍스트에서 관리 상태를 다룰 수도 있습니다. 이를 때로는 오퍼레이터 상태라고도 합니다. 관련된 인터페이스는 다소 다르며, 사용자 정의 함수가 non-keyed state가 필요한 경우가 드물기 때문에 여기에서는 다루지 않습니다. 이 기능은 주로 소스와 싱크의 구현에 사용됩니다.

Connected Streams

어떤 측면의 변환을 동적으로 변경할 수 있는 기능이 필요할 때가 있습니다. 예를 들어, 임계값, 규칙 또는 다른 매개변수를 스트리밍으로 제공하고자 할 수 있습니다. Flink에서 이를 지원하는 패턴은 연결된 스트림입니다. 여기서 하나의 연산자에는 두 개의 입력 스트림이 있습니다:

Example

이 예제에서는 제어 스트림을 사용하여 streamOfWords에서 제외해야 할 단어를 지정합니다. 이를 수행하기 위해 연결된 스트림에 RichCoFlatMapFunction인 ControlFunction을 적용합니다.

public static void main(String[] args) throws Exception {
    StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

    DataStream<String> control = env
        .fromElements("DROP", "IGNORE")
        .keyBy(x -> x);

    DataStream<String> streamOfWords = env
        .fromElements("Apache", "DROP", "Flink", "IGNORE")
        .keyBy(x -> x);
  
    control
        .connect(streamOfWords)
        .flatMap(new ControlFunction())
        .print();

    env.execute();
}

두 개의 연결된 스트림은 호환되는 방식으로 키가 지정되어야 합니다. keyBy의 역할은 스트림 데이터를 파티션하는 것이며, 키가 지정된 스트림이 연결될 때 이들은 동일한 방식으로 파티션되어야 합니다. 이렇게 하면 동일한 키를 갖는 두 스트림의 모든 이벤트가 동일한 인스턴스로 전송되도록 보장됩니다. 그런 다음 이러한 키를 사용하여 두 스트림을 결합할 수 있습니다.

이 경우 스트림은 모두 DataStream<String> 유형이며, 두 스트림 모두 문자열로 키가 지정됩니다. 아래에서 볼 수 있듯이, 이 RichCoFlatMapFunction은 keyed state에 Boolean 값을 저장하고 이 Boolean 값은 두 스트림에서 공유됩니다.

public static class ControlFunction extends RichCoFlatMapFunction<String, String, String> {
    private ValueState<Boolean> blocked;
      
    @Override
    public void open(Configuration config) {
        blocked = getRuntimeContext()
            .getState(new ValueStateDescriptor<>("blocked", Boolean.class));
    }
      
    @Override
    public void flatMap1(String control_value, Collector<String> out) throws Exception {
        blocked.update(Boolean.TRUE);
    }
      
    @Override
    public void flatMap2(String data_value, Collector<String> out) throws Exception {
        if (blocked.value() == null) {
            out.collect(data_value);
        }
    }
}

RichCoFlatMapFunction은 연결된 두 스트림에 적용할 수 있는 FlatMapFunction의 한 종류로, 풍부한 함수 인터페이스에 액세스할 수 있습니다. 즉, 상태를 가질 수 있습니다.

blocked 부울은 제어 스트림에서 언급된 키(이 경우 단어)를 기억하는 데 사용되며, 이러한 단어들은 streamOfWords 스트림에서 필터링됩니다. 이것은 keyed state이며, 두 스트림 간에 공유되므로 두 스트림은 동일한 키 공간을 공유해야 합니다.

flatMap1 및 flatMap2는 Flink 런타임에 의해 두 연결된 스트림의 요소와 함께 호출됩니다. 여기서는 제어 스트림의 요소가 flatMap1로 전달되고, streamOfWords의 요소가 flatMap2로 전달됩니다. 이것은 control.connect(streamOfWords)로 두 스트림이 연결된 순서에 따라 결정되었습니다.

flatMap1 및 flatMap2 콜백이 호출되는 순서를 제어할 수 없다는 점을 인식하는 것이 중요합니다. 이 두 입력 스트림은 서로 경쟁하며, Flink 런타임은 하나의 스트림 또는 다른 스트림에서 이벤트를 소비하는 방법을 스스로 결정할 것입니다. 타이밍 및/또는 순서가 중요한 경우, 관리되는 Flink 상태에서 이벤트를 버퍼링하여 응용 프로그램이 이를 처리할 준비가 될 때까지 기다리는 것이 필요할 수 있습니다. (참고: 만약 절대적으로 필요하다면, InputSelectable을 구현하는 사용자 정의 Operator를 사용하여 두 입력 연산자가 입력을 소비하는 순서를 제한적으로 제어할 수 있습니다.)
