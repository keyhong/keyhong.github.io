[Kubernetes] Concepts - Overview - Objects In Kubernetes - Labels and Selectors

Publish Date:   2024-04-14
Kubernetes

Labels and Selectors

Labels are key/value pairs that are attached to objects such as Pods. Labels are intended to be used to specify identifying attributes of objects that are meaningful and relevant to users, but do not directly imply semantics to the core system. Labels can be used to organize and to select subsets of objects. Labels can be attached to objects at creation time and subsequently added and modified at any time. Each object can have a set of key/value labels defined. Each Key must be unique for a given object.

"metadata": {
  "labels": {
    "key1" : "value1",
    "key2" : "value2"
  }
}

Labels allow for efficient queries and watches and are ideal for use in UIs and CLIs. Non-identifying information should be recorded using annotations.

레이블은 Pod와 같은 객체에 첨부되는 키/값 쌍입니다. 레이블은 사용자에게 의미 있고 관련 있는 객체의 식별 속성을 지정하는 데 사용됩니다. 그러나 레이블은 핵심 시스템에 직접 의미를 함축하지는 않습니다. 레이블은 객체의 하위 집합을 조직화하고 선택하는 데 사용할 수 있습니다. 레이블은 생성 시 객체에 첨부되며 이후 언제든지 추가하거나 수정할 수 있습니다. 각 객체는 정의된 키/값 레이블 세트를 가질 수 있습니다. 주어진 객체에 대해 각 키는 고유해야 합니다.

"metadata": {
  "labels": {
    "key1" : "value1",
    "key2" : "value2"
  }
}

레이블은 효율적인 쿼리 및 감시를 가능하게 하며 UI 및 CLI에서 사용하기에 이상적입니다. 식별에 필요하지 않은 정보는 주석을 사용하여 기록해야 합니다.

Motivation

Labels enable users to map their own organizational structures onto system objects in a loosely coupled fashion, without requiring clients to store these mappings.

Service deployments and batch processing pipelines are often multi-dimensional entities (e.g., multiple partitions or deployments, multiple release tracks, multiple tiers, multiple micro-services per tier). Management often requires cross-cutting operations, which breaks encapsulation of strictly hierarchical representations, especially rigid hierarchies determined by the infrastructure rather than by users.

레이블을 사용하면 사용자는 시스템 객체에 자체 조직 구조를 느슨하게 결합된 방식으로 매핑할 수 있습니다. 이를 통해 클라이언트가 이러한 매핑을 저장할 필요 없이 조직 구조를 객체에 적용할 수 있습니다.

서비스 배포 및 일괄 처리 파이프라인은 종종 다차원 개체입니다(예: 여러 파티션 또는 배포, 여러 릴리스 트랙, 여러 계층, 각 계층당 여러 마이크로 서비스). 관리는 종종 교차 작업이 필요하며, 이는 엄격한 계층적 표현의 캡슐화를 깨뜨립니다. 특히 인프라에 의해 결정된 엄격한 계층 구조가 아닌 사용자에 의해 결정된 구조입니다.

Example labels:

  • "release" : "stable", "release" : "canary"
  • "environment" : "dev", "environment" : "qa", "environment" : "production"
  • "tier" : "frontend", "tier" : "backend", "tier" : "cache"
  • "partition" : "customerA", "partition" : "customerB"
  • "track" : "daily", "track" : "weekly"

These are examples of commonly used labels; you are free to develop your own conventions. Keep in mind that label Key must be unique for a given object.

Syntax and character set

Labels are key/value pairs. Valid label keys have two segments: an optional prefix and name, separated by a slash (/). The name segment is required and must be 63 characters or less, beginning and ending with an alphanumeric character ([a-z0-9A-Z]) with dashes (-), underscores (_), dots (.), and alphanumerics between. The prefix is optional. If specified, the prefix must be a DNS subdomain: a series of DNS labels separated by dots (.), not longer than 253 characters in total, followed by a slash (/).

If the prefix is omitted, the label Key is presumed to be private to the user. Automated system components (e.g. kube-scheduler, kube-controller-manager, kube-apiserver, kubectl, or other third-party automation) which add labels to end-user objects must specify a prefix.

The kubernetes.io/ and k8s.io/ prefixes are reserved for Kubernetes core components.

레이블은 키/값 쌍입니다. 유효한 레이블 키는 두 세그먼트로 구성됩니다: 선택적인 접두사와 이름으로 구분되는 세그먼트가 있습니다. 이름 세그먼트는 필수이며 63자 이하이어야 합니다. 알파벳 문자([a-z0-9A-Z])로 시작하고 끝나야 하며, 중간에는 대시(-), 밑줄(_), 점(.) 및 알파벳 숫자만 허용됩니다. 접두사는 선택 사항입니다. 지정된 경우 접두사는 DNS 하위 도메인이어야 하며, 총 253자를 넘지 않는 점(.)으로 구분된 DNS 레이블 시리즈로 구성되어야 합니다. 그 후에는 슬래시(/)가 이어져야 합니다.

접두사가 생략된 경우 레이블 키는 사용자 개인용으로 간주됩니다. 자동화된 시스템 구성 요소(예: kube-scheduler, kube-controller-manager, kube-apiserver, kubectl 또는 기타 서드파티 자동화 도구)는 사용자 객체에 레이블을 추가하는 경우 접두사를 지정해야 합니다.

kubernetes.io/ 및 k8s.io/ 접두사는 Kubernetes 핵심 구성 요소를 위해 예약되어 있습니다.

Valid label value:

  • must be 63 characters or less (can be empty),
  • unless empty, must begin and end with an alphanumeric character ([a-z0-9A-Z]),
  • could contain dashes (-), underscores (_), dots (.), and alphanumerics between.

유효한 레이블 값:

  • 63자 이하이어야 합니다(비어 있을 수 있음),
  • 비어 있지 않은 경우, 알파벳 문자([a-z0-9A-Z])로 시작하고 끝나야 합니다.
  • 대시(-), 밑줄(_), 점(.), 및 알파벳 숫자만 포함할 수 있습니다.

예를 들어, 두 개의 레이블을 가진 Pod의 매니페스트입니다. environment: production 및 app: nginx:

For example, here’s a manifest for a Pod that has two labels environment: production and app: nginx:

apiVersion: v1
kind: Pod
metadata:
  name: label-demo
  labels:
    environment: production
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80

Label selectors

Unlike names and UIDs, labels do not provide uniqueness. In general, we expect many objects to carry the same label(s).

Via a label selector, the client/user can identify a set of objects. The label selector is the core grouping primitive in Kubernetes.

The API currently supports two types of selectors: equality-based and set-based. A label selector can be made of multiple requirements which are comma-separated. In the case of multiple requirements, all must be satisfied so the comma separator acts as a logical AND (&&) operator.

The semantics of empty or non-specified selectors are dependent on the context, and API types that use selectors should document the validity and meaning of them.

이름과 UID와 달리, 레이블은 고유성을 제공하지 않습니다. 일반적으로 많은 객체가 동일한 레이블을 가질 것으로 예상됩니다.

레이블 셀렉터를 통해 클라이언트/사용자는 객체 집합을 식별할 수 있습니다. 레이블 셀렉터는 Kubernetes에서의 핵심 그룹화 기본 요소입니다.

현재 API는 두 가지 유형의 셀렉터를 지원합니다: 동등성 기반 및 집합 기반 셀렉터입니다. 레이블 셀렉터는 쉼표로 구분된 여러 요구 사항으로 구성될 수 있습니다. 여러 요구 사항의 경우 모두 충족되어야 하므로 쉼표 구분자는 논리 AND (&&) 연산자로 작동합니다.

빈 선택기 또는 지정되지 않은 선택기의 의미는 맥락에 따라 다르며, 선택기를 사용하는 API 유형은 이러한 선택기의 유효성과 의미를 문서화해야 합니다.

Note: For some API types, such as ReplicaSets, the label selectors of two instances must not overlap within a namespace, or the controller can see that as conflicting instructions and fail to determine how many replicas should be present.

Caution: For both equality-based and set-based conditions there is no logical OR (||) operator. Ensure your filter statements are structured accordingly.

참고: ReplicaSets와 같은 일부 API 유형의 경우 네임스페이스 내에서 두 인스턴스의 레이블 셀렉터가 겹치면 컨트롤러가 충돌하는 지시로 인식할 수 있고 레플리카의 수를 결정하는 데 실패할 수 있습니다.

주의: 동등성 기반 및 집합 기반 조건 모두 논리 OR (||) 연산자가 없습니다. 필터 문이 적절하게 구성되었는지 확인하십시오.

Equality-based requirement

Equality- or inequality-based requirements allow filtering by label keys and values. Matching objects must satisfy all of the specified label constraints, though they may have additional labels as well. Three kinds of operators are admitted =,==,!=. The first two represent equality (and are synonyms), while the latter represents inequality. For example:

동등성 또는 비동등성 기반 요구 사항을 사용하면 레이블 키 및 값으로 필터링할 수 있습니다. 일치하는 객체는 지정된 모든 레이블 제약 조건을 충족해야 하지만 추가 레이블을 가질 수도 있습니다. =, ==, != 세 가지 종류의 연산자가 허용됩니다. 첫 두 연산자는 동등성을 나타내며 (동의어입니다), 마지막 연산자는 부등식을 나타냅니다. 예를 들어:

environment = production
tier != frontend

The former selects all resources with key equal to environment and value equal to production. The latter selects all resources with key equal to tier and value distinct from frontend, and all resources with no labels with the tier key. One could filter for resources in production excluding frontend using the comma operator: environment=production,tier!=frontend

One usage scenario for equality-based label requirement is for Pods to specify node selection criteria. For example, the sample Pod below selects nodes with the label “accelerator=nvidia-tesla-p100“.

전자는 키가 환경이고 값이 프로덕션인 모든 리소스를 선택합니다. 후자는 키가 티어이고 값이 프론트엔드와 다른 모든 리소스를 선택하며, 티어 키가 없는 모든 리소스도 선택합니다. 쉼표 연산자를 사용하여 프론트엔드를 제외한 프로덕션에서의 리소스를 필터링할 수 있습니다: environment=production,tier!=frontend

동등성 기반 레이블 요구 사항의 하나의 사용 시나리오는 Pod가 노드 선택 기준을 지정하는 것입니다. 예를 들어, 아래의 샘플 Pod는 “accelerator=nvidia-tesla-p100“ 레이블을 가진 노드를 선택합니다.

apiVersion: v1
kind: Pod
metadata:
  name: cuda-test
spec:
  containers:
    - name: cuda-test
      image: "registry.k8s.io/cuda-vector-add:v0.1"
      resources:
        limits:
          nvidia.com/gpu: 1
  nodeSelector:
    accelerator: nvidia-tesla-p100

Set-based requirement

Set-based label requirements allow filtering keys according to a set of values. Three kinds of operators are supported: in,notin and exists (only the key identifier). For example:

집합 기반 레이블 요구 사항은 값 집합에 따라 키를 필터링할 수 있습니다. 세 가지 종류의 연산자가 지원됩니다: in, notin 및 exists (키 식별자만). 예를 들어:

environment in (production, qa)
tier notin (frontend, backend)
partition
!partition
  • The first example selects all resources with key equal to environment and value equal to production or qa.
  • The second example selects all resources with key equal to tier and values other than frontend and backend, and all resources with no labels with the tier key.
  • The third example selects all resources including a label with key partition; no values are checked.
  • The fourth example selects all resources without a label with key partition; no values are checked.

Similarly the comma separator acts as an AND operator. So filtering resources with a partition key (no matter the value) and with environment different than qa can be achieved using partition,environment notin (qa). The set-based label selector is a general form of equality since environment=production is equivalent to environment in (production); similarly for != and notin.

Set-based requirements can be mixed with equality-based requirements. For example: partition in (customerA, customerB),environment!=qa.

  • 첫 번째 예제는 키가 환경이고 값이 프로덕션 또는 QA인 모든 리소스를 선택합니다.
  • 두 번째 예제는 키가 티어이고 값이 프론트엔드와 백엔드가 아닌 모든 리소스와 티어 키가 없는 모든 리소스를 선택합니다.
  • 세 번째 예제는 키가 파티션인 레이블을 포함한 모든 리소스를 선택합니다. 값을 확인하지 않습니다.
  • 네 번째 예제는 키가 파티션인 레이블이 없는 모든 리소스를 선택합니다. 값을 확인하지 않습니다.

비슷하게, 쉼표 구분자는 AND 연산자로 작동합니다. 따라서 파티션 키를 가진 리소스(값에 관계없이)와 환경이 qa가 아닌 리소스를 필터링하려면 partition,environment notin (qa)를 사용할 수 있습니다. 집합 기반 레이블 셀렉터는 환경=프로덕션이 environment in (프로덕션)과 동일하며, != 및 notin에 대해서도 비슷하게 동작합니다.

집합 기반 요구 사항은 동등성 기반 요구 사항과 혼합하여 사용할 수 있습니다. 예를 들어: partition in (customerA, customerB),environment!=qa.

API

LIST and WATCH filtering

LIST and WATCH operations may specify label selectors to filter the sets of objects returned using a query parameter. Both requirements are permitted (presented here as they would appear in a URL query string):

  • equality-based requirements: ?labelSelector=environment%3Dproduction,tier%3Dfrontend
  • set-based requirements: ?labelSelector=environment+in+%28production%2Cqa%29%2Ctier+in+%28frontend%29

Both label selector styles can be used to list or watch resources via a REST client. For example, targeting apiserver with kubectl and using equality-based one may write:

LIST 및 WATCH 작업은 쿼리 매개변수를 사용하여 반환되는 객체 세트를 필터링하기 위해 레이블 셀렉터를 지정할 수 있습니다. 여러 요구 사항이 허용됩니다 (여기에서 URL 쿼리 문자열로 표시됩니다):

  • 동등성 기반 요구 사항: ?labelSelector=environment%3Dproduction,tier%3Dfrontend
  • 집합 기반 요구 사항: ?labelSelector=environment+in+%28production%2Cqa%29%2Ctier+in+%28frontend%29

두 가지 레이블 셀렉터 스타일은 REST 클라이언트를 통해 리소스를 나열하거나 감시하는 데 사용할 수 있습니다. 예를 들어, kubectl을 사용하여 apiserver를 대상으로하고 동등성 기반 셀렉터를 사용할 수 있습니다:

kubectl get pods -l environment=production,tier=frontend

or using set-based requirements:

또는 집합 기반 요구 사항을 사용할 수 있습니다:

kubectl get pods -l 'environment in (production),tier in (frontend)'

As already mentioned set-based requirements are more expressive. For instance, they can implement the OR operator on values:

이미 언급한 대로 집합 기반 요구 사항은 더 표현력이 높습니다. 예를 들어, 값에 대한 OR 연산자를 구현할 수 있습니다:

kubectl get pods -l 'environment in (production, qa)'

or restricting negative matching via notin operator:

또는 notin 연산자를 사용하여 부정적인 일치를 제한할 수 있습니다:

kubectl get pods -l 'environment,environment notin (frontend)'

Set references in API objects

Some Kubernetes objects, such as [services]() and [replicationcontrollers](), also use label selectors to specify sets of other resources, such as pods.

일부 Kubernetes 객체인 서비스와 레플리케이션 컨트롤러는 파드와 같은 다른 리소스 세트를 지정하기 위해 레이블 셀렉터를 사용합니다.

Service and ReplicationController

The set of pods that a service targets is defined with a label selector. Similarly, the population of pods that a replicationcontroller should manage is also defined with a label selector.

Label selectors for both objects are defined in json or yaml files using maps, and only equality-based requirement selectors are supported:

서비스가 대상으로 하는 파드 집합은 레이블 셀렉터로 정의됩니다. 마찬가지로, 레플리케이션 컨트롤러가 관리해야 하는 파드 집합도 레이블 셀렉터로 정의됩니다.

두 객체의 레이블 셀렉터는 맵을 사용하여 JSON 또는 YAML 파일에 정의되며, 동등성 기반 요구 사항 셀렉터만 지원됩니다.

"selector": {
    "component" : "redis",
}

or

selector:
  component: redis

This selector (respectively in json or yaml format) is equivalent to component=redis or component in (redis).

이 셀렉터(각각 JSON 또는 YAML 형식으로)는 component=redis 또는 component in (redis)와 동등합니다.

Resources that support set-based requirements

Newer resources, such as Job, Deployment, ReplicaSet, and DaemonSet, support set-based requirements as well.

Job, Deployment, ReplicaSet, 및 DaemonSet과 같은 최신 리소스는 집합 기반 요구 사항도 지원합니다.

selector:
  matchLabels:
    component: redis
  matchExpressions:
    - { key: tier, operator: In, values: [cache] }
    - { key: environment, operator: NotIn, values: [dev] }

matchLabels is a map of {key,value} pairs. A single {key,value} in the matchLabels map is equivalent to an element of matchExpressions, whose key field is “key”, the operator is “In”, and the values array contains only “value”. matchExpressions is a list of pod selector requirements. Valid operators include In, NotIn, Exists, and DoesNotExist. The values set must be non-empty in the case of In and NotIn. All of the requirements, from both matchLabels and matchExpressions are ANDed together – they must all be satisfied in order to match.

matchLabels는 {key, value} 쌍의 맵입니다. matchLabels 맵의 단일 {key, value}는 matchExpressions의 요소와 동등합니다. 이때 matchExpressions의 키 필드는 “key”이고, 연산자는 “In”이며, 값 배열은 “value”만 포함합니다. matchExpressions는 팟 셀렉터 요구 사항의 목록입니다. 유효한 연산자는 In, NotIn, Exists 및 DoesNotExist가 포함됩니다. In 및 NotIn의 경우 값 세트는 비어있으면 안 됩니다. matchLabels 및 matchExpressions에서 모든 요구 사항이 AND로 결합됩니다. 즉, 모두 충족되어야 합니다.

Selecting sets of nodes

One use case for selecting over labels is to constrain the set of nodes onto which a pod can schedule. See the documentation on node selection for more information.

레이블을 기준으로 선택하는 하나의 사용 사례는 파드가 스케줄링될 노드 집합을 제한하는 것입니다. 자세한 내용은 노드 선택에 관한 문서를 참조하십시오.

Using labels effectively

You can apply a single label to any resources, but this is not always the best practice. There are many scenarios where multiple labels should be used to distinguish resource sets from one another.

For instance, different applications would use different values for the app label, but a multi-tier application, such as the guestbook example, would additionally need to distinguish each tier. The frontend could carry the following labels:

모든 리소스에 단일 레이블을 적용할 수 있지만, 이것이 항상 최선의 방법은 아닙니다. 서로 다른 리소스 집합을 구별하기 위해 여러 레이블을 사용해야 하는 많은 시나리오가 있습니다.

예를 들어, 서로 다른 애플리케이션은 앱 레이블에 대해 서로 다른 값을 사용할 수 있지만, 게스트북 예제와 같은 멀티 티어 애플리케이션은 각 티어를 구별할 필요가 있습니다. 프론트엔드는 다음과 같은 레이블을 가질 수 있습니다:

labels:
  app: guestbook
  tier: frontend

while the Redis master and replica would have different tier labels, and perhaps even an additional role label:

Redis 마스터와 레플리카는 서로 다른 티어 레이블을 가지며, 추가적으로 역할 레이블을 가질 수도 있습니다:

labels:
  app: guestbook
  tier: backend
  role: master

and

labels:
  app: guestbook
  tier: backend
  role: replica

The labels allow for slicing and dicing the resources along any dimension specified by a label:

레이블을 사용하면 레이블에 의해 지정된 모든 차원에 따라 리소스를 세분화하고 분석할 수 있습니다:

kubectl apply -f examples/guestbook/all-in-one/guestbook-all-in-one.yaml
kubectl get pods -Lapp -Ltier -Lrole
NAME                           READY  STATUS    RESTARTS   AGE   APP         TIER       ROLE
guestbook-fe-4nlpb             1/1    Running   0          1m    guestbook   frontend   <none>
guestbook-fe-ght6d             1/1    Running   0          1m    guestbook   frontend   <none>
guestbook-fe-jpy62             1/1    Running   0          1m    guestbook   frontend   <none>
guestbook-redis-master-5pg3b   1/1    Running   0          1m    guestbook   backend    master
guestbook-redis-replica-2q2yf  1/1    Running   0          1m    guestbook   backend    replica
guestbook-redis-replica-qgazl  1/1    Running   0          1m    guestbook   backend    replica
my-nginx-divi2                 1/1    Running   0          29m   nginx       <none>     <none>
my-nginx-o0ef1                 1/1    Running   0          29m   nginx       <none>     <none>
kubectl get pods -lapp=guestbook,role=replica
````

Updating labels

Sometimes you may want to relabel existing pods and other resources before creating new resources. This can be done with kubectl label. For example, if you want to label all your NGINX Pods as frontend tier, run:

새 리소스를 만들기 전에 기존 파드 및 기타 리소스의 레이블을 다시 지정할 수도 있습니다. 이를 위해 kubectl label을 사용할 수 있습니다. 예를 들어, 모든 NGINX 파드를 frontend 티어로 레이블링하려면 다음을 실행하십시오:

kubectl label pods -l app=nginx tier=fe
pod/my-nginx-2035384211-j5fhi labeled
pod/my-nginx-2035384211-u2c7e labeled
pod/my-nginx-2035384211-u3t6x labeled

This first filters all pods with the label “app=nginx”, and then labels them with the “tier=fe”. To see the pods you labeled, run:

먼저 “app=nginx” 레이블을 가진 모든 파드를 필터링한 다음 이를 “tier=fe”로 레이블합니다. 레이블이 지정된 파드를 확인하려면 다음을 실행하십시오:

kubectl get pods -l app=nginx -L tier
NAME                        READY     STATUS    RESTARTS   AGE       TIER
my-nginx-2035384211-j5fhi   1/1       Running   0          23m       fe
my-nginx-2035384211-u2c7e   1/1       Running   0          23m       fe
my-nginx-2035384211-u3t6x   1/1       Running   0          23m       fe

This outputs all “app=nginx” pods, with an additional label column of pods’ tier (specified with -L or --label-columns).

For more information, please see kubectl label.

이는 “app=nginx” 레이블이 지정된 모든 파드를 출력하며, 추가로 파드의 티어 레이블 열을 포함합니다 (-L 또는 --label-columns로 지정됨).

자세한 내용은 kubectl label을 참조하십시오.
