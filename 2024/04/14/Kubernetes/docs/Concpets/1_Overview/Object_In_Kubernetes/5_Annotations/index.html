[Kubernetes] Concepts - Overview - Objects In Kubernetes - Annotations

Publish Date:   2024-04-14
Kubernetes

Annotations

You can use Kubernetes annotations to attach arbitrary non-identifying metadata to objects. Clients such as tools and libraries can retrieve this metadata.

Kubernetes 어노테이션을 사용하여 객체에 임의의 식별되지 않는 메타데이터를 첨부할 수 있습니다. 도구 및 라이브러리와 같은 클라이언트는 이 메타데이터를 검색할 수 있습니다.

Attaching metadata to objects

You can use either labels or annotations to attach metadata to Kubernetes objects. Labels can be used to select objects and to find collections of objects that satisfy certain conditions. In contrast, annotations are not used to identify and select objects. The metadata in an annotation can be small or large, structured or unstructured, and can include characters not permitted by labels. It is possible to use labels as well as annotations in the metadata of the same object.

Kubernetes 객체에 메타데이터를 첨부하는 데 레이블이나 어노테이션을 사용할 수 있습니다. 레이블은 객체를 선택하고 특정 조건을 만족하는 객체 컬렉션을 찾는 데 사용될 수 있습니다. 반면 어노테이션은 객체를 식별하고 선택하는 데 사용되지 않습니다. 어노테이션의 메타데이터는 작거나 크며, 구조화되거나 구조화되지 않거나, 레이블에서 허용되지 않는 문자를 포함할 수 있습니다. 동일한 객체의 메타데이터에 레이블 및 어노테이션을 모두 사용할 수 있습니다.

Annotations, like labels, are key/value maps:

어노테이션은 레이블과 마찬가지로 키/값 맵입니다:

"metadata": {
  "annotations": {
    "key1" : "value1",
    "key2" : "value2"
  }
}

Note: The keys and the values in the map must be strings. In other words, you cannot use numeric, boolean, list or other types for either the keys or the values.

참고: 맵 안의 키와 값은 문자열이어야 합니다. 다시 말해, 키나 값으로 숫자, 부울, 리스트 또는 다른 타입을 사용할 수 없습니다.

Here are some examples of information that could be recorded in annotations:

여기에 어노테이션에 기록될 수 있는 정보의 예시 몇 가지가 있습니다:

  • Fields managed by a declarative configuration layer. Attaching these fields as annotations distinguishes them from default values set by clients or servers, and from auto-generated fields and fields set by auto-sizing or auto-scaling systems.

  • Build, release, or image information like timestamps, release IDs, git branch, PR numbers, image hashes, and registry address.

  • Pointers to logging, monitoring, analytics, or audit repositories.

  • Client library or tool information that can be used for debugging purposes: for example, name, version, and build information.

  • User or tool/system provenance information, such as URLs of related objects from other ecosystem components.

  • Lightweight rollout tool metadata: for example, config or checkpoints.

  • Phone or pager numbers of persons responsible, or directory entries that specify where that information can be found, such as a team web site.

  • Directives from the end-user to the implementations to modify behavior or engage non-standard features.

  • 선언적 구성 계층에 의해 관리되는 필드. 이러한 필드를 어노테이션으로 첨부하면 클라이언트나 서버에서 설정한 기본 값과 자동 생성된 필드, 자동 크기 조정이나 자동 확장 시스템에 의해 설정된 필드와 구별됩니다.

  • 타임스탬프, 릴리스 ID, Git 브랜치, PR 번호, 이미지 해시 및 레지스트리 주소와 같은 빌드, 릴리스 또는 이미지 정보.

  • 로깅, 모니터링, 분석 또는 감사 저장소로의 포인터.

  • 디버깅 용도로 사용될 수 있는 클라이언트 라이브러리 또는 도구 정보: 예를 들어 이름, 버전 및 빌드 정보.

  • 사용자 또는 도구/시스템 출처 정보. 다른 생태계 구성 요소의 관련 객체 URL과 같은 정보.

  • 경량 롤아웃 도구 메타데이터: 예를 들어 구성 또는 체크포인트.

  • 담당자의 전화번호 또는 페이저 번호, 또는 그 정보가 어디에서 찾을 수 있는지를 지정하는 디렉터리 항목, 예를 들어 팀 웹 사이트.

  • 사용자가 구현에게 동작을 수정하거나 비표준 기능을 활성화하도록 지시하는 지시사항.

Instead of using annotations, you could store this type of information in an external database or directory, but that would make it much harder to produce shared client libraries and tools for deployment, management, introspection, and the like.

어노테이션 대신에 이러한 유형의 정보를 외부 데이터베이스나 디렉터리에 저장할 수도 있지만, 그렇게 하면 배포, 관리, 내부 검사 등을 위한 공유 클라이언트 라이브러리와 도구를 생성하는 것이 훨씬 어려워집니다.

Syntax and character set

Annotations are key/value pairs. Valid annotation keys have two segments: an optional prefix and name, separated by a slash (/). The name segment is required and must be 63 characters or less, beginning and ending with an alphanumeric character ([a-z0-9A-Z]) with dashes (-), underscores (_), dots (.), and alphanumerics between. The prefix is optional. If specified, the prefix must be a DNS subdomain: a series of DNS labels separated by dots (.), not longer than 253 characters in total, followed by a slash (/).

If the prefix is omitted, the annotation Key is presumed to be private to the user. Automated system components (e.g. kube-scheduler, kube-controller-manager, kube-apiserver, kubectl, or other third-party automation) which add annotations to end-user objects must specify a prefix.

The kubernetes.io/ and k8s.io/ prefixes are reserved for Kubernetes core components.

For example, here’s a manifest for a Pod that has the annotation imageregistry: https://hub.docker.com/ :

어노테이션은 키/값 쌍입니다. 유효한 어노테이션 키는 두 부분으로 구성됩니다: 선택적인 접두사와 이름으로 슬래시 (/)로 구분됩니다. 이름 세그먼트는 필수이며, 알파벳 문자 ([a-z0-9A-Z])로 시작하고 끝나며, 문자 사이에는 대시 (-), 밑줄 (_), 점 (.), 및 알파벳 및 숫자를 사용할 수 있습니다. 접두사는 선택 사항입니다. 지정된 경우 접두사는 DNS 하위 도메인이어야 합니다: 점 (.)으로 구분된 일련의 DNS 레이블로 총 길이가 253자 이하이며, 슬래시 (/)로 끝납니다.

접두사가 생략된 경우 어노테이션 키는 사용자에게 비공개로 간주됩니다. 자동화된 시스템 구성 요소(예: kube-scheduler, kube-controller-manager, kube-apiserver, kubectl 또는 기타 제3자 자동화)는 끝 사용자 객체에 어노테이션을 추가할 때 접두사를 지정해야 합니다.

kubernetes.io/ 및 k8s.io/ 접두사는 Kubernetes 핵심 구성 요소를 위해 예약되어 있습니다.

예를 들어, imageregistry: https://hub.docker.com/를 가진 Pod의 매니페스트는 다음과 같습니다.

apiVersion: v1
kind: Pod
metadata:
  name: annotations-demo
  annotations:
    imageregistry: "https://hub.docker.com/"
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80
