SQL 성능을 개선하는 기본 원칙

SQL

1. SELECT 시 꼭 필요한 컬럼만 불러오기

빅데이터 환경에서는 데이터 저장 포맷이 컬럼 형식(columnar)인 경우가 많다(예: Parquet, ORC). 불필요한 데이터를 읽지 않는 것은 Read I/O에 큰 영향을 미친다. 따라서 모든 컬럼을 읽는 것이 아니라 필요 컬럼을 명시적으로 지정해 질의문(SELECT)을 실행해야 빠른 조회 결과를 얻을 수 있다.

필요한 컬럼만 읽는 쿼리 예시
-- bad
SELECT *
  FROM 테이블명
;

-- good
SELECT 컬럼1
     , 컬럼2
     , 컬럼3
  FROM 테이블명
;

2. 조건문의 인덱스 컬럼에 별도의 연산을 적용하지 않기

INDEX가 걸려 있는 컬럼에 연산을 적용하면, INDEX를 활용하지 못하고 풀 스캔을 수행하게 된다. 이 경우 인덱스의 효과가 사라진다. 따라서 WHERE 조건절에서 인덱스 컬럼을 활용할 때는 컬럼에 대한 연산을 피하고, 비교할 값의 포맷을 변경하는 등의 방법을 통해 인덱스를 활용해야 빠른 조회 결과를 얻을 수 있다.

인덱스 컬럼에 연산을 걸지 않는 쿼리 예시
-- bad
SELECT *
  FROM 테이블명
 WHERE YEAR(date) = 2021

-- good
SELECT *
  FROM 테이블명
 WHERE date > '2021-01-01' AND date < '2022-01-01'

3. LIKE 사용시 String 내 %를 앞부분에 배치하지 않는 것이 좋음

LIKE 연산자에서 앞에 %를 붙이면, 조건절에 명시한 문자열 컬럼의 개별 값의 처음부터 마지막 문자까지 모두 읽게 된다. 반대로, 연산자의 뒷부분에 %를 붙이면, % 앞에 있는 문자열 조건을 만족하는 경우 나머지 문자는 더 이상 확인하지 않고 다음 값을 탐색한다. 문자열의 길이가 길어질수록 이는 큰 영향을 미친다. 따라서 문자열 비교 연산 시 %는 뒤에 붙이는 것이 좋다.

효율적인 LIKE 연산자 쿼리 예시
-- bad
SELECT *
  FROM 테이블명
 WHERE 학교명 LIKE '%결대학교'

-- good
SELECT *
  FROM 테이블명
 WHERE 학교명 LIKE '결대%'

4. 연속 INNER JOIN 시에는 FROM절에 가장 작은 테이블을 넣을 것

연속된 테이블의 INNER JOIN을 사용할 때, 가장 작은 테이블을 기준으로 잡고 다른 테이블과 결합하는 것이 좋다. 큰 테이블끼리 먼저 결합하면 결과로 나온 테이블도 큰 테이블이 될 가능성이 높아진다. 마지막에 작은 테이블을 INNER JOIN하여 테이블의 크기를 줄이면 이전에 수행한 JOIN 연산이 비효율적일 수 있다.

  • 쿼리 옵티마이저는 실행 계획에 따라 스스로 작은 테이블을 먼저 가져오기도 한다. 그러나 테이블이 많아질수록 사용자가 명시적으로 지정해 주는 것이 좋다.
JOIN 시 데이터 사이즈를 최적화하는 INNER JOIN 쿼리 예시
-- good
SELECT A.age -- A 테이블은 행(row) 수가 가장 작은 테이블이다.
  FROM A
    JOIN B ON A.동일열1 = B.동일열1
    JOIN C ON B.동일열2 = C.동일열2

5. HAVING 보다 WHERE 절에서 데이터를 줄일 수 있는 경우 줄일 것

WHERE 조건절은 HAVING 절보다 먼저 수행된다. 같은 결과를 도출할 수 있다면, 더 먼저 실행되는 단계에서 데이터의 크기를 줄이면 메모리 사용률을 조기에 낮출 수 있어 효율적이다.

HAVING 보다 WHERE에서 데이터 사이즈를 줄이는 쿼리 예시
-- bad
SELECT *
  FROM A
 GROUP BY 닉네임
HAVING 닉네임 != '이름'

-- good
SELECT *
  FROM A
 WHERE 닉네임 != '이름'
 GROUP BY 닉네임
