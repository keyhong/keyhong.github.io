Docker Image와 Image Layer

도커 이미지는 이미지 레이어가 모인 논리적 대상. 레이어는 도커 엔진의 캐시에 물리적으로 저장된 파일
이미지 레이어는 여러 이미지와 컨테이너에서 공유되기 때문에 중요

  • 도커 이미지 히스토리 확인
$ docker image history [image]

  - CREATE BY : 해당 레이어를 구성한 Dockerfile 스크립트의 인스트럭션. Dockerfile 인스트럭션과 이미지레이어는 1:1 관계

도커 이미지는 최소한의 운영체제 레이어를 가진다. (리눅스 이미지는 약 75MB의 디스크 용량을 차지)
docker image ls 명령어를 통해 SIZE 확인가능. 하지만 이는 논리적 용량일 뿐, 해당 이미지가 실제로 차지하는 디스크 용량이 아님
docker system df 명령어를 통해 실제 사용된 디스크 용량 확인 가능

따라서, 같은 기반 레이어를 공유하는 애플리케이션의 숫자가 많을 수록 더욱 디스크 공간을 절약 가능
이미지 레이어를 여러 이미지가 공유한다면, 공유되는 레이어는 수정할 수 없어야함. 도커는 이미지 레이어를 읽기 전용으로 만든다.

이러한 이유로 이미지를 빌드하면서 만들어지는 레이어를 최적화하는 것이 필요. 즉, Dockerfile 스크립트의 인스트럭션의 순서를 최적화하여
도커 이미지의 용량을 줄이고 빌드를 빠르게 만드는 기법을 활용

이미지 레이어 캐시를 이용하여 Dockerfile 스크립트 최적화

도커의 이미지 레이어가 특정한 순서대로만 배치된다고 가정하기 때문에,
순서의 중간에 있는 레이어가 변경되면 변경된 레이어보다 위에 오는 레이어를 재사용 불가

Dockerfile 스크립트의 인스트럭션은 각각 하나의 이미지 레이어와 1:1로 연결
인스트럭션의 결과가 이전 빌드와 같다면, 이전에 캐시된 레이어를 재사용

캐시에 일치하는 레이어가 있는지 확인하기 위해 해시값을 이용. 기존 이미지 레이어에 해시값이 일치하는 것이 없다면 캐시 미스 발생 및 인스트럭션 실행
해시값은 Dockerfile 스크립트의 인스트럭션과 인스트럭션에 의해 복사되는 파일의 내용으로부터 계산

예를 들어 app.js의 파일이 변경된다면 COPY app.js . 이후에 실행되는 인스트럭션은 변화가 없더라도 재사용이 불가능
이런 경우, 변경이 언제든 발생가능한 COPY 인스트럭션을 제일 상위 레이어로 올림(마지막 실행)으로써 캐시를 적극 활용
이미지를 공유하는 과정에서 시간, 디스크 용량, 네트워크 대역폭을 모두 절약할 수 있는 방법

# 기존 Dockerfile을 최적한 경우
FROM diamol/node

CMD ["node", "/web-ping/app.js"]

ENV TARGET="blog.sixeyed.com" \
    METHOD="HEAD" \
    INTERVAL="3000"

WORKDIR /web-ping
COPY app.js .

Docker Image Command

  • 도커 이미지 목록 확인
docker images | docker image ls
  -'문자열*' : 특정 문자로 시작하는 이미지 목록 조회
  • 도커 이미지 다운로드
docker image pull <이미지명>
  • 도커 이미지 빌드
docker image build --tag <이미지명> [컨텍스트 : Dockerfile 및 이미지에 포함시킬 파일이 위치한 경로]
  • 도커 이미지 히스토리 확인
docker image history <이미지명>
  • 이미지 모두 제거
# rm -f (--force) : 강제 제거(SIGKILL)
# ls -f (--filter) : 제공된 조건에 따라 필터링
docker image rm -f $(docker image ls -f reference='<조건>' -q )
