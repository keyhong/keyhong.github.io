SQL 성능 튜닝

SQL

SQL 성능을 개선하는 원칙

  1. SELECT시 꼭 필요한 컬럼만 불러오기

빅데이터 환경에서는 데이터 저장 포맷이 columnar한 경우(예를 들면, parquet나 orc)가 많다. 불필요한 데이터를 읽진않은 것은 Read I/O에 큰 영향을 끼친다. 그렇기 때문에 모든 컬럼(*)을 읽는 게 아닌 경우에는, 필요 컬럼을 명시적으로 지정해서 질의문(SELECT)을 날려야 빠른 조회 결과를 받을 수 있다.

-- bad
SELECT *
  FROM 테이블명
;

-- good
SELECT 컬럼1
     , 컬럼2
     , 컬럼3
  FROM 테이블명
;
  1. 조건문의 인덱스 컬럼에 별도의 연산을 걸지 않기

INDEX가 걸려 있는 컬럼에 연산을 걸게 되면, INDEX를 타지 못하고 풀 스캔을 하게 된다. 이럴 경우 인덱스를 주는 효과가 없다. 따라서 WHERE 조건절에서 인덱스 컬럼을 활용할 경우 컬럼에 대한 연산이 아닌, 비교할 값의 포맷을 변경하거나 하는 방법을 통해 인덱스를 태우게 해야 빠른 조회 결과를 받을 수 있다. 

-- bad
SELECT *
  FROM 테이블명
 WHERE YEAR(date) = 2021

-- good
SELECT *
  FROM 테이블명
 WHERE date > '2021-01-01' AND date < '2022-01-01'
  1. LIKE 사용시 String 내 %를 앞부분에 배치하지 않는 것이 좋음

LIKE 연산자에서 앞에 %를 붙이게 되면 조건절에 명시한 문자열 컬럼의 개별 값의 처음부터 문자부터 마지막 문자까지 모든 읽게 된다. 반대로 연산자의 뒷 부분에 %를 붙이게 되면 %앞에 있는 문자열 조건을 만족하게 되면 나머지 문자는 더 이상 확인하지 않고 다음 값을 탐색한다. 컬럼 값의 문자열이 길 수록 이는 큰 영향을 준다. 따라서 문자열 비교 연산시 %는 뒤에 붙이는 것이 좋다. 

-- bad
SELECT *
  FROM 테이블명
 WHERE 학교명 LIKE '%결대학교'

-- good
SELECT *
  FROM 테이블명
 WHERE 학교명 LIKE '결대%'
  1. 연속된 테이블의 INNER JOIN 시에는 FROM절에 가장 작은 테이블을 넣을 것

연속된 테이블의 JOIN이 발생시, 가장 작은 테이블을 기준으로 잡고 다른 테이블을 결합하는 것이 좋다. 큰 테이블 끼리 먼저 결합한다면 결과로 나온 테이블 역시 큰 테이블일 경우가 많은데, 마지막에 작은 테이블을 INNER JOIN하여 작은 테이블이 되는 경우 앞에서 했던 JOIN 연산이 상당히 비효율적일 수 있기 때문이다.

  • 쿼리 옵티마이저의 실행 계획에 따라 스스로 작은 테이블을 먼저 가져오기도 한다. 다만, 테이블이 많아질 수록 사용자가 명시적으로 지정을 해주는 것이 좋다.
-- good
SELECT A.age -- A 테이블은 행(row) 수가 가장 작은 테이블이다.
  FROM A
    JOIN B ON A.동일열1 = B.동일열1
    JOIN C ON B.동일열2 = C.동일열2
  1. HAVING 보다 WHERE 절에서 데이터를 줄일 수 있는 경우 줄일 것

WHERE 조건절은 HAVING 절보다 먼저 수행된다. 같은 결과를 도출할 수 있다면 더 먼저 실행되는 동작에서 먼저 데이터의 크기를 줄여주면 좋다.

-- bad
SELECT *
  FROM A
 GROUP BY 닉네임
HAVING 닉네임 != '이름'

-- good
SELECT *
  FROM A
 WHERE 닉네임 != '이름'
 GROUP BY 닉네임
