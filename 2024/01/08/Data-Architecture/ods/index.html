[Data Architecture] ODS

Data Architecture ODS
Data Architecture
Publish Date:   2024-01-08

🙊 글을 쓰는 이유

회사에서 구축한 DW에 ODS라는 데이터베이스를 본 적이 있어요. 이게 뭔지 궁금해서 동료들에게 ODS에 대해 여쭤본 적이 있었지만 명확한 정의를 얻을 수는 없어요. 다들 어디에서는 본 적은 있지만, 왜 있는 지와 어떤 역할인지에 대한 정의가 너무나 다르더라구요.

그 때는 왜 그럴 까, 어쩌면 관례처럼 굳어진 것 때문일까? 라는 생각을 했었는 데, 강의를 들으면서
ODS의 존재 이유를 알게 됬어요. 또 왜 다들 ODS에 대해 애매모호 한지도요. 오늘은 그 내용을 설명해보려고 해요.

ODS (Operational Data Store)

DW와 유사하게 분석을 목적으로 만들어졌지만, 기업의 빠른 의사결정에 조금 더 초점이 맞춘 데이터베이스에요.

ODS vs DW 차이

이게 가장 중요한 부분일 것 같아요. 그럼 DW와 ODS 둘다 똑같이 분석용 DB인데 어떤 차이가 있나에 대한 질문이에요. 이를 하나의 서비스 상황을 만들어 설명해볼께요.

우리는 금융 시스템에 종사하고 있어요. 우리가 하려는 일은 고객의 현재 금융 상태를 파악해서 바로 즉각적인 액션을 하려고 하거든요. 고객의 재무를 알 수 있는 계좌, 대출, 부동산 등 데이터를 하나의 데이터베이스에 모았어요.

위의 예시의 핵심 키워드는 “현재 금융 상태를 파악” 한다는 것이에요. 여기서 두 가지에 관점을 얻을 수 있어요.

  • 첫째, 우리는 이력 데이터를 쌓거나 분석 할 필요가 없어요. 원하는 건 현재 상태이지, 과거의 상태가 아니에요.

  • 둘째, 최신 데이터를 반영하려면 빠른 주기로 ETL을 해야 해요. 보통 DW에 ETL하는 것보다 주기가 빨라요.

요약하자면 ODS는 DW처럼 과거의 모든 데이터를 필요로 하지도 않고, 보다 빠르게 준실시간성 데이터가 흐르는 곳이에요. 이러한 목적으로 설계되었기 때문에 운영적인 의사결정을 바로 할 수 있어요.

ODS의 두 가지 형태

  • 전체적인 구조에서 ODS는 생성 위치는 두 가지 형태로 존재 할 수 있어요.

parallel 구조

  • DW와 평행한 구조로 두 데이터베이스 모두 별도의 분석용 데이터베이스로 쓰여요.

  • DW와 ODS 모두 수집해야 하는 데이터가 있다면 transform을 중복 수행해야 하는 단점이 있어요.

sequential 구조

  • DW의 바로 직전 단계에서 데이터 소스 역할로 많이 쓰여요.

  • transform의 중복을 줄이기 위해 대부분 sequential한 구조를 많이 채택하는 편이에요.

ODS의 경계가 점점 없어지는 이유

최근 플랫폼의 발전에서 따라 ETL 속도가 훨씬 더 빨라졌어요. 그로 인해 DW에서도 빠르게 데이터를 받아볼 수 있죠. 실시간 데이터 처리에 Kafka, Spark, Storm, Flink 같은 플랫폼과 함께 이제는 병렬 처리도 가능해졌어요.

🙊 글 마무리

ODS의 경계가 많이 허물어지는 건 사실이에요. 하지만 그렇다고 이젠 없어도 된다는 건 아니에요. 분명히 운영용 데이터 분석이라는 목적이 있어요. 또, sequential 구조처럼 DW 앞에서 데이터 소스 역할도 충분히 할 수 있어요.

ODS가 "기존에 구축되어 있는 지 확인하고, 있다면 있는 것을 활용"함으로써 DW 전까지 해야 할 업무 리소스를 줄일 수 있다는 점을 인지해야 해요.
