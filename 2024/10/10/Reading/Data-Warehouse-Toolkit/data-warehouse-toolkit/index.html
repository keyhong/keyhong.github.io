1 고객관계관리 개요

과거 제품 중심적인 관점이 고객 니즈 기반으로 변화함에 따라, 고객 중심 관점으로 전환을 의미. 애매모호하지만 고객을 잘 이해하는 것에 기본을 두는 일반적인 행위나 사고

1.1 목적

고객의 전 생애 기간 동안 관계를 잘 유지하는 것. 마케팅, 영엉, 생산 및 서비스와 같은 전체 영역과 관련되어 있음. 때문에 전사적으로 통합된 단일 고객 데이터가 필요

1.2 효과

비용절감을 통한 영업 생산성 향상, 고객 수익성 향상, 고객 만족도 향상, 고객 유지율 향상을 통한 매출 상승과 운영 효율성 향상 효과

대부분 기업에서는 기업 내의 제품 라인별, 비즈니스 모델별 또는 지리학적 위치에 따라 고객에 대한 다양한 관점 존재함. 각각의 그룹은 서로 다른 고객 데이터를 다른 방법으로 사용하고 다른 결과를 만듬. 이러한 사일로(silo) 체계로부터 통합 관점으로의 변화는 전사적 합의가 필요함. 그렇기 때문에 고객관계관리를 사일로의 벽을 부셔 버리는 다이너마이트라고도 함. 이러한 고객관계관리는 비즈니스 프로세스, 인적 자원 및 애플리케이션 기술의 적절한 통합을 필요로 함

지난 10년간 새로운 기술(소셜 미디어, 위치 추적 기술, 네트워크 모니터링, 멀티미디어 애플리케이션, 센서 네트워크 등)과 함께 다양한 비정형 고객 정보를 획득할 수 있는 기회 발생. 새로운 빅데이터 기술은 DW/BI에 새로운 데이터를 수용할 수 있도록 변화. 또한 새로운 데이터 원천은 기존의 관계형 데이터베이스 관리시스템(RDBMS) 기반의 기간계 시스템 데이터 보다 더욱 비정형적이고 다루기 어렵다는 특성을 가지고 있음

이전에 다루지 않았던 비정형적인 고객 정보가 이전에 하지 못했던 효과적인 고객관리활동에 도움이 된다면 판매채널의 급진적인 변화가 발생하고 더욱 효과적인 조직과 인센티브 시스템에도 변화가 발생

고객관계관리는 명확한 비전과 구체적인 추진전략이 필요하고 그렇지 않으면 매우 공허한 주제가 되기 쉬움. 특히 IT와 현업부서의 매우 공고한 협조체계가 반드시 필요함

2. 운영 CRM & 분석 CRM

2.1 운영 CRM

고객관계관리는 고객과의 상호 작용에서 발생하는 데이터를 분석하고자 하는 요구 사항 뿐만 아니라 운영 관점의 요구 사항도 동시에 수용하여야 하는 특성을 가짐. 영업, 마케팅, 생산, 서비스와 같은 운영 프로세스에서 발생하는 다양한 고객과의 접촉으로 발생하는 데이터는 사일로화 되어 나눠질 수 밖에 없음. 고객관계관리는 이러한 각각의 비즈니스 프로세스를 고객 활동 기준으로 크로스펑셔널(Cross-Functional)하게 통합해야 함. 각 비즈니스 프로세스에서 발생하는 핵심 고객 지표나 특징들이 잘 수집되고 연관성 있는 타 프로세스에 자연스럽게 흘러 들어갈 수 있도록 고객관계관리 프로세스가 정리되어야 함

DW기술은 고객을 360도 관점으로 분석할 수 있는 기반으로, 운영 CRM에서는 통합 관점의 고객 정보가 생성. 이 데이터를 전형적인 DW환경에서 다각도로 분석하는 것이 고객관계관리의 핵심.

2.2 분석 CRM

분석 CRM은 DW/BI 시스템에서 정확하고 통합된 고객 데이터를 기반으로 구현. 분석 CRM은 고객관계관리 기반의 고객 관리 활동이 얼마나 효과적인 의사결정이었는 지 평가할 수 있게 함. 고객 데이터는 상향 판매(Up-sell), 교차 판매(Cross-sell), 고객 타겟팅 정교화(pipoint inefficiency), 수요 발생(generate demand), 고객 유지 활동에 활용되면서 이전보다 활용도가 확대 발전됨. 이 뿐만 아니라, 시계열적으로 축적되고 통합된 데이터는 운영 시스템 프로세스를 평가하거나, 분석 CRM의 활용성을 더욱 높이는 데 활요되며 때때로 운영 시스템에 데이터 모델을 추가시키는 역할도 수행. 즉 분석 CRM이 운영 CRM을 개선시키는 역할까지도 수행하게 되는 데 이러한 과정을 클로즈드 루프(Closed Loop) 분석 CRM 이라고 함


운영 데이터 모델의 각 테이블(영업 대표, 콜센터, 웹 사이트 등)의 관계를 통해 비즈니스 프로세스 상에서 처리되는 내용을 파악 가능. 위와 같이 BI 애플리케이션 기반 데이터 분석 및 리포팅을 통해 필요에 따라 CRM 데이터 모델이 수정하게 되며, 이때 운영 운영 시스템에 해당 데이터 항목이 존재하지 않거나 관리되지 않으면 운영 시스템에 피드백을 줌. 고객관계관리 전략 수행에 반드시 필요한 항목을 운영 시스템 프로세스에 반영하여 필요한 데이털를 저장하도록 하고 이후 발생하는 고객과의 상호 작용부터 그 데이터는 쌓이게 됨

실시간 분석 환경에서 이러한 분석 정보는 운영 시스템의 웹 사이트나 콜센터 시스템으로 반드시 피드백 되어야 함. 실시간 환경에서의 피드백은 클로즈드 루프보다 더욱 타이트하게 연결되어야 함. 왜냐하면 실시간 아키텍처는 수집과 저장이 하나로 이뤄지고 수집 시스템으로 바로 피드백 되기 때문.

최근 운영 시스템은 과거와 달리 현재 시점의 데이터 뿐만 아니라 과거 이력성 데이터까지 모두 필요로 하게 됨. 에를 들어, 고객 신용도 체크를 실시간으로 처리하는 업무 프로세스는 해당 고객의 신용도를 산출하는 과거 이력 데이터도 함께 고려하여 산출. 하지만 운영 CRM에서 과거 이력성 데이터까지 처리하는 통합 요구 사항을 충족시키는 것은 매우 어려움.

기업이 고객에 집중하면 할수록 DW/BI 시스템은 필수적인 요소. 고객관계관리는 DW 기반하에서 운영될 수 밖에 없는 운명적인 관계를 가지고 있다고 말할 수 있음. DW/BI 환경은 더 많은 고객 정보를 수집하도록 확장되고, ETL 처리과정은 더 많은 원천 시스템의 다양한 이질적인 데이터를 통합하고 매핑하기 위해 더욱 복잡해짐. 무엇보다도 표준 고객 디멘션이 가장 중요함.

3. 고객 디멘션 속성

표준 고객 디멘션은 성공적인 고객관계관리 구축의 핵심 요인. 제대로 설계되고 성공적으로 구축한 표준 고객 디멘션은 고객관계관리 분석 환경을 구축하는 주춧돌.

고객 디멘션은 DW/BI 시스템 구축에 있어서 가장 어려운 과제 중 하나. 대기업인 경우에 고객 디멘션 테이블은 수백만 개 데이터와 수십 개 ~ 수백 개 컬럼으로 이루어진 거대 테이블인 경우가 많음. 또 때로는 매우 빠르게 데이터가 갱신되기도 함. 업계에서 1, 2위를 다투는 대형 유통사, 신용카드사 그리고 전국민을 상대로하는 정부기관은 1억 건이 넘는 거대 고객 디멘션을 구축하기도 함. 가장 어려운 숙제는 이러한 고객 데이터가 다수의 이종 시스템에 산재되어 있고, 이를 통합하여 구축할 때 어떻게 구축할 것인가임.

고객명과 주소 데이터 파싱, 아웃리거 디멘션, 고객 공통 속성 및 각 산업에 특화된 고객 속성. 고객과 관련된 속성은 우리가 상상하는 것보다 훨씬 더 많다. 고개을 더욱 상세하게 설명하기 위해 필요한 속성을 갖추려면 그만큼 고객 디멘션은 더욱 커지게 되고 분석할 만한 정보도 많아지게 된다.

3.1 고객명과 주소 파싱

고객명과 주소 속성은 산업에 상관없이 공통적으로 설계에 포함됨. 고객명과 주소 데이터 처리는 단순할 수 있지만 매우 유용한 정보임. 운영 시스템으로부터 수집해온 고객명과 주소가 성명1, 성명2, 성명3과 주소1 부터 주소6으로 구성되어 있다고 가정. 고객과 주소 컬럼이 여러 개이고 각각의 컬럼명이 컬럼명 자체로는 어떤 의미인지 알 수 없는 경우가 종종 있음. 이러한 정보를 유용하게 활용하기 위해 DW모델에서는 적절한 가공이 필요함. 예를 들어 주소 컬럼을 활용하여 고객 세그멘테이션을 한다고 가정. 아래와 같이 주소가 주소1, 주소1 컬럼으로 운영 시스템에서 관리되는 경우에 데이터 품질 이슈가 발생. 주소 컬럼 끝에 1, 2와 같이 구분 짓는 경우 그 용도가 여러 가지 목적으로 활용됨. 예를 들어, 고객의 자택 주소와 직장 주소를 주소1, 주소1로 구분할 수도 있고, 콜센터 상담원이 고객에게 받은 주소를 주소1, 현장 설치기사가 기록한 주소를 주소1라고 명명할 수도 있음. 이와 같이 여러 목적으로 사용되는 컬럼들을 DW에서는 데이터 품질을 보장할 수 있도록 수정하여 반영해야 함.

3.2 비영어권 국가 언어 적용시, 성명과 주소 설계할 때 고려 사항

비영어권 언어로 데이터베이스를 구축하는 경우 영어권 알파벳과 다른 유럽 국가별 알파벳 문자뿐만 아니라 키릴어, 일본어, 중국어와 같은 수십 개의 언어로 된 성명과 주소 데이터가 생성. 이 때 폰트 부재와 같은 사소한 문제가 아니라 캐릭터 셋이 변경되는 심각한 상황이 발생. 폰트는 캐릭터 셋에 대한 아티스트의 렌더링 작업을 통해 만들어 낼 수 있음. 표준영어에는 수백 개의 폰트가 있지만 표준영어는 다른 언어에 비해 작은 캐릭터 셋을 사용. 이 작은 캐릭터 셋은 ASCII (American Standard Code For Information Interchange)로 인코딩되어 있으며 최대 255개까지 사용할 수 있ㄴ는 8비트(1바이트) 체계로 구성되어 있음. 이중에서 표준영어 키보드에서 표현할 수 있는 문자는 100개. 이 100개의 문자는 여엉권 사용자가 사용하기에 충분. 하지만 한편으로는 이 100개의 문자로 비영어권 언어를 표현하기 위한 수천 개의 문자를 수용하기에는 부족.

유니코드 컨소시엄은 대부분의 국가 언어를 표현할 수 있도록 유니코드 표준을 정의함. 유니코드 표준 6.2.0는 110,182개의 캐릭터를 정의하였고 미국, 유럽, 중동, 아프리카, 인도, 아이아 패시픽 국가의 문자를 표현할 수 있으므로 국제 언어 셋을 구현하기 위해서는 반드시 사용해야 함.

OS 레벨 위의 모든 SW는 유니코드와 호환 가능하게 구성되어 데이터를 수집, 저장, 전송 및 출력할 수 있어야 함. DW 구축 시 사용되는 정렬(sorting)툴, 프로그래밍 언어 및 ETL툴과 같은 것들은 반드시 유니코드 호환성을 가져야 하며, DBMS, 쿼리 툴, 웹서버, 리포팅 툴, 브라우저 등과 같은 모든 애플리케이션도 유니코드 호환성을 가져야 함. DW/BI 아키텍트는 각각의 패키지 벤더에게 유니코드 호환성을 요청하는 것뿐만 아니라, 반드시 종단간 테스트를 수행해야 함. 즉, 데이터 수집부터 사용자 보고서까지 모든 과정을 테스트해야 함. 레거시 시스템에서 유니코드 캐릭터 셋을 사용한 성명과 주소 데이터를 캡처하고 시스템을 통해 전송. 그리고 최종적으로 사용자의 분석환경에서 해당 리포트를 프린트하여 확인해야 함.

ETL 작업 수행 중에 이슈가 발생할 수 있는 특이한 문자가 있는지 확인 필요. 이러한 점검은 간단할 것 같지만 실제로 많은 공수가 소요될 수 있음. 예를 들어 a-umlaut와 같은 문자는 노르웨이와 독일에서 각각 다르게 정렬될 수 있음. 노르웨이와 독일에서 a-umlat 문자를 처리하는 것과 같이 모든 다국어 문자 간의 순서를 대조하고 순서를 맞추는 것은 매우 어려운 일임.

다국어 환경에서 고객의 지리적 위치정보를 다루는 것은 더 복잡할 수 있음. DW/BI 시스템이 국내 고객만 있고, 외국 고객을 보유하지 않더라도 외국어 성명과 주소가 데이터베이스 존재 가능. 예를 들어, SCM 시스템의 제품 공급사 데이터나 HR시스템의 인사기록 카드에 외국어 성명과 주소가 존재 가능. 실제로 운영 시스템에서 수집한 데이터에는 기상천외한 데이터가 존재한다는 것을 충분히 고려해야 함.

다국적 데이터 처리 목표

유니코드를 적용하고 나서 성명, 주소 데이터 파싱 작업을 처리할 때 아래와 같은 목표를 설정하고 따라야 함.

  • 범용성과 일관성. 설계하는 시스템이 국제적으로 사용하는 것을 염두에 두고 있다면 BI 툴도 다국어 지원 여부를 검토해야 하며, 디멘션 테이블 설계 시에도 여러 국가별 언어 버전의 디멘션 데이털르 생성해야 함. 하지만 여러 가지 언어별로 만들어진 디멘션은 미묘한 문제 발생 가능

    • 데이터 정렬이 언어별로 달라질 수 있음. BI 리포트를 각각의 언어별로 조회했을 때 정렬 순서가 달라지거나 자국어로 조회할 때 정렬이 되지 않을 수도 있음

    • 한글로는 다른 단어인데, 외국어로는 같은 단어로 번역되었을 경우 디멘션 카디널리티가 달라지게 됨. OLAP 툴에서 피벗 테이블을 만든 경우, 디멘션별 소계의 위치가 달라질 수 있고 중복된 헤더가 발생할 수도 있음. 이러한 문제를 해결하기 위해 리포트를 실행시키고 난 후에 디멘션을 번역하여야 함. 자국어로 피벗 테이블을 생성하여 디멘션 그룹별 소계를 생성한 후에 최종적으로 생성해야 할 타국어 디멘션으로 번역해야 원래의 모양과 동일한 리포틑를 생성 가능,

    • BI 툴의 각종 메세지나 프롬프트도 번역되어 제공되어야 함.

  • 데이터 수집부터 사용자 분석 단계까지의 데이터 품질 보장과 데이터 오류 발견 시, 역추적이 가능하게 구성해야 함. DW에서는 다국어 성명과 주소를 한 번의 작업으로 정제하여 사용할 수는 없음. 먼저 데이털르 수집하고 데이터의 정제 과정을 거쳐 최종 데이털르 저장해야 함. 마지막 단계인 사용자의 데이터 분석을 통해 지리적, 인구통계학적인 분석과 출력 과정을 거치면서 오류가 발견되면 반대 방향으로 오류의 원인을 찾아 제거하는 과정을 거침.

  • 문화적 오류 보정. 외국인의 성과 이름의 위치가 각 나라별로 다르다는 것을 인식하지 못하고 데이터를 설계하게 되면 이름과 성이 뒤바뀌는 실수를 범할 수 있음. 이러한 문화적 차이에 따른 오류들이 발생하면 외국 고객사는 로컬 업체를 선택하게 되므로 유의해야 함.

  • 실시간 고객 대응. 실시간 DW/BI를 구축하게 되면 실시간 고객 대응을 위한 운영 시스템 역할을 수행할 수도 있음. DW 시스템이 해당 고객에게 가장 적당한 인사말을 추천해서 콜센터 상담원 화면에 띄우는 데 약 5초 이하의 시간이 걸리게 됨.

  • 다른 종류의 주소들. 최근 IT기술 발전에 따라 DW에 저장하는 데이터가 다양해지고 있음. IP Address, URL과 같은 주소나 네트워크 보안에서 사용되는 암호 토큰(Security Token) 저장도 고려해야 함.

다국어 환경의 주소와 유사하게 전화번호도 어는 국가에서 전화가 발신되었느냐에 따라 다양한 전화번호 체계를 수용할 수 있도록 설계해야 함.

4. 고객 지향 일자 데이터

고객 디멘션에는 첫 구매 일자, 마지막 구매 일자, 생일 등과 같은 일자 속성을 포함. 이러한 속성은 일자 데이터 타입으로 설계되더라도 계절, 분기, 회계기준 기간 등으로 집계할 필요가 있음. 일자 속성은 일자 디멘션과 관계를 가지면서 팩트 테이블의 외래 키로 설계됨. 이러한 여러 개의 일자 속성은 데이터 자체는 동일하나 팩트 테이블에서 사용되는 의미가 각각 다른 경우도 있음. 이러한 경우에 물리적인 일자 디멘션 테이블을 여러 개 만들지 않고 뷰나 SQL의 별칭(Alias)과 같은 형태로 표현해야 함. 예를 들어 ‘첫 번째 구매 일자’ 디멘션은 물리적으로는 일자 디멘션과 동일하며 뷰로 만들어서 사용할 수 있음. 이러한 설계는 아래와 같이 아웃리거 디멘션과 같이 설계 할 수 있음


위와 같이 고객 디멘션 왼쪽의 최초 구매 일자 디멘션은 팩트 테이블과 직접적인 관게를 맺지 않고 고객 디멘션과 일대다(1:N) 관계를 가지게 되는 데 이러한 디멘션을 아웃리거 디멘션이라고 함. (스노우플레이크 스키마)
