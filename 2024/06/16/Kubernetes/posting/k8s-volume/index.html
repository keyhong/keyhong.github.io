Volume

Kubernetes

Stateless한 애플리케이션

일반적으로 컨테이너는 stateless(상태를 저장하지 않는) 방식의 앱 컨테이너를 사용합니다. 이러한 장점으로 어떤 노드에 문제가 발생했을 때, 다른 노드로 옮길 수 있는 장점이 있습니다.

필요성

하지만 이럴 경우 기존의 데이터가 사라집니다. 데이터베이스와 같이 데이터를 영구적으로 저장하는 앱에서는 큰 문제가 됩니다. 그럼 데이터베이스 앱플리케이션이 문제가 생겼을 때, 쿠버네티스가 다른 노드에 새로 컨테이너를 올려야 하는 상황이라면 기존 데이터를 어떻게 보관할까요? 바로 volume에 해답이 있습니다. 일반적으로 nfs처럼 여러 노드에서 원격으로 접속 가능한 volume을 만들고 pod에 volume을 매핑해서 앱을 옮겨도 똑같은 환경에서 실행 가능합니다.

Volume

컨테이너가 외부 스토리지에 액세스하고 공유하는 방법

  • pod의 각 컨테이너에는 고유의 분리된 파일 시스템 존재

  • volume은 pod의 컴포넌트이며 pod의 .spec에 의해 정의

  • 독립적인 쿠버네티스 오브젝트가 아니며 스스로 생성, 삭제 불가

  • 각 컨테이너의 파일 시스템의 볼륨을 마운트하여 생성

k8s에서 지원하는 volume plugin

volume도 목적에 따라 여러 가지로 나눠지게 됩니다.

  • 임시 volume: emptyDir (컨테이너끼리 데이터 공유 목적)

  • 컨테이너가 실행된 로컬 디스크를 사용하는 volume: hostpath, local (노드의 관리 목적, 노드와 공유 목적)

  • 클라우드 프로바이더가 제공하는 volume 서비스: awsElasticBlockStore, azureDisk, gcePersistentDisk ..

  • 네트워크 서버의 디스크를 사용하는 volume: iSCSI, NFS, cephFS, glusterFS …

주요 사용 가능한 volume의 유형

  • emptyDir: 일시적인 데이터 저장, 비어 있는 디렉터리

  • hostPath: 포드에 호스트 노드의 파일 시스템에서 파일이나 디렉터리를 마운트

  • nfs: 기존 NFS (네트워크 파일 시스템) 공유가 포드에 장착

  • gcePersistentDisk: 구글 컴퓨트 엔진 (GCE) 영구 디스크 마운트 (다른 프로바이더 volume도 가능)

  • persistentVolumeClaim: 사용자가 특정 클라우드 환경의 세부 사항을 모른 채 GCE PersistentDisk 또는 iSCSI 볼륨과 같은 내구성 스토리지를 요구(Claim) 할 수 있는 방법

  • configMap, Secret, downwardAPI: 특수한 유형의 볼륨

  • 볼륨 관련 레퍼런스: https://kubernetes.io/docs/concepts/storage/volumes/

emptyDir

pod가 실행되는 host의 디스크를 임시로 컨테이너 볼륨으로 할당해서 사용하는 방법

  • pod가 사라지면 emptyDir의 데이터도 함께 사라집니다.

  • 주로 메모리와 디스크를 함께 이용하는 대용량 데이터 계산에 사용합니다.

  • 과학 연산(오랜 연산 시간이 걸릴 때, 중간 데이터 저장용으로 디스크가 필요할 때 사용)에도 사용합니다.

  • 연산 중 컨테이너에 문제가 생겨도 pod만 살아있다면 pod내 컨테이너를 재시작해 emptyDir에 저장해둔 데이터를 계속 사용 가능합니다.

  • 동일 Pod 내의 여러 컨테이너 간에 임시 데이터를 공유하고 전달하는 데도 사용됩니다.

emptydir-pod.yaml
apiVersion: v1
kind: Pod
metadata:
    name: emptydir-pod
spec:
    containers:
    - name: <container-name>
      image: <container-image>
      volumeMounts:
      - mountPath: <마운트할 컨테이너 내부 경로 (i.e. /emptydir)>
        name: <생성한 볼륨 이름 (i.e. emptydir-v)>
    volumes:
    - name: <생성할 볼륨 이름 (i.e. emptydir-v)>
      emptyDir: {}

볼륨 정의 부분(.spec.volumes[])

.spec.volumes[] 하위 필드에서는 사용하려는 볼륨을 먼저 정의합니다.

  • .spec.volumes[].name: 생성할 볼륨의 이름을 지정합니다.

  • .spec.volumes[].emptyDir: 필드값을 {}(빈 값)으로 지정합니다.

컨테이너 내 마운트 부분 (.spec.containers[].volumeMounts)

.spec.containers[].volumeMounts 하위 필드에서는 생성한 볼륨을 컨테이너 내부와 마운트 합니다.

  • .spec.containers[].volumeMounts.mountPath: .spec.volumes[]에서 만든 volume을 마운트 할 컨테이너 내부의 디렉터리 경로입니다.

  • .spec.containers[].volumeMounts.name: .spec.volumes[]에서 만든 볼륨의 이름입니다.

hostPath

pod가 실행된 host의 파일이나 디렉터리를 pod에 마운트하는 방법

  • emtyDir과 다르게 pod가 삭제되어도 host에 데이터가 남습니다.(영구 스토리지)

  • Pod가 생성된 Node File system의 File 및 Directory에 Volume으로 mount 합니다.

  • 데이터가 특정 Node에 저장되므로 Pod가 재생성 되어 Node가 바뀌면 이전 Node에 저장되어 있어 hostPath Volume의 데이터를 읽을수 없습니다. 즉, 다른 노드의 pod끼리 데이터 공유는 아닙니다.

  • Node 내 저장된 파일을 Pod들 간 공유해야 하는 경우에 사용되지만 단일 Node에만 생성되므로 테스트 상황에 적합합니다.

이용 예시

  • /var/lib/docker 같은 도커 시스템용 디렉터리를 컨테이너에서 사용할 때나 시스템용 디렉터리를 마운트해서 시스템 모니터링하는 용도로도 사용 가능합니다. (i.e. gcp k8s의 로그수집기 fluentd)

  • 컨테이너에서 cAdvisor의 실행. /sys를 hostPath 로 이용함

코드

hostpath-pod.yaml
apiVersion: v1
kind: Pod
metadata:
	name: hostpath-pod
spec:
	containers:
	- name: <container-name>
	  image: <container-image>
	  volumeMounts:
	  - mountPath: <마운트 할 컨테이너 내부 경로 (i.e. /container-tmp)>
	    name: <생성한 볼륨 이름 (i.e. hostpath-v)>
	volumes:
	- name: <생성할 볼륨 이름 (i.e. hostpath-v)>
	  hostPath:
	    path: <공유할 호스트의 파일 또는 디렉터리(i.e. /tmp)>
		type: Directory
hostPath 테스트
# hostpath-pod 컨테이너에 진입
$ kubectl exec hostpath-pod -it -- sh 

# 마운트한 디렉터리에 파일 생성
$ cd /container-tmp; touch test.txt

# 컨테이너를 나온 후, 파드 삭제
$ kubectl delete pod hostpath-pod

# host에서 지정한 볼륨 디렉터리 확인 (컨테이너에서 만든 test.txt가 host 디렉터리 경로에 존재) 
$ ls /tmp/
test.txt

볼륨 정의 부분(.spec.volumes[])

.spec.volumes[] 하위 필드에서는 사용하려는 볼륨을 먼저 정의합니다.

  • .spec.volumes[].name: 생성할 볼륨의 이름을 지정합니다.

  • .spec.volumes[].hostPath.path: 호스트에서 컨테이너와 공유할 파일 또는 디렉터리 경로를 지정합니다.

.spec.volumes[].hostPath.type 필드값

필드 값 설명
Directory 설정한 경로에 디렉터리가 반드시 존재해야 합니다. 없으면 pod는 ContainerCreating 상태를 유지합니다.
DirectoryOrCreate 설정한 경로에 디렉터리가 없으면 permission 755인 빈 디렉터리를 생성합니다.
File 설정한 경로에 파일이 있어야 합니다. 없으면 pod는 ContainerCreating 상태를 유지합니다.
FileOrCreate 설정한 경로에 파일이 없으면 permission 644인 빈 파일을 생성합니다.
Socket 설정한 경로에 unix 소켓 파일이 있어야 합니다.
CharDevice 설정한 경로에 문자 디바이스가 있는 지 확인합니다.
BlockDevice 설정한 경로에 블록 디바이스가 있는 지 확인합니다.

컨테이너 내 마운트 부분 (.spec.containers[].volumeMounts)

.spec.containers[].volumeMounts 하위 필드에서는 생성한 볼륨을 컨테이너 내부와 마운트 합니다.

  • .spec.containers[].volumeMounts.mountPath: .spec.volumes[]에서 만든 volume을 마운트 할 컨테이너 내부의 디렉터리 경로입니다.

  • .spec.containers[].volumeMounts.name: .spec.volumes[]에서 만든 볼륨의 이름입니다.

경고

HostPath 볼륨에는 많은 보안 위험이 있으며, 가능하면 HostPath를 사용하지 않는 것이 좋습니다.

  • HostPath 볼륨을 사용해야 하는 경우, 필요한 파일 또는 디렉터리로만 범위를 지정하고 ReadOnly로 마운트해야 해야 합니다.

  • AdmissionPolicy를 사용하여 특정 디렉터리로의 HostPath 액세스를 제한하는 경우, readOnly 마운트를 사용하는 정책이 유효하려면 volumeMounts가 반드시 지정되어야 합니다.
