Project Configs

DBT

analysis-paths

dbt_project.yml
analysis-paths: [directorypath]

Definition

analyses이 위치한 디렉토리의 사용자 지정 목록을 지정합니다.

Default

이 설정을 지정하지 않으면, dbt는 .sql 파일을 분석으로 컴파일하지 않습니다.

그러나, dbt init command는 이 값을 analyses로 채웁니다 (source)

Examples

Use a subdirectory named analyses

이 값은 dbt init command에 의해 채워집니다.

dbt_project.yml
analysis-paths: ["analyses"]

Use a subdirectory named custom_analyses

dbt_project.yml
analysis-paths: ["custom_analyses"]

asset-paths

dbt_project.yml
asset-paths: [directorypath]

Definition

analyses이 위치한 디렉토리의 사용자 정의 목록을 지정하세요.

docs generate 명령어의 일부로 target 디렉토리에 복사할 사용자 정의 디렉토리 목록을 선택적으로 지정할 수 있습니다. 이것은 프로젝트 문서에서 귀하의 저장소 내 이미지를 렌더링하는 데 유용합니다.

Default

기본적으로 dbt는 docs generate의 일환으로 추가 파일을 복사하지 않습니다. 즉, asset-paths: []입니다.

Examples

Compile files in the assets subdirectory as part of docs generate

dbt_project.yml
asset-paths: ["assets"]

이 디렉토리에 포함된 모든 파일은 dbt docs generate의 일환으로 target/ 디렉토리로 복사됩니다. 이로써 프로젝트 문서에서 이미지로 접근할 수 있게 됩니다.

이미지를 설명에 포함하는 방법에 대한 자세한 내용은 here에서 전체 글을 확인하세요.

clean-targets

dbt_project.yml
clean-targets: [directorypath]

Definition

선택적으로 dbt clean command로 제거할 사용자 지정 디렉터리 목록을 지정할 수 있습니다. 따라서 이 목록에는 아티팩트(예: 컴파일된 파일, 로그, 설치된 패키지)를 포함하는 디렉터리만 포함해야 합니다.

Default

이 configuration이 dbt_project.yml 파일에 포함되지 않은 경우, clean 명령어는 target-path의 파일을 제거할 것입니다.

Examples

Remove packages and compiled files as part of dbt clean (preferred)

패키지와 컴파일된 파일을 모두 제거하려면, clean-targets 구성에 packages-install-path 구성의 값을 포함하십시오.

dbt_project.yml
clean-targets:
    - target
    - dbt_packages

이제, dbt clean을 실행하십시오.

target 및 dbt_packages 디렉터리가 모두 제거됩니다.

Note: 이것은 init 명령으로 생성된 dbt starter project의 구성입니다.

Remove logs when running dbt clean

dbt_project.yml
clean-targets: [target, dbt_packages, logs]

config-version

dbt v1.5부터는 config-version: 태그가 선택 사항입니다.

dbt_project.yml
config-version: 2

Definition

dbt_project.yml을 v2 구조를 사용하도록 지정하십시오.

이 configuration은 선택 사항입니다.

Default

  • 이 구성을 사용하지 않으면, dbt는 dbt_project.yml이 dbt v0.19.0에서 사용이 deprecated된 version 1 syntax를 사용하는 것으로 간주합니다.

dbt_project.yml

모든 dbt project에는 dbt_project.yml 파일이 필요합니다. 이 파일은 dbt가 디렉토리를 dbt 프로젝트로 인식하는 방법입니다. 또한 dbt가 프로젝트를 운영하는 방법을 알려주는 중요한 정보를 포함하고 있습니다.

dbt는 몇 군데에서 YAML을 사용합니다. YAML이 처음이라면 배열, 사전 및 문자열이 어떻게 표현되는지 학습하는 것이 좋습니다.

기본적으로 dbt는 현재 작업 디렉토리와 해당 상위 디렉토리에서 dbt_project.yml을 찾지만, --project-dir 플래그나 DBT_PROJECT_DIR 환경 변수를 사용하여 다른 디렉토리를 설정할 수 있습니다.

dbt v1.5 이상에서는 dbt_project.yml 파일의 dbt-cloud config 아래에 project-id를 사용하여 dbt Cloud 프로젝트 ID를 지정할 수 있습니다. 프로젝트 ID를 확인하려면 dbt Cloud 프로젝트 URL을 확인하세요. 예: https://cloud.getdbt.com/11/projects/123456에서 프로젝트 ID는 123456입니다.

참고할 점은 dbt_project.yml 파일에서 config가 아닌 경우(예: macros), “property”을 설정할 수 없습니다. 이는 모든 종류의 리소스에 적용됩니다. 자세한 내용은 Configs and properties를 참조하세요.

Naming convention

It’s important to follow the correct YAML naming conventions for the configs in your dbt_project.yml file to ensure dbt can process them properly. This is especially true for resource types with more than one word.

Use dashes (-) when configuring resource types with multiple words in your dbt_project.yml file. Here’s an example for saved queries:

dbt_project.yml
saved-queries:  # Use dashes for resource types in the dbt_project.yml file.
  my_saved_query:
    config:
      +cache:
        enabled: true

Use underscore (_) when configuring resource types with multiple words for YAML files other than the dbt_project.yml file. For example, here’s the same saved queries resource in the semantic_models.yml file:

models/semantic_models.yml
saved_queries:  # Use underscores everywhere outside the dbt_project.yml file.
  - name: saved_query_name
    ... # Rest of the saved queries configuration.
    config:
      cache:
        enabled: true

.dbtignore

dbt project의 루트에 .dbtignore 파일을 생성하여 dbt가 완전히 무시해야 하는 파일을 지정할 수 있습니다. 이 파일은 .gitignore file과 동일한 문법을 사용하여 작동합니다. 패턴과 일치하는 파일 및 하위 디렉토리는 dbt에 의해 읽히거나, 해석되거나, 다른 방식으로 감지되지 않습니다—마치 존재하지 않는 것처럼.

Examples

.dbtignore
# .dbtignore

# ignore individual .py files
not-a-dbt-model.py
another-non-dbt-model.py

# ignore all .py files
**.py

# ignore all .py files with "codegen" in the filename
*codegen*.py

dispatch (config)

dispatch:
  - macro_namespace: packagename
    search_order: [packagename]
  - macro_namespace: packagename
    search_order: [packagename]

Definition

일부 네임스페이스에서 매크로의 dispatch 검색 위치를 선택적으로 재정의할 수 있습니다. 지정되지 않으면 기본적으로 dispatch는 first 루트 프로젝트에서 검색한 다음 macro_namespace으로 명명된 패키지에서 구현을 찾습니다.

Examples

저는 dbt_utils 패키지를 spark_utils 호환 패키지로 “shim”하고 싶습니다.

dbt_project.yml
dispatch:
  - macro_namespace: dbt_utils
    search_order: ['spark_utils', 'dbt_utils']

dbt_utils 패키지의 일부 매크로를 루트 프로젝트('my_root_project')에서 다시 구현했으며, 내 버전이 우선되기를 원합니다. 그렇지 않으면 dbt_utils의 버전으로 되돌아갑니다.

Note: 이것은 기본 동작입니다. 필요에 따라 검색 순서를 명시적으로 표현할 수 있습니다.

dbt_project.yml
dispatch:
  - macro_namespace: dbt_utils
    search_order: ['my_root_project', 'dbt_utils']

docs-paths

dbt_project.yml
docs-paths: [directorypath]

Definition

선택적으로 docs blocks이 위치한 사용자 정의 디렉터리 목록을 지정할 수 있습니다.

Default

기본적으로 dbt는 문서 블록을 찾기 위해 모든 리소스 경로 (즉, model-paths, seed-paths, analysis-paths, macro-paths 및 snapshot-paths의 결합된 목록)를 검색합니다. 이 옵션이 구성되면 dbt는 문서 블록을 찾기 위해 지정된 디렉터리만 검색합니다.

Examples

문서 블록에는 docs라는 하위 디렉터리를 사용합니다.

dbt_project.yml
docs-paths: ["docs"]

Note: 일반적으로 우리는 dbt의 기본 동작을 선호하여 이 구성을 생략합니다.

macro-paths

dbt_project.yml
macro-paths: [directorypath]

Definition

선택적으로 매크로가 위치한 디렉토리의 사용자 지정 목록을 지정할 수 있습니다. 모델과 매크로를 같은 위치에 둘 수 없다는 점에 유의하십시오.

Default

기본적으로, dbt는 macros라는 이름의 디렉토리에서 매크로를 검색합니다, 즉, macro-paths: ["macros"]

Examples

dbt_project.yml
macro-paths: ["custom_macros"]

Default

기본적으로 dbt는 모델과 소스를 models 디렉터리에서 찾습니다. 즉, model-paths: ["models"]

Troubleshooting

Invalid project name

Encountered an error while reading the project:
  ERROR: Runtime Error
  at path ['name']: 'jaffle-shop' does not match '^[^\\d\\W]\\w*$'
Runtime Error
  Could not run dbt

이 프로젝트에는 다음이 포함됩니다:

dbt_project.yml
name: jaffle-shop

이 경우, 프로젝트 이름을 snake_case로 변경하십시오:

dbt_project.yml
name: jaffle_shop

on-run-start & on-run-end

dbt_project.yml
on-run-start: sql-statement | [sql-statement]
on-run-end: sql-statement | [sql-statement]

Definition

다음 명령어가 시작되거나 끝날 때 실행할 SQL 문(또는 SQL 문 목록): dbt build, dbt compile, dbt docs generate, dbt run, dbt seed, dbt snapshot, dbt test.

on-run-start와 on-run-end 훅은 또한 SQL 문을 반환하는 매크로를 호출할 수도 있습니다.

Usage notes

  • on-run-end 훅은 추가적인 Jinja 변수를 컨텍스트에서 사용할 수 있습니다 — docs를 확인하세요.

Examples

이전 버전의 dbt에서는 post-hook의 가장 일반적인 용도는 grant 문을 실행하여 모델을 생성한 직후 데이터베이스 권한을 적용하는 것이었습니다. 대신 grants resource config을 사용하여 dbt 모델이 실행될 때 자동으로 권한을 적용하는 것을 권장합니다.

Grant privileges on all schemas that dbt uses at the end of a run

이것은 on-run-end 훅에서만 사용할 수 있는 schemas 변수를 활용합니다.

dbt_project.yml
on-run-end:
  - "{% for schema in schemas %}grant usage on schema {{ schema }} to group reporter; {% endfor %}"

Call a macro to grant privileges

dbt_project.yml
on-run-end: "{{ grant_select(schemas) }}"

Additional examples

here에서 더 심도 있는 예제를 모았습니다.

profile

dbt_project.yml
profile: string

Definition

dbt 프로젝트가 data warehouse에 연결하는 데 사용할 프로필.

  • dbt Cloud에서 개발 중인 경우: 이 구성은 적용되지 않습니다.
  • 로컬에서 개발 중인 경우: 이 구성은 필수이며, 명령줄 옵션(예: --profile)이 제공되지 않는 한 필요합니다.

Related guides

  • Connecting to your warehouse using the command line

Recommendation

종종 조직은 하나의 data warehouse만 보유하므로, 프로필 이름을 snake_case로 조직 이름을 사용하는 것이 합리적입니다. 예를 들어:

  • profile: acme
  • profile: jaffle_shop

프로필 이름에 데이터 웨어하우스 기술의 이름을 포함하는 것도 합리적입니다. 특히 여러 개의 웨어하우스를 보유한 경우에는 더욱 그렇습니다. 예를 들어:

  • profile: acme_snowflake
  • profile: jaffle_shop_bigquery
  • profile: jaffle_shop_redshift

query-comment

dbt_project.yml
query-comment: string

query-comment 구성은 다음과 같이 딕셔너리 입력도 허용합니다:

dbt_project.yml
models:
  my_dbt_project:
    +materialized: table

query-comment:
  comment: string
  append: true | false
  job-label: true | false  # BigQuery only

Definition

dbt가 데이터베이스에 대해 실행하는 각 쿼리에 주석으로 삽입할 문자열입니다. 이 주석은 모델 및 테스트와 같은 특정 dbt 리소스에 SQL 문을 할당하는 데 사용될 수 있습니다.

query-comment 구성은 문자열을 반환하는 매크로를 호출할 수도 있습니다.

Default

기본적으로 dbt는 실행하는 리소스의 정보를 포함하는 JSON 주석을 쿼리 맨 위에 삽입합니다. 이 정보에는 dbt 버전, 프로필 및 타겟 이름, 그리고 노드 ID 등이 포함됩니다. 예를 들어:

/* {"app": "dbt", "dbt_version": "0.15.0rc2", "profile_name": "debug",
    "target_name": "dev", "node_id": "model.dbt2.my_model"} */

create view analytics.analytics.orders as (
    select ...
  );

Using the dictionary syntax

딕셔너리 구문은 두 개의 키를 포함합니다:

  • comment(선택 사항, 기본값은 위의 설명 참조): 쿼리에 주석으로 삽입될 문자열입니다.
  • append(선택 사항, 기본값=false): 주석을 쿼리의 맨 아래에 추가할지 여부를 지정합니다. 기본적으로 주석은 쿼리의 맨 위에 추가됩니다 (append: false).

이 구문은 Snowflake와 같은 데이터베이스에서 유용합니다. Snowflake는 보통 remove leading SQL comments 때문에 이 구성을 사용하여 주석을 원하는 위치에 삽입할 수 있습니다.

Examples

Prepend a static comment

이 설정은 dbt가 실행하는 모든 SQL 쿼리의 맨 위에 주석으로 /* executed by dbt */를 추가합니다. append: false로 설정되어 있으므로 주석은 쿼리의 맨 위에 삽입됩니다.

dbt_project.yml
models:
  my_dbt_project:
    +materialized: table

query-comment:
  comment: string
  append: true | false
  job-label: true | false  # BigQuery only

Example output:

/* executed by dbt */

select ...

Disable query comments

dbt_project.yml
query-comment:

Or:

dbt_project.yml
query-comment: null

Prepend a dynamic comment

다음 예제는 활성화된 dbt 타겟에서 구성된 user에 따라 달라지는 주석을 삽입하는 방법을 보여줍니다.

dbt_project.yml
query-comment: "run by {{ target.user }} in dbt"

Example output:

/* run by drew in dbt */

select ...

Append the default comment

다음 예제는 딕셔너리 구문을 사용하여 기본 주석을 추가(선행하지 않음)하는 방법을 보여줍니다.

comment: 필드가 생략되어 기본값이 추가될 수 있도록 합니다.

dbt_project.yml
query-comment:
  append: True

Example output:

select ...
/* {"app": "dbt", "dbt_version": "0.16.`0rc2`", "profile_name": "debug", "target_name": "dev", "node_id": "model.dbt2.my_model"} */
;

BigQuery: include query comment items as job labels

만약 query-comment.job-label이 true로 설정되면, dbt는 실행하는 쿼리에 대해 query comment 항목들(만약 딕셔너리로 지정된 경우) 또는 comment 문자열을 job label로 포함시킵니다. 이 레이블들은 BigQuery-specific config에서 지정된 레이블들에 추가하여 포함됩니다.

dbt_project.yml
query-comment:
  job-label: True

Append a custom comment

다음 예제는 활성 dbt 타겟에서 구성된 user에 따라 달라지는 주석을 추가하는 데 딕셔너리 구문을 사용합니다.

dbt_project.yml
query-comment:
  comment: "run by {{ target.user }} in dbt"
  append: True

Example output:

select ...
/* run by drew in dbt */
;

Intermediate: Use a macro to generate a comment

query-comment 구성은 dbt 프로젝트 내의 매크로를 참조할 수 있습니다. 단순히 macros 디렉터리에 원하는 이름의 매크로(query_comment이 좋은 시작점입니다!)를 생성하면 됩니다.

macros/query_comment.sql
{% macro query_comment() %}

  dbt {{ dbt_version }}: running {{ node.unique_id }} for target {{ target.name }}

{% endmacro %}

그런 다음 dbt_project.yml 파일에서 해당 매크로를 호출하면 됩니다. YAML 파서가 {를 딕셔너리의 시작으로 해석하지 않도록 매크로를 따옴표로 묶어야 합니다.

dbt_project.yml
query-comment: "{{ query_comment() }}"

Advanced: Use a macro to generate a comment

다음 예제는 JSON 쿼리 주석을 보여줍니다. 이를 통해 dbt 프로젝트의 성능 특성을 이해할 수 있습니다.

macros/query_comment.sql
{% macro query_comment(node) %}
    {%- set comment_dict = {} -%}
    {%- do comment_dict.update(
        app='dbt',
        dbt_version=dbt_version,
        profile_name=target.get('profile_name'),
        target_name=target.get('target_name'),
    ) -%}
    {%- if node is not none -%}
      {%- do comment_dict.update(
        file=node.original_file_path,
        node_id=node.unique_id,
        node_name=node.name,
        resource_type=node.resource_type,
        package_name=node.package_name,
        relation={
            "database": node.database,
            "schema": node.schema,
            "identifier": node.identifier
        }
      ) -%}
    {% else %}
      {%- do comment_dict.update(node_id='internal') -%}
    {%- endif -%}
    {% do return(tojson(comment_dict)) %}
{% endmacro %}

위와 같이 이 매크로를 호출하세요:

dbt_project.yml
query-comment: "{{ query_comment(node) }}"

Compilation context

쿼리 주석을 생성할 때 사용할 수 있는 다음 컨텍스트 변수들이 있습니다:

Context Variable Description
dbt_version 사용 중인 dbt의 버전입니다. 릴리즈 버전 관련 자세한 내용은 Versioning 문서를 참조하세요.
env_var env_var를 참조하세요.
modules modules를 참조하세요.
run_started_at dbt 호출이 시작된 시점
invocation_id dbt 호출의 고유 ID
fromjson fromjson를 참조하세요.
tojson tojson를 참조하세요.
log log를 참조하세요.
var var를 참조하세요.
target target를 참조하세요.
connection_name 연결의 내부 이름을 나타내는 문자열입니다. 이 문자열은 dbt에 의해 생성됩니다.
node 파싱된 노드 객체의 사전 표현입니다. node.unique_id, node.database, node.schema 등을 사용할 수 있습니다.

Configuring quoting in projects

dbt_project.yml
quoting:
  database: true | false
  schema: true | false
  identifier: true | false

Definition

dbt가 데이터베이스, 스키마 및 식별자를 따옴표로 묶어야 할지 선택적으로 구성하는 옵션입니다:

  • 관계(테이블/뷰) 생성 시
  • ref 함수를 직접적인 관계 참조로 해석할 때

Default

기본값은 데이터베이스에 따라 다릅니다.

대부분의 어댑터에서는 기본적으로 식별자를 따옴표로 묶는 것이 true로 기본 설정되어 있습니다.

왜냐하면 따옴표를 사용하든 사용하지 않든 관계에서 선택하는 것은 동일하기 때문입니다. 그러나 식별자를 따옴표로 묶으면 예약어와 특수 문자를 식별자에 사용할 수 있습니다. 이는 데이터베이스에서 식별자로 사용할 수 있는 범위를 확장하지만, 가능한 경우 이러한 사용을 피하는 것이 좋습니다.

dbt_project.yml
quoting:
  database: true
  schema: true
  identifier: true

Examples

프로젝트에서 quoting을 false로 설정하세요.

dbt_project.yml
quoting:
  database: false
  schema: false
  identifier: false

그러면 dbt는 따옴표 없이 관계(테이블/뷰)를 생성합니다.

create table analytics.dbt_alice.dim_customers

Recommendation

Snowflake

모든 quoting 구성을 False로 설정합니다. 이는 예약어를 식별자로 사용할 수 없음을 의미하지만, 보통 이러한 예약어를 사용하지 않는 것이 좋은 방법입니다.

만약 Snowflake 소스 테이블이 따옴표로 묶인 데이터베이스, 스키마 또는 테이블 식별자를 사용한다면, 이를 source.yml 파일에서 구성할 수 있습니다. Refer to configuring quoting for more info.

Explanation:

대부분의 데이터베이스는 따옴표를 사용하지 않은 식별자를 소문자로 변환하지만, Snowflake는 따옴표를 사용하지 않은 식별자를 대문자로 변환합니다. 따라서 모델 이름이 소문자로 작성되고 따옴표로 묶인 경우, 따옴표 없이 참조할 수 없습니다! 아래 예제를 확인하여 더 많은 정보를 얻으세요.

/*
    You can run the following queries against your database
    to build an intuition for how quoting works on Snowflake.
*/

-- This is the output of an example `orders.sql` model with quoting enabled
create table "analytics"."orders" as (

  select 1 as id

);

/*
    These queries WILL NOT work! Since the table above was created with quotes,
    Snowflake created the orders table with a lowercase schema and identifier.

    Since unquoted identifiers are automatically uppercased, both of the
    following queries are equivalent, and neither will work correctly.
*/

select * from analytics.orders;
select * from ANALYTICS.ORDERS;

/*
    To query this table, you'll need to quote the schema and table. This
    query should indeed complete without error.
*/

select * from "analytics"."orders";


/*
    To avoid this quoting madness, you can disable quoting for schemas
    and identifiers in your dbt_project.yml file. This means that you
    won't be able to use reserved words as model names, but you probably
    shouldn't be doing that anyway! Assuming schema and identifier quoting is
    disabled, the following query would indeed work:
*/

select * from analytics.orders;

Other warehouses

기본값을 데이터 웨어하우스에 대해 유지하세요.

require-dbt-version

dbt_project.yml
require-dbt-version: version-range | [version-range]

Definition

require-dbt-version을 사용하여 프로젝트가 특정 dbt 버전 범위에서만 작동하도록 제한할 수 있습니다.

이 설정을 사용하면, 지원되지 않는 dbt 버전으로 프로젝트를 실행하려는 사용자에게 유용한 오류 메시지가 dbt에서 전송됩니다. 이는 패키지 유지자(dbt-utils와 같은)가 사용자의 dbt 버전이 패키지와 호환되는지 확인하는 데 유용할 수 있습니다. 이 설정을 사용하면 팀 전체가 로컬 개발에서 동일한 dbt 버전을 사용하여 동기화를 유지할 수 있어서 변경된 동작으로 인한 호환성 문제를 방지할 수 있습니다.

만약 이 구성이 지정되지 않으면, 버전 검사가 수행되지 않습니다.

YAML quoting

이 구성은 YAML 파서에 의해 문자열로 보간되어야 합니다. 따라서 구성의 값을 따옴표로 묶어야 하며, 공백을 피해야 합니다. 예를 들어:

dbt_project.yml
# ✅ These will work
require-dbt-version: ">=1.0.0" # Double quotes are OK
require-dbt-version: '>=1.0.0' # So are single quotes

# ❌ These will not work
require-dbt-version: >=1.0.0 # No quotes? No good
require-dbt-version: ">= 1.0.0" # Don't put whitespace after the equality signs

Examples

Specify a minimum dbt version

다음 예에서는 최소 경계를 지정하기 위해 >= 연산자를 사용하여, 이 프로젝트가 1.0.0 이상의 모든 dbt 버전과 함께 실행되도록 설정합니다:

dbt_project.yml
require-dbt-version: ">=1.0.0"

Pin to a range

다음 예에서는 상한 및 하한 경계를 설정하기 위해 쉼표로 구분된 목록을 사용하여, 이 프로젝트가 dbt 1.x.x 버전과 함께 실행되도록 설정합니다:

dbt_project.yml
require-dbt-version: [">=1.0.0", "<2.0.0"]

또는 

dbt_project.yml
require-dbt-version: ">=1.0.0,<2.0.0"

Require a specific dbt version

다음 예에서는 프로젝트가 dbt v1.5 버전에서만 실행되도록 설정하는 방법입니다:

dbt_project.yml
require-dbt-version: 1.5

Invalid dbt versions

프로젝트 또는 포함된 패키지에서 지정된 require-dbt-version과 호출된 dbt 버전이 일치하지 않으면, dbt는 즉시 다음과 같은 오류로 실패합니다:

$ dbt compile
Running with dbt=0.21.0
Encountered an error while reading the project:
Runtime Error
  This version of dbt is not supported with the 'my_project' package.
    Installed version of dbt: =0.21.0
    Required version of dbt for 'my_project': ['>=1.0.0', '<2.0.0']
  Check the requirements for the 'my_project' package, or run dbt again with --no-version-check

Disabling version checks

호환되지 않는 dbt 버전으로 인한 실패를 억제하려면 dbt run 실행 시 --no-version-check 플래그를 제공하십시오.

$ dbt run --no-version-check
Running with dbt=0.21.0
Found 13 models, 2 tests, 1 archives, 0 analyses, 204 macros, 2 operations....

자세한 사용법은 global configs을 참조하세요.

Recommendation

  • 이 구성은 권장됩니다.
  • v1 이전에는 필요한 dbt 버전을 마이너 릴리스에 고정해야 합니다. v1 이후에는 메이저 릴리스에 고정해야 합니다 (위 example 참조).

seed-paths

dbt_project.yml
seed-paths: [directorypath]

Definition

선택적으로 seed 파일이 위치한 사용자 정의 디렉터리 목록을 지정할 수 있습니다.

Default

기본적으로 dbt는 씨드 파일이 seeds 디렉터리에 위치한다고 예상합니다. 즉, seed-paths: ["seeds"]입니다.

Examples

Use a subdirectory named custom_seeds instead of seeds

dbt_project.yml
seed-paths: ["custom_seeds"]

Co-locate your models and seeds in the models director

Note: 이는 dbt가 씨드(.csv 파일)와 모델(.sql 파일)을 위한 서로 다른 파일 유형을 찾기 때문에 작동합니다.

dbt_project.yml
seed-paths: ["models"]
model-paths: ["models"]

Split your seeds across two directories

Note: 비슷한 효과를 얻기 위해 대신 seeds/ 디렉터리 내에 두 개의 하위 디렉터리를 사용하는 것을 권장합니다.

dbt_project.yml
seed-paths: ["seeds", "custom_seeds"]

snapshot-paths

dbt_project.yml
snapshot-paths: [directorypath]

Definition

snapshots이 위치하는 사용자 정의 디렉터리 목록을 선택적으로 지정할 수 있습니다. 모델과 스냅샷을 동일한 위치에 두는 것은 불가능합니다.

Default

기본적으로 dbt는 스냅샷을 snapshots 디렉토리에서 찾습니다. 즉, snapshot-paths: ["snapshots"]

Examples

Use a subdirectory named archives instead of snapshots

dbt_project.yml
snapshot-paths: ["archives"]

test-paths

dbt_project.yml
test-paths: [directorypath]

Definition

singular tests가 위치하는 사용자 정의 디렉터리 목록을 선택적으로 지정할 수 있습니다.

Default

이 설정을 지정하지 않으면 dbt는 기본적으로 tests 디렉터리에서 테스트를 찾습니다. 즉, test-paths: ["tests"]. 구체적으로는 다음과 같은 .sql 파일을 찾습니다:

  • tests/generic 하위 디렉터리에 있는 일반적인 테스트 정의
  • 다른 모든 파일에 있는 단일 테스트

Examples

Use a subdirectory named custom_tests instead of tests for data tests

dbt_project.yml
test-paths: ["custom_tests"]

version

dbt 프로젝트에는 두 가지 다른 유형의 version 태그가 있습니다. 이 필드의 의미는 위치에 따라 다릅니다.

dbt_project.yml versions

dbt_project 파일의 버전 태그는 dbt 프로젝트의 버전을 나타냅니다.

dbt 버전 1.5부터 dbt_project.yml 파일의 version 필드는 선택적 매개변수입니다. 사용할 경우 버전은 1.0.0과 같은 semantic version 형식이어야 합니다. 지정되지 않은 경우 기본값은 None입니다. dbt 버전 1.4 이하를 사용하는 사용자는 이 태그가 필요하지만 현재 dbt에서 의미 있는 방식으로 사용되지 않습니다.

Core 버전에 대해 자세히 알아보려면 About dbt Core versions을 참조하세요.

dbt_project.yml
version: version

.yml property file versions

.yml 속성 파일의 버전 태그는 제어 태그를 제공하여 dbt가 속성 파일을 처리하는 방식을 지정합니다.

1.5 버전부터 dbt는 리소스 .yml 파일에서 이 구성을 더 이상 필요로하지 않습니다. 이 태그가 이전에 필요했던 이유에 대해 더 알고 싶다면 FAQs를 참조하십시오. dbt 버전 1.4 이하를 사용하는 사용자는 이 태그가 필요합니다.

속성 파일에 대한 자세한 내용은 동일한 페이지의 일반 documentation를 참조하세요.

Resource property file with version specified

.yml
version: 2  # Only 2 is accepted by dbt versions up to 1.4.latest.

models: 
    ...

Resource property file without version specified

.yml
models: 
    ...

FAQS

Why do model and source yml files always start with version: 2?

옛날에는 이 .yml 파일의 구조가 매우 다르게 되어 있었습니다 (그때 dbt를 사용하던 사람들에게 인사드립니다!). version: 2를 추가함으로써 이 구조를 더 확장 가능하게 만들 수 있었습니다.

현재 리소스 yml 파일은 이 구성을 필요로하지 않습니다. 지정된 경우에만 version: 2를 지원합니다. .yml 파일을 곧 version: 3으로 업데이트 할 계획은 없지만, 이 구성이 우리에게 미래에 새로운 구조를 도입하기 쉽게 만들어 줍니다.
