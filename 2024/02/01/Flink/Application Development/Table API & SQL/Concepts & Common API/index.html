Table API & SQL - Concepts & Common API

Publish Date:   2024-02-01
Flink

Concepts & Common API

Table API 및 SQL은 통합된 공통 API로 제공됩니다. 이 API의 중심 개념은 쿼리의 입력 및 출력으로 작용하는 Table입니다. 이 문서는 Table API 및 SQL 쿼리를 사용한 프로그램의 공통 구조, Table 등록 방법, Table 쿼리 방법, Table 발행 방법을 보여줍니다.

Structure of Table API and SQL Programs

다음 코드 예제는 Table API 및 SQL 프로그램의 일반적인 구조를 보여줍니다.

모든 Flink Scala API가 폐기되고 향후 Flink 버전에서 제거될 예정입니다. 여전히 Scala로 애플리케이션을 작성할 수 있지만 DataStream 및/또는 Table API의 Java 버전으로 이동해야 합니다.

FLIP-265 Deprecate and remove Scala API support 참조

java
import org.apache.flink.table.api.*;
import org.apache.flink.connector.datagen.table.DataGenConnectorOptions;

// Create a TableEnvironment for batch or streaming execution.
// See the "Create a TableEnvironment" section for details.
TableEnvironment tableEnv = TableEnvironment.create(/*…*/);

// Create a source table
tableEnv.createTemporaryTable("SourceTable", TableDescriptor.forConnector("datagen")
    .schema(Schema.newBuilder()
      .column("f0", DataTypes.STRING())
      .build())
    .option(DataGenConnectorOptions.ROWS_PER_SECOND, 100L)
    .build());

// Create a sink table (using SQL DDL)
tableEnv.executeSql("CREATE TEMPORARY TABLE SinkTable WITH ('connector' = 'blackhole') LIKE SourceTable (EXCLUDING OPTIONS) ");

// Create a Table object from a Table API query
Table table1 = tableEnv.from("SourceTable");

// Create a Table object from a SQL query
Table table2 = tableEnv.sqlQuery("SELECT * FROM SourceTable");

// Emit a Table API result Table to a TableSink, same for SQL result
TableResult tableResult = table1.insertInto("SinkTable").execute();
python
from pyflink.table import *

# Create a TableEnvironment for batch or streaming execution
table_env = ... # see "Create a TableEnvironment" section

# Create a source table
table_env.executeSql("""CREATE TEMPORARY TABLE SourceTable (
  f0 STRING
) WITH (
  'connector' = 'datagen',
  'rows-per-second' = '100'
)
""")

# Create a sink table
table_env.executeSql("CREATE TEMPORARY TABLE SinkTable WITH ('connector' = 'blackhole') LIKE SourceTable (EXCLUDING OPTIONS) ")

# Create a Table from a Table API query
table1 = table_env.from_path("SourceTable").select(...)

# Create a Table from a SQL query
table2 = table_env.sql_query("SELECT ... FROM SourceTable ...")

# Emit a Table API result Table to a TableSink, same for SQL result
table_result = table1.execute_insert("SinkTable")

Table API 및 SQL 쿼리는 DataStream 프로그램과 쉽게 통합되어 내장될 수 있습니다. DataStream을 Table로 변환하거나 그 반대로 변환하는 방법을 알아보려면 DataStream API Integration 페이지를 참조하십시오.

Create a TableEnvironment

TableEnvironment는 Table API 및 SQL 통합의 진입점이며 다음을 담당합니다:

  • 내부 카탈로그에 Table 등록
  • 카탈로그 등록
  • 플러그 가능한 모듈 로드
  • SQL 쿼리 실행
  • 사용자 정의 (스칼라, 테이블 또는 집계) 함수 등록
  • DataStream과 Table 간 변환 (StreamTableEnvironment의 경우)

Table은 항상 특정 TableEnvironment에 바인딩됩니다. 서로 다른 TableEnvironment의 테이블을 동일한 쿼리에서 조인하거나 유니온하는 것은 불가능합니다. TableEnvironment는 정적인 TableEnvironment.create() 메서드를 호출하여 생성됩니다.

java
import org.apache.flink.table.api.EnvironmentSettings;
import org.apache.flink.table.api.TableEnvironment;

EnvironmentSettings settings = EnvironmentSettings
    .newInstance()
    .inStreamingMode()
    //.inBatchMode()
    .build();

TableEnvironment tEnv = TableEnvironment.create(settings);
python
from pyflink.table import EnvironmentSettings, TableEnvironment

# create a streaming TableEnvironment
env_settings = EnvironmentSettings.in_streaming_mode()
table_env = TableEnvironment.create(env_settings)

# create a batch TableEnvironment
env_settings = EnvironmentSettings.in_batch_mode()
table_env = TableEnvironment.create(env_settings)

대안으로 사용자는 기존의 StreamExecutionEnvironment에서 StreamTableEnvironment를 생성하여 DataStream API와 상호 운용할 수 있습니다.

java
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.table.api.EnvironmentSettings;
import org.apache.flink.table.api.bridge.java.StreamTableEnvironment;

StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
StreamTableEnvironment tEnv = StreamTableEnvironment.create(env);
python
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment

s_env = StreamExecutionEnvironment.get_execution_environment()
t_env = StreamTableEnvironment.create(s_env)

Create Tables in the Catalog

TableEnvironment는 테이블 맵을 유지하며 식별자와 함께 생성된 테이블의 카탈로그를 관리합니다. 각 식별자는 카탈로그 이름, 데이터베이스 이름 및 객체 이름으로 구성됩니다. 카탈로그 또는 데이터베이스가 지정되지 않은 경우 현재 기본값이 사용됩니다(Table identifier expanding 섹션의 예제 참조).

테이블은 가상(VIEWS) 또는 일반(TABLES)일 수 있습니다. VIEWS는 일반적으로 Table API 또는 SQL 쿼리의 결과인 기존 Table 개체에서 생성될 수 있습니다. TABLES는 파일, 데이터베이스 테이블 또는 메시지 큐와 같은 외부 데이터를 설명합니다.

Temporary vs Permanent tables.

테이블은 단일 Flink 세션의 수명 주기에 묶여 있는 일시적인 것이거나 여러 Flink 세션 및 클러스터에서 볼 수 있는 영구적인 것일 수 있습니다.

영구적인 테이블은 테이블에 대한 메타데이터를 유지하기 위해 catalog (예: 하이브 메타스토어)가 필요합니다. 영구적인 테이블이 생성되면 해당 카탈로그에 연결된 모든 Flink 세션에서 볼 수 있으며 테이블이 명시적으로 삭제될 때까지 계속 존재합니다.

반면에 일시적인 테이블은 항상 메모리에 저장되며 생성된 Flink 세션의 기간 동안만 존재합니다. 이러한 테이블은 다른 세션에서는 볼 수 없습니다. 이러한 테이블은 어떤 카탈로그나 데이터베이스에 바인드되지 않지만 해당 이름 공간 내에서 생성될 수 있습니다. 일시적인 테이블은 해당하는 데이터베이스가 제거되어도 삭제되지 않습니다.

Shadowing

기존의 영구 테이블과 동일한 식별자로 일시적인 테이블을 등록하는 것이 가능합니다. 일시적인 테이블은 영구 테이블을 가립니다(shadowing) 그리고 일시적인 테이블이 존재하는 한 영구 테이블에 접근할 수 없게 됩니다. 해당 식별자로 실행되는 모든 쿼리는 일시적인 테이블에 대해 실행됩니다.

이는 실험에 유용할 수 있습니다. 예를 들어, 데이터의 하위 집합을 가진 일시적인 테이블이나 데이터가 완전히 가려진 테이블과 같은 경우에 정확히 동일한 쿼리를 먼저 실행할 수 있습니다. 쿼리가 올바른지 확인한 후 실제 프로덕션 테이블에 대해 실행할 수 있습니다.

Create a Table

Virtual Tables

Table API 객체는 SQL 용어로는 VIEW(가상 테이블)에 해당합니다. 논리적 쿼리 계획을 캡슐화합니다. 다음과 같이 카탈로그에 생성할 수 있습니다:

java
// get a TableEnvironment
TableEnvironment tableEnv = ...; // see "Create a TableEnvironment" section

// table is the result of a simple projection query 
Table projTable = tableEnv.from("X").select(...);

// register the Table projTable as table "projectedTable"
tableEnv.createTemporaryView("projectedTable", projTable);
python
# get a TableEnvironment
table_env = ... # see "Create a TableEnvironment" section

# table is the result of a simple projection query 
proj_table = table_env.from_path("X").select(...)

# register the Table projTable as table "projectedTable"
table_env.register_table("projectedTable", proj_table)

Note: Table 객체는 관계형 데이터베이스 시스템의 VIEW와 유사합니다. 즉, Table을 정의하는 쿼리는 최적화되지 않지만 다른 쿼리가 등록된 Table을 참조할 때 인라인됩니다. 여러 쿼리가 동일한 등록된 Table을 참조하는 경우, 각 참조 쿼리마다 인라인되어 여러 번 실행됩니다. 즉, 등록된 Table의 결과가 공유되지 않습니다.

Connector Tables

또한 connector 선언으로부터 관계형 데이터베이스에서 알려진 TABLE을 생성하는 것도 가능합니다. 커넥터는 테이블의 데이터를 저장하는 외부 시스템을 설명합니다. 여기에는 Apache Kafka나 일반 파일 시스템과 같은 저장 시스템이 포함될 수 있습니다.

이러한 테이블은 Table API를 직접 사용하여 생성하거나 SQL DDL로 전환하여 생성할 수 있습니다.

java
// Using table descriptors
final TableDescriptor sourceDescriptor = TableDescriptor.forConnector("datagen")
    .schema(Schema.newBuilder()
    .column("f0", DataTypes.STRING())
    .build())
    .option(DataGenConnectorOptions.ROWS_PER_SECOND, 100L)
    .build();

tableEnv.createTable("SourceTableA", sourceDescriptor);
tableEnv.createTemporaryTable("SourceTableB", sourceDescriptor);

// Using SQL DDL
tableEnv.executeSql("CREATE [TEMPORARY] TABLE MyTable (...) WITH (...)");
python
# Using table descriptors
source_descriptor = TableDescriptor.for_connector("datagen") \
    .schema(Schema.new_builder()
            .column("f0", DataTypes.STRING())
            .build()) \
    .option("rows-per-second", "100") \
    .build()

t_env.create_table("SourceTableA", source_descriptor)
t_env.create_temporary_table("SourceTableB", source_descriptor)

# Using SQL DDL
t_env.execute_sql("CREATE [TEMPORARY] TABLE MyTable (...) WITH (...)")

Expanding Table identifiers

테이블은 항상 카탈로그, 데이터베이스 및 테이블 이름으로 구성된 3부분 식별자로 등록됩니다.

사용자는 “현재 카탈로그” 및 “현재 데이터베이스”로 설정할 수 있는 카탈로그 하나와 그 안의 데이터베이스 하나를 지정할 수 있습니다. 이들을 사용하면 위에서 언급한 3부분 식별자의 첫 두 부분이 선택적으로 지정될 수 있습니다. 즉, 제공되지 않으면 현재 카탈로그와 현재 데이터베이스가 참조됩니다. 사용자는 테이블 API나 SQL을 통해 현재 카탈로그와 현재 데이터베이스를 전환할 수 있습니다.

식별자는 SQL 요구 사항을 따르므로 백틱 문자(`)로 이스케이프할 수 있습니다.

java
TableEnvironment tEnv = ...;
tEnv.useCatalog("custom_catalog");
tEnv.useDatabase("custom_database");

Table table = ...;

// register the view named 'exampleView' in the catalog named 'custom_catalog'
// in the database named 'custom_database' 
tableEnv.createTemporaryView("exampleView", table);

// register the view named 'exampleView' in the catalog named 'custom_catalog'
// in the database named 'other_database' 
tableEnv.createTemporaryView("other_database.exampleView", table);

// register the view named 'example.View' in the catalog named 'custom_catalog'
// in the database named 'custom_database' 
tableEnv.createTemporaryView("`example.View`", table);

// register the view named 'exampleView' in the catalog named 'other_catalog'
// in the database named 'other_database' 
tableEnv.createTemporaryView("other_catalog.other_database.exampleView", table);
python
# get a TableEnvironment
t_env = TableEnvironment.create(...)
t_env.use_catalog("custom_catalog")
t_env.use_database("custom_database")

table = ...

# register the view named 'exampleView' in the catalog named 'custom_catalog'
# in the database named 'custom_database'
t_env.create_temporary_view("other_database.exampleView", table)

# register the view named 'example.View' in the catalog named 'custom_catalog'
# in the database named 'custom_database'
t_env.create_temporary_view("`example.View`", table)

# register the view named 'exampleView' in the catalog named 'other_catalog'
# in the database named 'other_database'
t_env.create_temporary_view("other_catalog.other_database.exampleView", table)

Query a Table

Table API

Table API는 Scala 및 Java용으로 제공되는 언어 통합형 쿼리 API입니다. SQL과 달리 쿼리는 문자열로 지정되지 않고 호스트 언어에서 단계별로 작성됩니다.

이 API는 테이블(스트리밍 또는 배치)을 나타내는 Table 클래스를 기반으로 하며 관계형 작업을 적용하는 메서드를 제공합니다. 이러한 메서드는 입력 Table에 관계형 작업을 적용한 결과를 나타내는 새로운 Table 객체를 반환합니다. 일부 관계형 작업은 table.groupBy(...).select()와 같이 여러 메서드 호출로 구성됩니다. 여기서 groupBy(...)는 table의 그룹화를 지정하고, select(...)는 table의 그룹에 대한 프로젝션을 지정합니다.

Table API 문서는 스트리밍 및 배치 테이블에서 지원되는 모든 Table API 작업을 설명합니다.

다음 예제는 간단한 Table API 집계 쿼리를 보여줍니다:

java
// get a TableEnvironment
TableEnvironment tableEnv = ...; // see "Create a TableEnvironment" section

// register Orders table

// scan registered Orders table
Table orders = tableEnv.from("Orders");
// compute revenue for all customers from France
Table revenue = orders
  .filter($("cCountry").isEqual("FRANCE"))
  .groupBy($("cID"), $("cName"))
  .select($("cID"), $("cName"), $("revenue").sum().as("revSum"));

// emit or convert Table
// execute query

# register Orders table

# scan registered Orders table
orders = table_env.from_path("Orders")
# compute revenue for all customers from France
revenue = orders \
    .filter(col('cCountry') == 'FRANCE') \
    .group_by(col('cID'), col('cName')) \
    .select(col('cID'), col('cName'), col('revenue').sum.alias('revSum'))

SQL

Flink의 SQL 통합은 SQL 표준을 구현한 Apache Calcite를 기반으로 합니다. SQL 쿼리는 일반 문자열로 지정됩니다.

SQL 문서는 Flink의 스트리밍 및 배치 테이블에 대한 SQL 지원을 설명합니다.

다음 예제는 쿼리를 지정하고 결과를 Table로 반환하는 방법을 보여줍니다.

java
// get a TableEnvironment
TableEnvironment tableEnv = ...; // see "Create a TableEnvironment" section

// register Orders table

// compute revenue for all customers from France
Table revenue = tableEnv.sqlQuery(
    "SELECT cID, cName, SUM(revenue) AS revSum " +
    "FROM Orders " +
    "WHERE cCountry = 'FRANCE' " +
    "GROUP BY cID, cName"
  );

// emit or convert Table
// execute query
python
# get a TableEnvironment
table_env = ... # see "Create a TableEnvironment" section

# register Orders table

# compute revenue for all customers from France
revenue = table_env.sql_query(
    "SELECT cID, cName, SUM(revenue) AS revSum "
    "FROM Orders "
    "WHERE cCountry = 'FRANCE' "
    "GROUP BY cID, cName"
)

# emit or convert Table
# execute query

Mixing Table API and SQL

  • A SQL query can be defined on the result of a Table API query by registering the resulting Table in the TableEnvironment and referencing it in the FROM clause of the SQL query.

Table API 및 SQL 쿼리는 모두 Table 객체를 반환하기 때문에 쉽게 혼합될 수 있습니다:

  • Table API 쿼리는 SQL 쿼리에서 반환된 Table 객체에서 정의될 수 있습니다.
  • SQL 쿼리는 Table API 쿼리의 결과에 정의될 수 있으며, 결과로 반환된 Table을 TableEnvironment에 등록하고 SQL 쿼리의 FROM 절에서 참조할 수 있습니다.

Emit a Table

테이블은 TableSink에 쓰여지는 것으로 출력됩니다. TableSink는 다양한 파일 형식 (예: CSV, Apache Parquet, Apache Avro), 저장 시스템 (예: JDBC, Apache HBase, Apache Cassandra, Elasticsearch) 또는 메시징 시스템 (예: Apache Kafka, RabbitMQ)을 지원하기 위한 일반적인 인터페이스입니다.

배치 Table은 BatchTableSink로만 쓰여질 수 있으며, 스트리밍 Table은 AppendStreamTableSink, RetractStreamTableSink 또는 UpsertStreamTableSink 중 하나가 필요합니다.

사용 가능한 Sink에 대한 자세한 내용과 사용자 정의 DynamicTableSink를 구현하는 방법에 대한 지침은 Table Sources & Sinks 문서를 참조하십시오.

Table.insertInto(String tableName) 메서드는 등록된 Sink 테이블로 소스 테이블을 출력하는 완전한 엔드 투 엔드 파이프라인을 정의합니다. 이 메서드는 테이블 이름에 따라 카탈로그에서 테이블 sink을 찾고, Table의 스키마가 sink의 스키마와 동일한지 확인합니다. 파이프라인은 TablePipeline.explain()으로 설명할 수 있으며, TablePipeline.execute()를 호출하여 실행할 수 있습니다.

다음 예제는 Table을 출력하는 방법을 보여줍니다:

java
// get a TableEnvironment
TableEnvironment tableEnv = ...; // see "Create a TableEnvironment" section

// create an output Table
final Schema schema = Schema.newBuilder()
    .column("a", DataTypes.INT())
    .column("b", DataTypes.STRING())
    .column("c", DataTypes.BIGINT())
    .build();

tableEnv.createTemporaryTable("CsvSinkTable", TableDescriptor.forConnector("filesystem")
    .schema(schema)
    .option("path", "/path/to/file")
    .format(FormatDescriptor.forFormat("csv")
        .option("field-delimiter", "|")
        .build())
    .build());

// compute a result Table using Table API operators and/or SQL queries
Table result = ...;

// Prepare the insert into pipeline
TablePipeline pipeline = result.insertInto("CsvSinkTable");

// Print explain details
pipeline.printExplain();

// emit the result Table to the registered TableSink
pipeline.execute();

Translate and Execute a Query

Table API 및 SQL 쿼리는 입력이 스트리밍인지 배치인지에 관계없이 DataStream 프로그램으로 변환됩니다. 쿼리는 내부적으로 논리적 쿼리 계획으로 표현되며 두 단계로 변환됩니다:

  1. 논리적 계획의 최적화
  2. DataStream 프로그램으로의 번역

Table API 또는 SQL 쿼리가 번역되는 경우는 다음과 같습니다:

  • TableEnvironment.executeSql() 메서드가 호출될 때. 이 메서드는 주어진 문을 실행하는 데 사용되며, 이 메서드가 호출되면 SQL 쿼리가 즉시 번역됩니다.

  • TablePipeline.execute() 메서드가 호출될 때. 이 메서드는 소스에서 싱크로의 파이프라인을 실행하는 데 사용되며, 이 메서드가 호출되면 Table API 프로그램이 즉시 번역됩니다.

  • Table.execute() 메서드가 호출될 때. 이 메서드는 테이블 내용을 로컬 클라이언트로 수집하는 데 사용되며, 이 메서드가 호출되면 Table API가 즉시 번역됩니다.

  • StatementSet.execute() 메서드가 호출될 때. TablePipeline (StatementSet.add()를 통해 싱크로 전달되는) 또는 INSERT 문 (StatementSet.addInsertSql()을 통해 지정됨)이 먼저 StatementSet에 버퍼링됩니다. StatementSet.execute()가 호출되면 이러한 쿼리가 변환됩니다. 모든 싱크는 하나의 DAG로 최적화됩니다.

  • Table이 DataStream으로 변환될 때 (Integration with DataStream 참조). 변환된 후에는 일반적인 DataStream 프로그램이 되며 StreamExecutionEnvironment.execute()가 호출될 때 실행됩니다.

Query Optimization

아파치 Flink는 복잡한 쿼리 최적화를 수행하기 위해 아파치 Calcite를 활용하고 확장합니다. 이는 규칙 및 비용 기반 최적화 시리즈를 포함합니다:

  • Subquery decorrelation based on Apache Calcite
  • Project pruning
  • Partition pruning
  • Filter push-down
  • Sub-plan deduplication to avoid duplicate computation
  • Special subquery rewriting, including two parts:
    • Converts IN and EXISTS into left semi-joins
    • Converts NOT IN and NOT EXISTS into left anti-join
  • Optional join reordering
    • Enabled via table.optimizer.join-reorder-enabled

Note: IN/EXISTS/NOT IN/NOT EXISTS are currently only supported in conjunctive conditions in subquery rewriting.

옵티마이저는 계획뿐만 아니라 데이터 소스에서 제공되는 풍부한 통계 및 각 연산자의 입출력, CPU, 네트워크 및 메모리와 같은 세밀한 비용에 기반하여 지능적인 결정을 내립니다.

고급 사용자는 TableEnvironment#getConfig#setPlannerConfig를 호출하여 테이블 환경에 제공할 CalciteConfig 객체를 통해 사용자 정의 최적화를 제공할 수 있습니다.

Explaining a Table

Table API는 Table을 계산하는 논리적 및 최적화된 쿼리 계획을 설명하는 메커니즘을 제공합니다. 이는 Table.explain() 메서드나 StatementSet.explain() 메서드를 통해 수행됩니다. Table.explain()는 테이블의 계획을 반환합니다. Table.StatementSet.explain()은 여러 싱크의 계획을 반환합니다. 이는 세 가지 계획을 설명하는 문자열을 반환합니다:

  • 관계형 쿼리의 Abstract Syntax Tree, 즉 최적화되지 않은 논리적 쿼리 계획,
  • 최적화된 논리적 쿼리 계획,
  • 물리적 실행 계획.

TableEnvironment.explainSql() 및 TableEnvironment.executeSql()은 EXPLAIN 문을 실행하여 계획을 가져오는 것을 지원합니다. EXPLAIN 페이지를 참조하십시오.

다음 코드는 Table.explain() 메서드를 사용하여 주어진 Table에 대한 예제 및 해당 출력을 보여줍니다:

java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
StreamTableEnvironment tEnv = StreamTableEnvironment.create(env);

DataStream<Tuple2<Integer, String>> stream1 = env.fromElements(new Tuple2<>(1, "hello"));
DataStream<Tuple2<Integer, String>> stream2 = env.fromElements(new Tuple2<>(1, "hello"));

// explain Table API
Table table1 = tEnv.fromDataStream(stream1, $("count"), $("word"));
Table table2 = tEnv.fromDataStream(stream2, $("count"), $("word"));
Table table = table1
  .where($("word").like("F%"))
  .unionAll(table2);

System.out.println(table.explain());
python
env = StreamExecutionEnvironment.get_execution_environment()
t_env = StreamTableEnvironment.create(env)

table1 = t_env.from_elements([(1, "hello")], ["count", "word"])
table2 = t_env.from_elements([(1, "hello")], ["count", "word"])
table = table1 \
    .where(col('word').like('F%')) \
    .union_all(table2)
print(table.explain())
