Operators

연산자는 하나 이상의 데이터 스트림을 새로운 데이터 스트림으로 변환합니다. 프로그램은 여러 변환을 결합하여 정교한 데이터 플로우 토폴로지를 만들 수 있습니다.

이 섹션은 기본 변환에 대한 설명, 해당 변환을 적용한 후의 실제 물리적 분할, 그리고 Flink의 연산자 연결에 대한 통찰력을 제공합니다.

DataStream Transformations

Map

DataStream → DataStream

하나의 요소를 가져와 하나의 요소를 생성합니다. 입력 스트림의 값들을 두 배로 하는 맵 함수입니다.

DataStream<Integer> dataStream = //...
dataStream.map(new MapFunction<Integer, Integer>() {
    @Override
    public Integer map(Integer value) throws Exception {
        return 2 * value;
    }
});
dataStream.map(MapFunction<Int, Int> { value -> 2 * value })

FlatMap

DataStream → DataStream

하나의 요소를 가져와서 제로, 하나 또는 여러 요소를 생성합니다. 문장을 단어로 나누는 flatMap 함수입니다.

dataStream.flatMap(new FlatMapFunction<String, String>() {
    @Override
    public void flatMap(String value, Collector<String> out)
        throws Exception {
        for(String word: value.split(" ")){
            out.collect(word);
        }
    }
});
dataStream.flatMap(FlatMapFunction<String, String> { value, out ->
    value.split(" ").forEach { word -> out.collect(word) }
})

Filter

DataStream → DataStream

각 요소에 대해 bool 함수를 평가하고 함수가 true를 반환하는 경우 해당 요소를 보존하는 필터입니다. 함수가 0을 반환하면 해당 요소를 필터링하는 필터입니다.

dataStream.filter(new FilterFunction<Integer>() {
    @Override
    public boolean filter(Integer value) throws Exception {
        return value != 0;
    }
});
dataStream.filter(FilterFunction<Int> { value -> value != 0 })

KeyBy

DataStream → KeyedStream

스트림을 상호 배타적인 파티션으로 논리적으로 분할합니다. 동일한 키를 가진 모든 레코드는 동일한 파티션에 할당됩니다. 내부적으로 keyBy()는 해시 파티셔닝으로 구현됩니다. 키를 지정하는 방법에는 여러 가지가 있습니다.

dataStream.keyBy(value -> value.getSomeKey());
dataStream.keyBy(value -> value.f0);
dataStream.keyBy { value -> value.someKey }
dataStream.keyBy { value -> value.f0 }

키로 사용될 수 없는 유형은 다음과 같습니다:

  1. POJO(Plain Old Java Object, 순수자바객체) 유형인데 hashCode() 메서드를 오버라이드하지 않고 Object.hashCode() 구현에 의존하는 경우.

  2. 어떤 유형이든지 배열인 경우.

Reduce

KeyedStream → DataStream

“rolling reduce”는 키가 지정된 데이터 스트림에서 사용되며, 현재 요소를 마지막으로 줄어든 값과 결합하여 새 값을 생성합니다.

부분 합계 스트림을 생성하는 reduce 함수는 다음과 같습니다:

keyedStream.reduce(new ReduceFunction<Integer>() {
    @Override
    public Integer reduce(Integer value1, Integer value2)
    throws Exception {
        return value1 + value2;
    }
});
keyedStream.reduce(ReduceFunction<Int> { value1, value2 ->
    value 1 + value2
})

Window

KeyedStream → WindowedStream

윈도우는 이미 파티셔닝된 KeyedStreams에 대해 정의될 수 있습니다. 윈도우는 각 키별로 데이터를 특정 기준에 따라 그룹화합니다(예: 지난 5초 동안 도착한 데이터). 윈도우에 대한 완전한 설명은 windows에서 확인하십시오.

dataStream
  .keyBy(value -> value.f0)
  .window(TumblingEventTimeWindows.of(Time.seconds(5))); 
dataStream
  .keyBy { value -> value.f0 }
  .window(TumblingEventTimeWindows.of(Time.seconds(5)))

WindowAll

DataStream → AllWindowedStream

윈도우는 일반적인 DataStream에 대해 정의될 수 있습니다. 윈도우는 모든 스트림 이벤트를 특정한 특성에 따라 그룹화합니다(예: 지난 5초 동안 도착한 데이터). 윈도우에 대한 완전한 설명은 윈도우를 참조하세요.

이 경우 많은 경우에 병렬 변환과 관련이 없습니다. windowAll 연산자의 경우 모든 레코드가 하나의 작업으로 모이게 됩니다.

dataStream
  .windowAll(TumblingEventTimeWindows.of(Time.seconds(5)));

Window Apply

WindowedStream → DataStream

AllWindowedStream → DataStream

윈도우 전체에 일반 함수를 적용합니다. 아래는 윈도우의 요소를 수동으로 합산하는 함수입니다.

windowAll 변환을 사용하는 경우 AllWindowFunction을 사용해야 합니다.

windowedStream.apply(new WindowFunction<Tuple2<String,Integer>, Integer, Tuple, Window>() {
    public void apply (Tuple tuple,
            Window window,
            Iterable<Tuple2<String, Integer>> values,
            Collector<Integer> out) throws Exception {
        int sum = 0;
        for (value t: values) {
            sum += t.f1;
        }
        out.collect (new Integer(sum));
    }
});

// applying an AllWindowFunction on non-keyed window stream
allWindowedStream.apply (new AllWindowFunction<Tuple2<String,Integer>, Integer, Window>() {
    public void apply (Window window,
            Iterable<Tuple2<String, Integer>> values,
            Collector<Integer> out) throws Exception {
        int sum = 0;
        for (value t: values) {
            sum += t.f1;
        }
        out.collect (new Integer(sum));
    }
});

WindowReduce

WindowedStream → DataStream

윈도우에 기능적인 reduce 함수를 적용하고 줄인 값을 반환합니다.

windowedStream.reduce (new ReduceFunction<Tuple2<String,Integer>>() {
    public Tuple2<String, Integer> reduce(Tuple2<String, Integer> value1, Tuple2<String, Integer> value2) throws Exception {
        return new Tuple2<String,Integer>(value1.f0, value1.f1 + value2.f1);
    }
});

Union

DataStream* → DataStream

두 개 이상의 데이터 스트림을 합쳐서 각 스트림의 모든 요소를 포함하는 새로운 스트림을 생성합니다. 참고: 동일한 데이터 스트림을 자신과 합칠 경우 결과적인 스트림에서 각 요소가 두 번 나타납니다.

dataStream.union(otherStream1, otherStream2, ...);

Window Join

DataStream,DataStream → DataStream

주어진 키와 공통 윈도우에 따라 두 개의 데이터 스트림을 결합합니다.

dataStream.join(otherStream)
    .where(<key selector>).equalTo(<key selector>)
    .window(TumblingEventTimeWindows.of(Time.seconds(3)))
    .apply (new JoinFunction () {...});

Interval Join

KeyedStream,KeyedStream → DataStream

두 개의 키가 지정된 스트림에서 e1과 e2 요소를 주어진 시간 간격 동안 공통 키에 따라 결합하여 e1.timestamp + lowerBound <= e2.timestamp <= e1.timestamp + upperBound를 충족합니다.

// this will join the two streams so that
// key1 == key2 && leftTs - 2 < rightTs < leftTs + 2
keyedStream.intervalJoin(otherKeyedStream)
    .between(Time.milliseconds(-2), Time.milliseconds(2)) // lower and upper bound
    .upperBoundExclusive(true) // optional
    .lowerBoundExclusive(true) // optional
    .process(new IntervalJoinFunction() {...});

Window CoGroup

DataStream,DataStream → DataStream

주어진 키와 공통 윈도우에 따라 두 개의 데이터 스트림을 코그룹화합니다.

dataStream.coGroup(otherStream)
    .where(0).equalTo(1)
    .window(TumblingEventTimeWindows.of(Time.seconds(3)))
    .apply (new CoGroupFunction () {...});

Connect

DataStream,DataStream → ConnectedStream

두 개의 데이터 스트림을 “연결”하여 그들의 타입을 유지합니다. 이 연결은 두 스트림 간에 공유 상태를 허용합니다.

DataStream<Integer> someStream = //...
DataStream<String> otherStream = //...

ConnectedStreams<Integer, String> connectedStreams = someStream.connect(otherStream);

CoMap, CoFlatMap

ConnectedStream → DataStream

연결된 데이터 스트림에서 map 및 flatMap과 유사한 동작을 수행합니다.

connectedStreams.map(new CoMapFunction<Integer, String, Boolean>() {
    @Override
    public Boolean map1(Integer value) {
        return true;
    }

    @Override
    public Boolean map2(String value) {
        return false;
    }
});
connectedStreams.flatMap(new CoFlatMapFunction<Integer, String, String>() {

   @Override
   public void flatMap1(Integer value, Collector<String> out) {
       out.collect(value.toString());
   }

   @Override
   public void flatMap2(String value, Collector<String> out) {
       for (String word: value.split(" ")) {
         out.collect(word);
       }
   }
});

Iterate

DataStream → IterativeStream → ConnectedStream

흐름에서 “피드백” 루프를 생성하여 하나의 연산자의 출력을 이전 연산자로 리디렉션합니다. 이는 모델을 계속 업데이트하는 알고리즘을 정의하는 데 특히 유용합니다. 다음 코드는 스트림을 시작하고 반복적으로 반복 본문을 적용합니다. 0보다 큰 요소는 피드백 채널로 다시 보내고 나머지 요소는 하류로 전달됩니다.

IterativeStream<Long> iteration = initialStream.iterate();
DataStream<Long> iterationBody = iteration.map (/*do something*/);
DataStream<Long> feedback = iterationBody.filter(new FilterFunction<Long>(){
    @Override
    public boolean filter(Long value) throws Exception {
        return value > 0;
    }
});
iteration.closeWith(feedback);
DataStream<Long> output = iterationBody.filter(new FilterFunction<Long>(){
    @Override
    public boolean filter(Long value) throws Exception {
        return value <= 0;
    }
});

Cache

DataStream → CachedDataStream

변환 중간 결과를 캐시합니다. 현재 batch execution mode로 실행되는 작업만 지원됩니다. 캐시된 중간 결과는 처음으로 중간 결과가 계산될 때 게으르게 생성되어 나중에 사용될 수 있습니다. 캐시가 손실되면 원래의 변환을 사용하여 다시 계산됩니다.

DataStream<Integer> dataStream = //...
CachedDataStream<Integer> cachedDataStream = dataStream.cache();
cachedDataStream.print(); // Do anything with the cachedDataStream
...
env.execute(); // Execute and create cache.
        
cachedDataStream.print(); // Consume cached result.
env.execute();

Physical Partitioning

Flink은 원하는 경우 변환 후 정확한 스트림 파티셔닝에 대한 저수준 제어도 제공합니다. 이를 위해 다음 함수를 사용할 수 있습니다.

Custom Partitioning

DataStream → DataStream

사용자 정의 Partitioner를 사용하여 각 요소에 대한 대상 태스크를 선택합니다.

dataStream.partitionCustom(partitioner, "someKey");
dataStream.partitionCustom(partitioner, 0);

Random Partitioning

DataStream → DataStream

요소를 균일한 분포에 따라 무작위로 파티션합니다.

dataStream.shuffle();

Rescaling

DataStream → DataStream

요소를 라운드 로빈 방식으로 다운스트림 작업의 하위 집합으로 파티션합니다. 이는 예를 들어 소스의 각 병렬 인스턴스에서 여러 매퍼의 하위 집합으로 분산 로드를 분배하지만 rebalance()가 발생하는 전체 리밸런스를 원하지 않을 때 유용합니다. 이렇게 하면 다른 구성 값에 따라 네트워크 상에서 데이터를 전송하는 대신 로컬 데이터 전송만 필요할 수 있습니다.(TaskManagers의 슬롯 수 등).

상위 작업이 보내는 요소의 하위 작업 하위 집합은 상위 및 하위 작업의 병렬도에 따라 달라집니다. 예를 들어, 상위 작업이 병렬로 2이고 하위 작업이 병렬로 6이면 하나의 상위 작업이 세 개의 하위 작업에 요소를 분배하고 나머지 상위 작업이 다른 세 개의 하위 작업에 요소를 분배할 것입니다. 반면에, 하위 작업이 병렬로 2인 경우 상위 작업이 병렬로 6인 경우 세 개의 상위 작업이 하나의 하위 작업에 요소를 분배하고 나머지 세 개의 상위 작업이 다른 하위 작업에 요소를 분배할 것입니다.

다른 병렬도가 서로의 배수가 아닌 경우 하나 이상의 하위 작업에서 상위 작업의 입력 수가 다를 수 있습니다.

위의 예에서 연결 패턴의 시각화를 보려면 다음 도표를 참조하십시오

dataStream.rescale();

Broadcasting

DataStream → DataStream

요소를 모든 파티션으로 브로드캐스트합니다.

dataStream.broadcast();

Task Chaining and Resource Groups

연속적인 두 개의 변환을 연결하는 것은 더 나은 성능을 위해 동일한 스레드 내에서 이들을 동일한 위치로 두는 것을 의미합니다. Flink는 가능한 경우 (예: 두 개의 연속적인 맵 변환) 연산자를 기본적으로 연결합니다. API는 원하는 경우 연결에 대한 세밀한 제어를 제공합니다.

전체 작업에서 연결을 비활성화하려면 StreamExecutionEnvironment.disableOperatorChaining()를 사용하십시오. 더 세밀한 제어를 위해 다음과 같은 함수를 사용할 수 있습니다. 이러한 함수는 이전 변환을 참조하므로 DataStream 변환 바로 다음에만 사용할 수 있습니다. 예를 들어, someStream.map(…).startNewChain()을 사용할 수 있지만 someStream.startNewChain()을 사용할 수는 없습니다.

리소스 그룹은 Flink의 슬롯입니다. 원하는 경우 연산자를 별도의 슬롯에 수동으로 격리할 수 있습니다.

Start New Chain

이 연산자로 새로운 체인을 시작합니다. 두 개의 매퍼가 연결되고, 필터는 첫 번째 매퍼에 연결되지 않습니다.

someStream.filter(...).map(...).startNewChain().map(...);

Disable Chaining

맵 연산자를 연결하지 않습니다.

someStream.map(...).disableChaining();

Set Slot Sharing Group

작업의 슬롯 공유 그룹을 설정합니다. Flink는 동일한 슬롯 공유 그룹을 가진 작업을 동일한 슬롯에 배치하며 슬롯 공유 그룹이없는 작업을 다른 슬롯에 유지합니다. 이를 사용하여 슬롯을 격리하는 데 사용할 수 있습니다. 모든 입력 작업이 동일한 슬롯 공유 그룹에 있는 경우 슬롯 공유 그룹이 상속됩니다. 기본 슬롯 공유 그룹의 이름은 “default”이며, 작업은 명시적으로 slotSharingGroup(“default”)를 호출하여 이 그룹에 추가할 수 있습니다.

someStream.filter(...).slotSharingGroup("name");

Name And Description

Flink의 연산자와 작업 정점은 이름과 설명을 가지고 있습니다. 이름과 설명은 연산자나 작업 정점이 하는 일에 대한 소개이지만, 서로 다르게 사용됩니다.

연산자와 작업 정점의 이름은 웹 UI, 스레드 이름, 로깅, 메트릭 등에서 사용됩니다. 작업 정점의 이름은 그 안에 있는 연산자의 이름을 기반으로 구성됩니다. 이름은 가능한 한 간결해야 하며 외부 시스템에 부하를 최소화하기 위해 노력해야 합니다.

설명은 실행 계획에서 사용되며 웹 UI에서 작업 정점의 세부 정보로 표시됩니다. 작업 정점의 설명은 그 안에 있는 연산자의 설명을 기반으로 구성됩니다. 설명에는 런타임 디버깅을 용이하게 하기 위해 연산자에 대한 상세 정보가 포함될 수 있습니다.

someStream.filter(...).name("filter").setDescription("x in (1, 2, 3, 4) and y > 1");

기본적으로 작업 정점의 설명 형식은 트리 형식의 문자열입니다. 이전 버전과 같이 설명을 캐스케이딩 형식으로 설정하려면 사용자는 pipeline.vertex-description-mode를 CASCADING으로 설정할 수 있습니다.

Flink SQL에서 생성된 연산자는 기본적으로 연산자 유형과 ID로 구성된 이름과 자세한 설명을 가집니다. 이전 버전과 같이 이름을 자세한 설명으로 설정하려면 사용자는 table.exec.simplify-operator-name-enabled를 false로 설정할 수 있습니다.

파이프라인의 토폴로지가 복잡한 경우 사용자는 pipeline.vertex-name-include-index-prefix를 true로 설정하여 정점의 이름에 토폴로지 인덱스를 추가할 수 있습니다. 이렇게 하면 로그나 메트릭 태그를 통해 그래프에서 정점을 쉽게 찾을 수 있습니다.
