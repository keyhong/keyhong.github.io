[Toy] Dimension Modeling + Hadoop + Hive + Kudu + Trino + DBT

Publish Date:   2024-02-21
Toy

Next Step

다시 취준생이 되고 항상 탈락이라는 경험은 아픈 것 같습니다. 하지만 엔지니어로써 더 나아갈 내일을 봐야하고, 더 좋은 날이 올 것이라 믿기에 마지막으로 글을 정리하고 툴툴 털어내려고 합니다. 제가 면접에서 보여드리지는 못했지만 하둡 구축에 어려움을 겪으시는 분들이 활용할 수 있도록 만든 play ground와 생각했던 과정들을 정리하려고 합니다.

play ground는 Hadoop + Hive + Kudu + Trino가 연결된 docker-compose.yaml 입니다. Hadoop을 올릴 떄 처음 메타스토어를 초기화하는 작업을 하는 과정 모두 Dockerfile에 만들어 넣었습니다. 사용하신다면, 컨테이너가 올라간 이후 DBeaver 같은 툴로 Trino(localhost:8080)에 연결해서 바로 kudu와 hive를 사용할 수 있습니다 😊

차원 모델링

이번에 ERD를 직접 그려 보았습니다. 도구는 draw.io 입니다. 바탕색과 테이블 색을 설정할 수 있고, 엔티티 간의 관계를 쉽게 그릴 수 있는 도구입니다. 저는 Fact와 Dimension으로 차원 모델링을 했고, 이력 테이블에 대해서는 SCD 4 (SCD2 + Historical Table)로 풀어보았습니다.

** 테이블 내의 상세 컬럼은 가려두었습니다.

목표했던 아키텍처

전체 프로세스를 도커로 올리려는 시도를 하였습니다. 제 단일 노트북에서 구동되는 환경을 만들기 위해 replica를 조정하고 yarn, mapreduce 프로세스는 띄우지 않았습니다. hive, kudu, trion, dbt를 위한 playground를 최대한 가볍게 만들어 보았습니다. 짧은 시간으로 CI/CD 까지는 하지 못했지만 Hadoop, Hive, Kudu, Trion, DBT는 사용가능합니다.

프로젝트 구조

폴더 구조는 .env(secret file), conf, docker, mnt,init-sql로 나누었습니다. 이 최상단 폴더에 .gitignore, docker-compose.yml, Makefile, README.md는 같은 레벨에 있습니다.

프로세스 정리

프로그램 버전 프로세스 포트 실행 환경
MySQL 8.0.36 namenode 3306 container
Apache Hadoop 3.3.6 namenode 9870 container
datanode 9864 container
Apache Hive 3.1.3 metastore 10000 container
hiveserver2 10002 container
Apache Kudu 1.17 master 8051 container
tserver 8050 container
Trino 438 coordinator(worker) 8080 container
dbt-trino 1.7.1 local

Comment

  • Hadoop과 Kudu의 replica를 1으로 설정

  • ELT 프로세스를 하나만 사용하기 때문에 yarn은 올리지 않았습니다.

  • 쿼리엔진으로 trino를 사용하기 때문에 mapreduce는 올리지 않았습니다.

  • hiveserver2는 올려도 되고, 안올려도 상관없습니다. 저는 초기에 10000번 포트로 접속해서 작업을 했습니다.

  • trino는 JDK 버전호환 (21이상)을 위해 eclipse-temurin 이미지를 사용하였습니다.

  • dbt-trino는 로컬에서 trino(8080)으로 연결하여 사용하였습니다.

Makefile로 시작하기

make가 설치되어있다면, 순서대로 이미지 먼저 빌드합니다. 아파치 프로그램을 다운받는 CDN 사이트를 변경했지만, 그래도 시간이 조금은 걸리는 편입니다.

# base 이미지 빌드
$ make build

# base 이미지 위의 추가 이미지 빌드 & docker-compose 실행
$ make up

# docker container 내리기
$ make down

# image와 mnt 디렉터리 안 저장한 데이터 지우기
$ make delete-all

Kudu를 도입한 이유

빅데이터 환경에서 데이터 모델링을 그려보다 보니, Dimension 테이블에서 변경이 발생한다면 어떻게 해결할까라는 고민이 들었습니다. Data Lake에서는 Schema-on-read이기 때문에 언제든지 특정 컬림이 추가되어 들어올 수가 있습니다. 이럴 때 Hive만으로 테이블에 새로운 컬럼을 추가하려면 데이터를 지우고 테이블을 다시 만든 후 insert를 해야하기 때문에 테이블의 확장성에 있어 어려움이 많았습니다. 이에 Dimension 테이블을 Kudu에 저장해 향후에도 확장성을 높이려는 시도를 하였습니다.

Kudu는 Columnar storage로, ACID가 가능한 분산 스토리지 입니다. Replica를 설정하여 데이터의 durability를 보장할 수 있습니다. 또한 Hadoop 환경이 아니어도 구축가능합니다.

kudu에서 master는 카탈로그를 저장하는 역할을 하고, tablet 서버는 실제 데이터를 저장합니다. Hadoop의 master-slave와 비슷한 구조를 갖고 있습니다. 테이블을 trino로 생성하는 쿼리를 보시면 여러가지 특징을 알 수 있습니다.

-- 1-2. 고객 테이블 생성
CREATE TABLE IF NOT EXISTS KUDU.DW.DIM_CUSTOMER_INFO (
    CUSTOMER_ID                       INT         WITH (primary_key=true)
    -- 컬럼 일부 생략
  , EFFECTIVE_DATE                    DATE        
  , END_DATE                          DATE        
  , DW_LOAD_DT                        TIMESTAMP   
)
WITH (
  partition_by_hash_columns = ARRAY['CUSTOMER_ID'],
  partition_by_hash_buckets = 5
);

  • PK를 지정 할 수 있습니다.

  • PK로 사용되는 컬럼 중 일부를 반드시 partition_by_hash_columns으로 설정해야 한다.

  • partition_by_hash_buckets은 최소 2 이상이어야 합니다.

  • Secondary 인덱스나 foreign keys는 지원이 되지 않습니다.

  • Other usage limitations

단, Kudu는 저장소 역할만 하는 플랫폼이기에 쿼리를 처리를 할 수 있는 엔진이 필요한 데 흔히 impala를 함께 결합하지만, 저는 hive와 kudu를 하나의 세션에서 같이 사용하는 게 편리하다고 생각되어 trino를 사용하였습니다.

ELT Tool 선택 : dbt-trino

impala 대신 trino를 사용한 이유

Transform 도구로 DBT Core를 사용하려고 시도했습니다.

여러 adapter 중 하나를 선택할 수 있었는 데, dbt가 아직은 성장하는 플랫폼이라 trino에서 제공하는 검증된 어댑터(Verified adapters)를 사용하는 것이 좋다고 생각했습니다. 그래서 impala가 아닌 spark와 trino 중 고민을 했습니다.

dbt-trino vs. dbt-spark

spark는 사용자도 많고 충분히 성장한 플랫폼이지만, 제가 생각하는 spark의 장점은 스파크에서 제공하는 함수로 복잡한 프로세싱을 IDE와 결합하여 처리하는 프로그램밍 방식인 데, sql만으로는 그 장점이 줄어든다고 생각했습니다. 또한, learning curve가 다소 있는 플랫폼이라는 점도 고려했습니다.

trino는 hive의 한계인 이종 시스템에 대한 연합 쿼리를 가능하게 하는 분산처리 엔진입니다. 확장성이 높고, ANSI SQL 문법을 지향합니다. 그래서 빠른 학습이 가능하다는 장점이 있다고 생각했습니다.

dbt와 trino의 시너지

dbt는 파이썬으로 구성된 오픈소스 transform 툴입니다. 쿼리를 CTE(Common Table Expression) 문법으로 구상하기 때문에 서브쿼리가 덜하다는 장점이 있습니다. 하지만 dbt를 통해 짠 쿼리가 파이썬의 모델이라는 객체를 통해 변환되고, 변환된 쿼리는 시각적으로 확인하기 어려운 단점이 있습니다.

이에 대해 trino는 웹에서 실행된 쿼리 구문을 볼 수 있습니다. dbt로 etl를 실행했을 때, 실패한다면 파싱이 제대로 안되었는 지 혹은 구문이 잘못되었는 지 웹을 통해 쿼리를 직접 확인할 수 있는 장점이 있습니다. 저는 이를 통해 쿼리에 대한 트러블 슈팅을 보다 편하게 할 수 있었습니다.

dbt가 제공하는 무결성 테스트

dbt를 써보며 느꼈던 큰 장점은 파이썬이라는 프로그래밍이 접목되어 있어, 유틸리티를 편하게 쓸 수 있었다는 점이었습니다. 보시다시피 dbt에서는 무결성을 위한 테스트를 할 수 있도록 지원하는 데, 따로 쿼리를 짤 필요 없이 yaml 파일을 통해 쉽게 테스트를 할 수 있었습니다.

  • generic-data-tests

  • singular-data-tests

물론, 테스트의 결과를 알려줄 뿐 실패시 에러가 아닌 에러구문의 정상 표준 출력하기 때문에 json으로 로그를 남기고 파싱하거나 하는 방법을 추가로 구현해야 할 것 같다는 생각은 들었습니다. 이는 dbt를 조금 더 숙지하면 다른 방법으로 풀어낼 수 있을 것 같습니다.
