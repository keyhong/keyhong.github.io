품질진단 항목 (데이터 값 진단)

데이터 거버넌스

1. 여부

분류 데이터(OO여부, OO유무 등)가 2개의 유효 값을 갖는지 컬럼 집단

1.1 진단 방법

  • 컬럼을 구성하는 2개의 유효 값 이외의 데이터를 오류로 판단

  • 컬럼 분석 DISTINCT 분석을 통해서도 오류 예측 가능

테이블명 컬럼명 총건수 DISTINCT DUPLICATE
고객 통합고객여부 10 3 7
  • 유효 값을 파악할 수 없는 경우, 전체 데이터에 대한 값 분포를 파악하여 오류 데이터를 예측

1.2 오류 예시

테이블명 컬럼명 컬럼값 건수
고객 통합고객여부 Y 33
N 8
1 1
0 1
O 1
X 1
(NULL) 1

1.3 여부 검증 SQL

여부 검증 SQL
SELECT comb_res_yn
     , COUNT(*)
 FROM customer
WHERE comb_res_yn NOT INT ('Y', 'N')
GROUP BY comb_res_yn
ORDER BY COUNT(*) DESC
;

1.4 오류 원인

  • 다수의 응용 프로그램에서 데이터가 입력되는 경우

  • Default 값 설정이 되어 있지 않은 경우

1.5 개선방안 및 고려사항

  • 여부 값에 대한 표준화

  • 기존 값에 대한 정제

  • Default 값 설정

2. 날짜

날짜/시간 유형 데이터에 대한 유효성 진단

2.1 진단방법

  • 유효한 형식과 날짜로 구성되지 않은 값을 오류로 판단

  • 보통 데이터 타입이 DATE형이 아닌 문자로 구성된 날짜 컬럼을 진단

컬럼명 데이터타입 길이 도메인
생산년도 VARCHAR2 4 -> 년도 (YYYY)
고객등록일자 VARCHAR2 8 -> 년월일 (YYYYMMDD)
주문일시 VARCHAR2 12 -> 년월일시분 (YYYYMMDDHH24MI)
  • 단, 타입이 DATE형이지만 날짜의 유효범위가 존재하는 경우 날짜에 범위조건을 입력하여 진단수행 가능
    (예, 미래일자는 존재 불가)

2.1 오류 예시

테이블명 컬럼명 컬럼값 건수
주문상세 배송일자
(YYYYMMDD)		 20150232 1
201506 1
2015 8
11111112 1
2015-11-13 1

2.3 오류 원인

  • 입력자의 데이터 수기 입력 오류

  • 날짜 입력 시 날짜 형식에 대한 검증 부재

2.4 개선방안 및 고려사항

  • 날짜 입력 시 날짜를 선택하여 입력하도록 응용 프로그램 개선

  • 날짜 형식에 대한 표준화 수행

3. 코드

개별코드/통합코드 마스터와 트랜잭션 코드와의 유효성 진단

3.1 진단방법

  • 단일코드, 목록형코드 등으로 정의된 컬럼이 정의된 컬럼 값으로 구성되어 있는지 분석

  • 공통코드 테이블에 의한 참조분석과 개별코드 참조분석으로 분류하여 수행

코드유형 설명
단일코드 하나의 key로 구성된 코드로서 열거형, 범위형 코드 값을 갖음
일반적으로 단일코드는 공통코드로 관리
복합코드 하나 이상의 key로 구성된 코드
목록형코드 코드, 코드명 이외의 속성을 갖거나 외부로부터 추출하여 변경이 낮은 코드
목록코드는 별도 Entity에서 관리하므로 반드시 개별코드 Entity를 참조

3.2 오류 예시

테이블명 컬럼명 컬럼값 건수
고객 고객상태코드
(CUST_STCD)

- ST01: 휴면
- ST02: 활동중
- ST03: 탈퇴		 ST3 1
법인 2

3.3 코드 검증 SQL

코드 마스터 테이블과의 참조분석 검증 SQL
SELECT A.cust_stcd
  FROM customer AS A -- 진단 대상 테이블
 WHERE A.cust_stcd IS NOT NULL
   AND NOT EXISTS (
      SELECT *
        FROM code_dtl AS X -- 공통코드 테이블
       WHERE X.code_id = 'cust_stcd' -- 코드명
         AND X.code_val = A.cust_stcd -- 코드값명
   )
;

3.4 오류 원인

  • 코드의 신규, 변경 건에 대한 현행화 관리 미흡

  • 응용 프로그램에서 코드 값을 하드 코딩하여 사용

3.5 개선방안 및 고려사항

  • 코드 신규, 변경 시 공통코드에 현행화 관리 및 배포 통제

  • 프로그램 상에서 코드 값 매핑시 하드코딩 금지

4. 번호(형식)

문자, 숫자 등으로 구성된 패턴을 갖는 값의 패턴 일관성 진단

4.1 진단 방법

  • 데이터의 생성규칙(패턴)에 어긋나는 값을 오류로 판단

  • 일반적으로 내부 표준에 따른 번호·문자체계의 기준에 따라 단순 패턴 검증을 수행

  • 주민등록번호, 사업자등록번호와 같은 오류 검증번호가 존재하는 데이터는 검증 계산식에 의해 검증을 수행

컬럼명 검증 규칙
고객번호 고객번호는 5자리의 숫자로 구성된다
주문번호 주문번호는 5자리의 숫자와 5자리의 영문으로 구성된다
전화번호 전화번호는 하이픈 구분자를 갖는 숫자로 구성된다
주민등록번호 계산에 의한 값이 마지막 13번째 번호(검증번호)와 일치해야 한다
사업자등록번호 계산에 의한 값이 마지막 10번째 번호(검증번호)와 일치해야 한다
··· ···

4.2 오류 예시

테이블명 컬럼명 패턴 건수
공급업체 전화번호
(하이픈구분자)		 99-9999-**** 1
99B999B9999 1
- 8
11111112 1
2015-11-13 1

4.3 오류 원인

  • 데이터 생성규칙 미정의

  • 데이터 입력 시점에 생성규칙에 의한 패턴 검증기능 부재

  • 같은 속성에 다른 구분자 사용

4.4 개선방안 및 고려사항

  • 데이터 입력 시 패턴 검증 응용 프로그램 개선

  • 데이터 값의 구분자 통일 등 데이터 생성규칙 정의

5. 금액/수량/율

범위가 존재하는 데이터의 유효 범위를 벗어난 값을 오류로 진단

5.1 진단방법

  • 숫자로 저장된 ‘금액’, ‘수량’, ‘율’ 도메인 컬럼의 모든 유효 범위를 벗어나는 오류를 측정

  • ‘건수’, ‘횟수’, ‘사업비’, ‘가격’, ‘비율’, ‘산정율’ 등과 같이 숫자로 저장되는 데이터 값의 단위가 혼재되어 있거나, 유효 값의 범위를 벗어나는 데이터를 오류로 측정

  • 컬럼분석 MIN 값과 MAX 값을 통하여 수칱 값의 범위를 파악

⚠️ 일반적으로 금액, 수량과 관련된 컬럼은 음수(-)가 나올 수 없음

테이블명 컬럼명 MIN MAX
주문상세 판매가격 -99999 9999999999

5.2 오류 예시

테이블명 컬럼명 패턴 건수
주문상세 판매가격 -99999 1
99B999B9999 33
9999 8
-1111 3
0~10000 45
99999999999 45

5.3 금액/수량 검증 SQL

금액/수량 검증 SQL
SELECT sal_price
     , COUNT(*) AS err_cnt
  FROM order_detail
 WHERE 1=1
   AND (sal_price < 0 OR sal_price > 100000) --범위 지정
 GROUP BY sal_price
 ORDER BY sal_price
;

5.4 오류 원인

  • 입력자의 데이터수기 입력오류

  • 데이터 입력 시 수치 값의 허용범위 검증기능 부재

  • 수량, 금액의 단위가 혼재하는 경우

5.5 개선방안 및 고려사항

  • 데이터 입력 시 이상치 범위 값에 대한 응용 프로그램 개선

  • 수량, 금액 등 단위 존재하는 값에 대한 표준화 수행

6. 참조값

부모-자식 관계 데이터의 참조 무결성 진단

6.1 진단방법

  • 부모-자식 간의 관계 정합성을 검증하여 부모가 없는 자식 데이터가 존재하는지 진단

  • 데이터를 관리하는 구조의 설계, 구축의 문제로 인해 데이터 값이 일관되게 관리되지 못하거나 정합성을 확보하지 못하는 경우 발생

  • 데이터 모델(ERD)상 식별관계로 정의하였으나, 데이터베이스의 물리제약(FK)으로 설정하지 않은 경우 발생함

6.2 오류 예시

주문 테이블(자식)의 고객번호(FK) 속성 값에서 고객 테이블(부모)에 없는 고객번호가 발견된 경우

6.3 참조값 검증 SQL

참조값 검증 SQL
SELECT X.*
  FROM (
    SELECT B.cust_id AS 기준정보
        ,  A.cust_id AS 참조정보
      FROM order_m AS A
        LEFT JOIN customer AS B
               ON a.cust_id = b.cust_id
  ) AS X
 WHERE x.기준정보 IS NULL
 ;

6.4 오류 원인

  • 부모 데이터 삭제 시, 자식 데이터가 그대로 남아 있는 경우

  • 프로그램 개발, 성능 확보 등을 위해 물리적인 제약(FK)을 설정하지 않은 경우

  • DBMS 제약을 설정하지 않고 응용 프로그램에서 참조무결성을 확보하려 하였으나, 오류가 발생한 경우

6.5 개선방안 및 고려사항

  • 물리적인 제약(FK)을 설정하거나, 응용 프로그램 개선 등을 통해 데이터 간 참조무결성이 보장되도록 개선

  • 데이터 모델(ERD) 현행화 및 변경관리 수행

7. 시간순서 일관성/선후관계

날짜 컬럼에 시간 선/후 관계가 존재하는 경우를 검증

7.1 진단방법

  • 날짜 컬럼 간의 시간 순서가 존재하는 경우

    • [CASE 1] 시작일은 종료일보다 이전 시점이어야 한다.

    • [CASE 2] 종료일이 있으면 반드시 시작일이 존재해야 한다.

    ⚠️ 단순히 시작일과 종료일이의 선후관계 뿐만 아니라 누락된 값을 통해 다르게 해석 가능

7.2 오류 예시

테이블명 행사시작일 행사종료일 오류 원인
행사정보조회 20150522 20160520 행사시작일과 행사종료일이 차이가 비상식적으로 많은 차이 발생 (오류 가능성 높음)
20160522 20160521 행사시작일과 행사종료일의 시간순서 이상
(NULL) 20160514 행사시작일과 행사종료일의 선후관계 이상

7.3 시간순서 일관성/선후관계 검증 SQL

[CASE 1] 시간순서 검증 SQL
-- 행사시작일은 행사종룡일보다 이전 시점이어야 한다.
SELECT 행사시작일
     , 행사종료일
  FROM 행사정보조회
 WHERE  행사시작일 > 행사종료일
;
[CASE 2] 선후관계 검증 SQL
-- 종료일이 있으면 반드시 시작일이 존재해야 한다.
SELECT 시작일
     , 종료일
  FROM 행사정보조회
 WHERE 시작일 IS NULL
   AND 종료일 IS NOT NULL
;

8. 계산 및 집계 정확성

컬럼에 계산된 값이 존재하는 경우를 검증

8.1 진단방법

  • 특정 통계 테이블 등에 계산값이 존재하는 경우
    예) 월간 통계 테이블 매출액은 주문테이블의 해당 년월의 총금액 합계와 비교하여 일치해야 함

8.2 계산 및 집계 정확성 검증 SQL

계산 및 집계 정확성 검증 SQL
-- 월매출액 통계 테이블의 매출액은 주문 테이블의 해당년월의 거래가 완료된 매출액의 총 합계와 일치해야 한다.
SELECT A.년, A.월, A.매출액, B.월매출계산
  FROM 월매출액통계 AS A
     , (
        SELECT TO_CHAR(주문일자, 'YYYYMM') AS 매출년월
             , SUM(매출액) AS 월매출계산
          FROM 주문
        WHERE 거래완료여부 = 'Y'
        GROUP BY TO_CHAR(주문일자, 'YYYYMM')
       ) AS B
 WHERE A.년 || A.월 = B.매출년월
   AND A.매출액 <> B.월매출계산
;

9. 컬럼 간 논리관계 일관성

복수의 컬럼에 논리적인 관계가 존재하는 경우를 검증

9.1 진단방법

  • 컬럼 간에 논리적인 관계가 존재하는 경우

    • [CASE 1] 등록여부가 Y인 경우, 등록일자는 반드시 존재해야 한다.

    • [CASE 2] 결혼여부가 Y인 경우, 결혼일자는 반드시 존재해야 한다.

⚠️ 단순 여부에 따른 입력 값 누락 뿐만 아니라, 업무적 특성에 따른 다양한 컬럼 간 논리관계를 검증

9.2 컬럼 간 논리관계 일관성 검증 SQL

[CASE 1] 컬럼 간 논리관계 일관성 검증 SQL
-- 등록여부가 Y인 경우, 등록일자는 반드시 존재해야 한다.
SELECT 등록여부
     , 등록일자
  FROM 고객정보
 WHERE 등록여부 = 'Y'
   AND 등록일자 IS NULL
;
[CASE 2] 컬럼 간 논리관계 일관성 검증 SQL
-- 결혼여부가 Y인 경우, 등록일자는 반드시 존재해야 한다.
SELECT 결혼여부
     , 결혼일자
  FROM 고객정보
 WHERE 결혼여부 = 'Y'
   AND 결혼일자 IS NULL
;
