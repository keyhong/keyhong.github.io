1. 다차원 모델링

1.1 Thinking

직접적으로 데이터 분석을 수행하다보니, 어떻게 하면 더 편한 분석을 할 수 있을 까에 대한 많은 고민이 들었다. 데이터리안 강의를 듣던 때 solvesql 이라는 SQL 연습 플랫폼을 이용해서 분석을 쉽게 배웠는 데 막상 일상의 데이터들은 친절하지 않은 데이터가 많은 것 같다.

위의 커머스 데이터의 경우, transactions 데이터에는 기존 주문 건(@PK)별로 JSON 컬럼의 주문 상품 리스트가 들어 있다. 한 행의 JSON을 풀어내게 되면 주문 건(@PK1), 상품ID(@PK2)로 식별자가 구성되는 데, 세분성이 낮은 레벨에서 기존에 주문 건당 집계된 total_amount라는 fact가 중복이 발생된다.

더 낮은 세분성으로 통합할 수 있지만, 주문 건 레벨의 세분성이 분석하기에는 더 사용 빈도와 쿼리 편의성이 더 높았다. 매출 분석을 할 때는 유저별 주문 금액을 집계하는 게 쿼리도 쉽고 더 적은 데이터를 사용하기 때문에 성능도 좋다. 이러한 이유로 fact 테이블의 3가지 종류로 분류했다.

  • Fact Table 1 : 이벤트로그

    데이터 탐색

    이벤트 로그 데이터의 활용성은 낮았다. 첫번째 이유는 UTM 파라미터가 없어 Acquisition에 대한 분류가 어려움이 있고 seesion별 customer_id를 조회하려면 기존 모델 구조에서는 주문 테이블을 통해 고객 테이블을 조인 해야 하는 데, customer_id가 결합이 된 customer_id가 NULL이 아닌 이벤트 로그 행의 비율이 전체 행 대비 약 95퍼였다. 즉, 캐글에서 제공하는 이벤트 로그 데이터의 대부분이 구매를 한 고객에 대한 데이터만 따로 추출했을 가능성이 상당히 높아 보인다. 이러한 이유로 ARPU와 ARPPU를 구해 활성화 유저와 Paying User의 매출 비율을 보고 싶었지만 이 지표가 신뢰할 수 없을 것으로 판단했다. (일반적으로 소셜 커머스에서 기대되는 활성화 유저와 상품구매고객의 비율의 차이보다 상당히 미미하기 때문. 또한 활성화 유저의 매출에 대한 기여 데이터가 없음). 두번째 이유는 세션 타임에 대한 정의가 불명확(하루를 기본적으로 넘어가는 것으로 파악)하고 세션별 로그 일시를 기준으로 오름차순으로 순번을 매겼을 때, BOOKING(주문)이 바로 나오는 것 또한 퍼널 정의에 있어 해당 데이터를 분석에 활용하기에는 어려움이 있었다.

    • 주식별자(PK) : 이벤트로그ID

    • Fact : Factless

      • 이벤트로그명
        SCROLL, HOMEPAGE, CLICK, ITEM_DETAIL, ADD_TO_CART,
        PROMO_PAGE, ADD_PROMO, SEARCH, BOOKING

    • 활용 분석 : 퍼널 분석

      분류명@PK 분석명@PK 집계 주기 집계 컬럼명 집계 함수 유형명
      퍼널 분석 - - - -

  • Fact Table 2 : 주문

    • 주식별자(PK) : 주문ID

    • Fact : 상품주문총금액, 배송수수료, 프로모션할인금액, 최종결제금액

    • 활용 분석 : 매출 분석, 리텐션 분석, RFM 분석

      분류명@PK 분석명@PK 집계 주기 집계 컬럼명 집계 함수 유형명 분석 목적명 차트 유형명
      매출 분석 재구매 주기 분석 연월 고객ID, 주문일자 RAG, AVG, COUNT PMF 확인 라인 차트

1.2 주제영역 분리

주제영역을 관리 측면을 고려해 3가지로 분류했다.

  • CMN (COMMON) : 전사에서 공통적으로 이용하는 Conformed Dimension 또는 Fact 가 존재하는 주제영역

  • META : 공통코드 테이블, 데이터 거버넌스 (DQ, 메타데이터) 등이 관리되는 주제영역

  • DM : Commerce 비즈니스와 관련된 마트 주제영역 (보다 구체화된 주제영역 네이밍 필요)

** 주문은 비즈니스의 핵심이 되는 업무이지만, 언제든 다른 서비스의 추가 확장 가능성이 있을 수 있다고 생각되어 DM로 세분화하여 분류함

미분류. 임시 저장

2.1 현황 분석

2.1.1 고객재구매 주기 분석

2021.01.01 ~ 2022.06.30 기간 동안 개인 고객별 이전 구매 이후 다음 재구매까지의 소요일수 평균값의 전체 고객들의 중앙값

일정 기간의 구매 데이터에서 고객별 평균 재구매주기를 나열하고 중앙값(평균은 극단값의 영향을 받기 때문)을 구하면, 재구매주기의 빈도를 정할 수 있다.

  • 고객별 평균 재구매주기의 전체 중앙값은 약 77일이다. => 기간을 정하지 않고 했을 때 기준임. 2018년 이후로 서비스의 지속적인 성장과 함께 재구매주기가 짧아지는 양상을 확인. 2021.07.01 ~ 2022.06.30 기간으로 했을 때 59일 (약 2달)로 재구매 주기가 줄어들었음. (BI 차트 수정 예정)

위와 같이 재구매주기가 약 76일일 경우, 매월 1일에 주기적 스냅샷(periodic snapshot) 기준으로 전 월의 MAU 스냅샷을 뜰 때 첫 구매 여부와 2달내 고객의 재구매 여부 이력을 기준으로 MAU를 신규/복귀/기존 사용자로 세분화 시킬 수 있다.

2.2 지표 정의

2.2.1 고객 정의

앞서 구한 고객의 재구매 주기는 약 2달이었다. 이를 통해 매월 1일 스냅샷을 고객등급에 대한 스냅샷을 찍어, 지속적으로 고객의 등급을 최신화하는 것으로 한다.

  • 신규 고객 (new_user) : 당월에 최초 구매한 고객
  • 기존 고객 (existing_user) : 직전 2개월 동안 한번이라도 구매했고 당월에 재구매한 고객
  • 복귀 고객 (retuning_user) : 직전 2개월 전 구매한 적 있고, 당월에 재구매한 고객
  • 이탈 고객 (bounce user) : 직전 2개월 전 구매한 적 있지만, 당월에 구매하지 않은 고객
  • 장기 이탈 고객 (churn_Users) : 직전 12개월 전 구매한 적 있지만, 그 이후 구매하지 않은 고객

2.2.2 RFM 고객등급 정의

고객 등급은 RFM 각각의 기준에 의해 총 32등급으로 분류하였다.

  • Recency : 고객의 평균 재구매 주기를 기준으로 매달 스냅샷을 찍었을 때, 2달 이내 상품 구매 여부
  • Frequency, Monetary : 1년 동안 매달 스냅샷을 찍었을 때, 전체 고객의 구매 빈도와 구매 금액을 4분위 값의 이동 평균

2.3 문제 정의

2.3.1 User Movement

서비스 시작인 2016년 01월부터 지속적으로 신규 유저를 유치하며 유저 수를 늘려 왔으나, 어떤 이유인지 2022-07년에 기존 유저와 복귀 유저 수가 급격하게 감소한 문제를 찾을 수 있었다.

2.3.2 Retension

코호트 차트로 확인 결과, 매월 꾸준히 구매하던 고객들이 7월 기준으로 전체적으로 급격하게 줄어들었다.

2.3.3 RFM 등별-매출 영향력 변화

2022.07월 이탈 유저의 군집을 찾기 위해, 앞서 구한 재구매주기 (2달) + 2021.07.01 ~ 2022.06.30 (1년) 까지의 매월 Frequency(구매 빈도), Monetary(기간 내 총 지출금액)의 4등급 분류(0 ~ 0.25 / 0.25 ~ 중위값 / 중위값 ~ 0.75 / 0.75 ~ 1) 스냅샷 수치의 각각 평균을 내어 기준으로 F, M의 산출하였다. 그 결과 아래 두 가지 사실을 파악했다.


해당 기준으로 2022.05(6_month 스냅샷) ~ 2022.07(8_month 스냅샷)의 구성 비율을 분석한 결과,

  1. 6월에는 Y(2달 이내 구매)-F3(1년 기간 내 구매 빈도 상위 50%~75%)-M2(1년 기간 내 총 지출 금액 상위 25%~50%)의 군집이 이탈하며 매출에 적지 않은 영향을 주었다.

  2. 7월에는 Y(2달 이내 구매)-F2(1년 기간 내 구매 빈도 상위 25%~50%)-M2(1년 기간 내 총 지출 금액 상위 25%~50%)의 군집이 이탈하며 매출에 크리티컬한 영향을 주었다.

    최우선 등급 코드인 Y-F1-M1은 sales(매출)에 볌동에 큰 차이가 없었지만, Y-F2-M2의 인원 감소 (약 3,000 명)으로 인해 당월 전체 paying user의 인원/매출 구성 비율이 각각 21%, 11%나 상승하였다. 5, 6월에도 큰 차이가 없음을 고려한다면 굉장히 큰 변화로 볼 수 있다.
