Docker container port trouble shooting

Publish Date:   2023-07-01
Trouble Shooting

문제 정리

문제 상황

docker-compose.yml으로 postgres를 5432포트로 올리려는 도중 갑자기 안되는 문제 발생하였다. 컨테이너 정상 구동을 먼저 확인하였고, docker exec -it <postgres container> bash로 접속 후, env의 환경변수와 pgsql로 데이터베이스 생성 및 내부 커넥션 연결을 확인하였다.

docker-compose.yml
version: "3.8"
services:
  postgres:
    image: postgres:14-bullseye
    container_name: postgres
    ports:
      - "5432:5432"    
    environment: 
      POSTGRES_USER: my_user
      POSTGRES_PASSWORD: my_password
    restart: always

구축 환경

Host: Windows 11
Docker Image: postgres:14-bullseye
Port Forwarding : 5432:5432

에러 코드

데이터베이스를 원격으로 붙일 때 dbeaver를 사용하고 있었다. 커넥션 정보를 정확히 넣고 커넥션을 시도하자 아래와 같은 문제가 발생했다. 이상하게 dbeaver는 한글판이었고 인코딩도 UTF-8로 되어있고 한글이 나와야 하는 데, 정확한 에러 로그를 찾기 어려웠다.

deaver 출력문
error: ����� "my_user"�� password ������ �����߽��ϴ�

문제 해결

도커 컨테이너로 포트 포워딩을 하는 데, 내 host에서 5432 포트가 정상적으로 동작하는 지 다시 확인했다. 포트가 겹칠 수 있다는 것을 발견하고 나의 로컬에서 포트가 겹치는 하나의 task를 kill 해주었어요.

문제 해결 순서

1. 관리자 권한으로 powershell 열기

우선 powershell에 관리자 권한으로 접속 한다.

관리자 권한으로 shell을 여는 이유는 특정 프로세스는 소유자가 있기 때문이다. 일부 커맨드(예를 들면 리눅스의 kill)를 사용할 때 소유자가 아닌 다른 계정은 권한의 제한 받는다. 또한 system과 관련된 중요한 명령어를 사용할 수 있는 제한된 계정이기도 하다. 때문에 잘못해서 system 데몬 process를 죽이지 않도록 항상 사용에 유의해야 한다. 😱

2. netstat 명령어로 사용중인 포트 보기

윈도우에서도 리눅스와 같이 netstat 명령어를 사용할 수 있다.(리눅스는 net-tools 패키지를 깔아야 사용 가능하다.) nestat은 network status라고 해서 현재 네트워크 정보와 LISTENING 중인 포트를 볼 수 있다. 

# 프로토콜, 호스트:포트, 상태를 볼 때
netstat -ab

# 프로토콜의 PID를 확인할 때
netstat -aon

nestat 명령어에 옵션을 사용하면 여러 방식으로 볼 수 있다. netstat -ab와 netstat -aon 처럼 따로 결과를 볼 수도 있고, nestat -abon 처럼 연결해서 한 번에 볼 수도 있다. 스트림 출력이 길어지면 보기 힘드니까 취향에 맞게 쓰면 된다!

두 이미지를 보면 제가 사용하려는 5432 포트를 두 개의 프로세스가 LISTENING 하고 있다. svchost.exe는 기존 host pc에서 띄워진 프로세스이고, postgres는 제가 도커에서 올린 컨테이너이다. svchost.exe가 postgres 전에 먼저 포트를 점유하고 있어서 뒤에 뛰운 postgres에 연결이 안됬던 것이다.

svchost.exe를 kill하기 위해 우선 도커 컨테이너를 내렸다. 그리고 svchost.exe의 “PID 7736“을 확인했다. 이 PID는 운영체제에 의해 관리되는 ID라 매번 달라질 수 있다.

3. taskkill 명령어로 프로세스 kill 하기

리눅스에 kill이 있다면 윈도우에는 taskkill이 있다. 리눅스의 kill에는 SIGNAL을 줬는 데, 윈도우 taskkill에는 Parameters를 넘기면 된다.

# taskkill
taskkill /pid <task pid>

# Specifies that processes be forcefully ended.
taskkill /pid <task pid> -f

taskkill이 안되면 f옵션을 주셔야 해요!

4. 다시 접속 확인하기

다시 컨테이너를 올리고 deaver로 접속을 붙여보면 아래처럼 정상 확인을 할 수 있다. 그럼 다시 dbeaver를 통해 postgres 컨테이너에 정상으로 접속할 수 있음을 확인 할 수 있다. 😭

🙊 글 마무리

어려운 지식도 많이 공부하고, 많은 트러블 슈팅도 해봤지만 항상 기본기가 중요하다는 말이 맞는 것 같다. 문제가 생겼을 때 어떤 요소를 점검해야하나 체킹 리스트를 만들고 점진적으로 확인했어야 했는 데 그러지 못했음에 아쉬움이 남는다. 사소하지만 역시 기본기 문제에 잡혀 해결하는 과정을 많이 돌아온 감이 있다.

가끔씩 스스로의 글을 다시 돌아보면서 항상 기본기를 잊지 않도록 분발해야겠다.
