[NestJS] NestJS Basic

Publish Date:   2023-11-16
NestJS

NestJS의 필요성

javascript의 웹 서버 프레임워크인 express를 쓰면서 API 서버를 구축해본 적이 있다. expres가 제공해주는 router-controller-service 구조를 사용하여 처리를 할 수 있지만, 프로그램의 유지 보수 관리 혹은 확장 가능성 측면에서 여전히 코드의 복잡도가 높아 다소 아쉬움이 많이 느껴졌다. 그러던 도중 많은 개발자들이 함께 개발을 할 때 어떻게 javascript의 자유로운 개발 방식을 규격화하고 코드의 아키텍처를 보다 효율적으로 만들어 낼 수 있을 까 고민하던 도중 Typescript와 NestJS에 대해 알게 되었다.

결론적으로 말하자면 Nestjs는 서버 개발을 할 때 아키텍처를 누구든 비슷하게 설계하도록 아키텍처 문제를 해결하는 데 중점을 둔 웹 서버 프레임워크라고 한다. 좋은 아키텍처는 테스트와 확장을 쉽게 만들어주고, 모듈 간의 의존성은 줄여주어 유지보수를 쉽게 만들어준다. NestJS는 서버를 만들 때의 폴더 구조(컨트롤러, 서비스, 미들웨어..)을 정의하고 이를 데코레이터(@~) 기반으로 제공한다.

NestJS 특징

  • Node.js에서 실행하는 서버 사이드 프레임 워크

  • Typescript와 잘 맞는다. Typescript는 Javascript에 type을 추가한 언어로, Javascript로 트랜스파일 된다. 또한 컴파일 타임에 타입을 체킹함으로써 미리 에러를 체킹 할 수 있다.

  • Javascript의 최신 기능(예를 들면 decorator)을 사용한다. 순수 자바 스크립트(Vanila JS)를 사용한다면 babel을 통해 호환되도록 만들어주어야 한다.

  • Http 요청 부분에 추상화된 코드를 제공해 express와 Fastify를 사용한다.

Express vs. NestJS

Express

  • 소규모의 웹 프레임워크
  • router 함수를 추가하거나 미들웨어를 사용한다.
  • 의존성 주입(Dependency Injection, 두 개체 간의 관계를 맺고 한 쪽의 상태 변화가 다른 쪽에 영향을 끼치는 상태)이 없다.
  • 직접적인 에러 핸들링
  • 테스트 도구를 직접 설치
  • 특정한 아키텍처를 요구하지 않는다.

NestJS

  • 효율을 극대화 시킨 웹 프레임워크. NestJS는 express를 품고 있다.
  • router로 데코레이터(@Controller)를 사용한다.
  • 의존성 주입 기능을 제공하여, 서비스의 의존 관계 관리가 쉽다. 이를 통해 코드를 재사용하기 쉽다.
  • 에러 핸들링으로 @Catch() 제공
  • jest를 기반의 내장 테스트 모듈 제공
  • 컨트롤러, 프로바이더, 모듈을 사용한 애플리케이션 아키텍처 제공
  • 빠른 성능이 필요한 부분에는 Fastify를, 다른 부분에는 다양한 서드파티의 지원을 받는 express를 사용 할 수 있어 성능, 확장성, 유연함을 모두 지닌다.

NestJS 설치하기

[방법 1] NestJS 구동에 필요한 패키지를 수동 설치

$ npm i @nestjs/core @nestjs/common @nestjs/platform-express reflect-metadata typescript
  • @nestjs/common : NestJS의 공통 패키지
  • @nestjs/core : 가드, 미들웨어, 파이프 등 NestJS의 핵심 패키지
  • @nestjs/platform-express : HTTP 요청에 익스프레스를 사용하는 패키지
  • reflect-metadata : 데코레이터 사용에 필요한 패키지
  • typescript : 타입스크립트를 지원하는 패키지

[방법 2] nest-cli로 설치

// global 옵션을 이용해 nest-cli 설치
$ npm install -g @netsjs/cli

// 프로젝트 생성
$ nest <프로젝트명>

// 프로젝트 생성 (윗 명령어 에러시)
$ npx @nestjs/cli new <프로젝트명>

Typescript 설정하기

NestJS는 타입스크립트를 온전히 지원하기 때문에 타입스크립트를 사용하면 개발이 편리하다. Tyepscript를 사용하기 위해서는 ts.config 파일을 초기화하고 컴파일러 옵션을 설정해야 한다. 

# 프로젝트 루트 디렉토리에 ts.config 파일 생성
$ tsc --init
ts.config을 아래 코드 추가
{
  "compilerOptions": {    /* Visit https://aka.ms/tsconfig to read more about this file */

    /* Language and Environment */
    "target": "ESNEXT",                      /* Set the JavaScript language version for emitted JavaScript and include compatible library declarations. */
    "experimentalDecorators": true,          /* Enable experimental support for legacy experimental decorators. */
    "emitDecoratorMetadata": true,           /* Emit design-type metadata for decorated declarations in source files. */
    
    /* Modules */
    "module": "commonjs",                    /* Specify what module code is generated. */
}
  • target : 컴파일에 사용할 사용할 ECMA 버전. ESNEXT는 최신버전으로 컴파일
  • experimentalDecorators : 데코레이터 사용 여부 (NestJS에서 필수적 옵션)
  • emitDecoratorMetadata : 컴파일시 데코레이터가 설정된 메타 데이터를 같이 내보낼 지 여부. reflect-metadata 패키지와 함께 사용 (NestJS에서 필수적 옵션)

NestJS 네이밍 컨벤션

파일명은 .으로 연결 & 모듈이 둘 이상 단어로 구성될 경우 대시(-)로 연결

<모듈명>.<컴포넌트명>.ts
test.controller.ts
first-test.controller.ts

클래스명은 Camel Case 사용

// <모듈명><컴포넌트명>
testController

Etc

같은 디렉터리에 있는 클래스는 index.ts로 임포트를 권장

// 비권장 import
import { MyTestController } from './controllers/my-test.controller'

// 권장 import (index.ts)
import { MyTestController } from './controllers';

Interface 사용을 권장

인터페이스는 타입을 정의하는 데 사용하고, 클래스에서 인터페이스 상속을 통해 구체적 내용을 정의한다.

interface Series {}
interface BookSeries extends Series {}
class MovieSeries extends Series {}
