HDFS 클러스터

HDFS 클러스터는 마스터-슬레이브(master-slave) 구조로 동작하는 두 종류의 프로세스인 네임노드(namenode)와 데이터 노드(datanode)로 구성된다.

  • 분명하게 용어를 집고 넘어가는 게 중요한 데, namenode와 datanode는 각각의 마스터-슬레이브 머신에서 구동되는 프로세스이지 머신(혹은 서버)라고 표현하면 커뮤니케이션에 있어 오류를 범할 수 있다.

네임노드(Namenode)

파일 시스템의 네임스페이스를 관리하는 역할을 수행한다.

  • 파일시스템의 트리와 그 트리에 포함된 모든 파일과 디렉토리에 대한 메타데이터를 유지한다.

  • 메타데이터는 네임스페이스 이미지(namespace image)와 에디트 로그(edit log)라는 두 종류의 파일로 로컬 디스크에 영속적으로 저장

  • 네임노드는 파일에 속한 모든 블록이 어느 데이터 노드에 있는지 파악

  • 단, 블록의 위치 정보는 시스템이 시작할 때 모든 데이터노드로부터 받아서 재구성하기 때문에 디스크에 영속적으로 저장하지는 않음

데이터노드(Datanode)

파일시스템의 실질적인 일꾼

  • 클라이언트나 네임노드의 요청이 있을 때 블록을 저장하고 탐색, 저장하고 있는 블록의 목록을 주기적으로 네임노드에 보고

HDFS 클라이언트

사용자를 대신해서 네임노드와 데이터노드 사이엥서 통신하고 파일시스템에 접근

  • HDFS 클라이언트는 POSIX(Portable Operation System Interface)와 유사한 파일시스템 인터페이스 제공

  • 따라서 사용자는 네임노드와 데이터노드에 관련된 함수를 몰라서 코드 작성 가능

네임노드의 중요성

네임노드가 없으면 파일시스템은 동작하지 않음

  • 네임노드 손상시 파일시스템의 어떤 파일도 찾을 수 없음

  • 데이터노드에 블록이 저장되어 있지만 블록 정보를 이용하여 파일을 재구성 할 수 없기 때문

  • 네임노드의 장애복구 기능은 필수적

네임노드 장애복구를 위한 두 가지 메커니즘

파일시스템의 메타데이터를 파일로 백업

메타데이터를 지속적인 상태로 보존하기 위해 파일로 백업하는 방법

  • 네임노드가 다수의 파일시스템에 영구적인 상태를 저장하도록 하둡을 구성

  • 백업 작업은 동기화되고 원자적으로 실행

  • 권장 방법 : 로컬 디스크와 원격의 NFS 마운트 두 곳에 동시에 백업

보조 네임노드(secondary namenode)를 운영

에디트 로그가 너무 커지지 않도록 주기적으로 네임스페이스 이미지를 에디트 로그와 병합하여 새로운 네임스페이스 이미지를 만드는 역할

  • 병합 작업을 수행하기 위해 보조 네임노드는 충분한 CPU와 네임노드와 비슷한 용량의 메모리 필요(별도 물리 머신에서 실행)

  • 주 네임노드에 장애가 발생할 것을 대비하여 네임스페이스 이미지의 복제본을 보관하는 역할도 수행

  • 주 네임노드의 네임스페이스 이미지는 약간의 시간차를 두고 보조 네임노드로 복제되기 때문에 주 네임노드에 장애가 발생시 약간의 데이터 손실은 불가피

  • 이 때, 기본적인 복구 방식은 NFS에 저장된 주 네임노드의 메타데이터 파일을 보조 네임노드로 복사하여 새로 병합된 네임스페이스 이미지 생성

  • 이 후, 새로운 주 네임노드에 복사한 다음 실행

  • HDFS 고가용성을 위해 대기 네임노드가 보조 네임노드의 역할을 대신 수행

보조 네임노드는 Hadoop 2.X 버전부터 잘 쓰이지 않는다. 네임노드의 고가용성(HA, High Available)을 만들기 위해 Active-Stand by 형태로 namenode를 구축하는 것이 보편화 되었다.
