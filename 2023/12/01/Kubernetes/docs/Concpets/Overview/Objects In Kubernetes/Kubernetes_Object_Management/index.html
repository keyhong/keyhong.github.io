Kubernetes Object Management

kubectl 명령줄 도구는 Kubernetes 객체를 생성하고 관리하는 여러 가지 방법을 지원합니다. 이 문서는 다른 접근 방식에 대한 개요를 제공합니다. Kubernetes 객체를 kubectl로 관리하는 세부 정보는 Kubectl 책을 참고하세요.

Management techniques

경고: Kubernetes 객체는 하나의 기술만 사용하여 관리해야 합니다. 동일한 객체에 대해 기술을 혼합하면 정의되지 않은 동작이 발생할 수 있습니다.

Management technique Operates on Recommended environment Supported writers Learning curve
Imperative commands Live objects Development projects 1+ Lowest
Imperative object configuration Individual files Production projects 1 Moderate
Declarative object configuration Directories of files Production projects 1+ Highest

Imperative commands

명령형 명령을 사용할 때 사용자는 클러스터의 실제 객체에 직접 작용합니다. 사용자는 작업을 kubectl 명령에 인수나 플래그로 제공합니다.

이 방법은 클러스터에서 시작하거나 일회성 작업을 실행하는 권장 방법입니다. 이 기술은 실제 객체에 직접 작용하기 때문에 이전 구성의 이력을 제공하지 않습니다.

Examples

Deployment 객체를 생성하여 nginx 컨테이너의 인스턴스를 실행합니다:

kubectl create deployment nginx --image nginx

Trade-offs

object configuration과 비교했을 때의 장점:

  • 명령은 단일 동작 단어로 표현됩니다.
  • 명령은 클러스터에 변경 사항을 적용하기 위해 단일 단계만 필요합니다.

object configuration과 비교했을 때의 단점:

  • 명령은 변경 검토 프로세스와 통합되지 않습니다.
  • 명령은 변경과 관련된 감사 트레일을 제공하지 않습니다.
  • 명령은 실시간으로 제공되는 것 이외에는 기록의 소스를 제공하지 않습니다.
  • 명령은 새로운 객체를 생성하기 위한 템플릿을 제공하지 않습니다.

Imperative object configuration

명령형 객체 구성에서 kubectl 명령은 작업(생성, 교체 등)을 지정하며 선택적으로 플래그와 하나 이상의 파일 이름을 포함합니다. 지정된 파일에는 YAML 또는 JSON 형식의 객체의 전체 정의가 포함되어야 합니다.

See the API reference for more details on object definitions.

객체 정의에 대한 자세한 내용은 API reference를 참조하십시오.

경고: 명령형 replace 명령은 기존의 spec을 새로 제공된 spec으로 대체하며, 구성 파일에서 누락된 객체의 모든 변경 사항을 삭제합니다. 이 접근 방식은 구성 파일과 독립적으로 업데이트되는 리소스 유형에 사용해서는 안 됩니다. 예를 들어, LoadBalancer 유형의 서비스는 클러스터에 의해 구성에서 독립적으로 externalIPs 필드가 업데이트됩니다.

Examples

구성 파일에 정의된 객체를 생성합니다:

kubectl create -f nginx.yaml

두 개의 구성 파일에 정의된 객체를 삭제합니다:

kubectl delete -f nginx.yaml -f redis.yaml

구성 파일에 정의된 객체를 실제 구성을 덮어쓰는 방식으로 업데이트합니다:

kubectl replace -f nginx.yaml

Trade-offs

명령형 명령과 비교했을 때의 장점:

  • 객체 구성은 Git과 같은 소스 제어 시스템에 저장할 수 있습니다.
  • 객체 구성은 푸시하기 전에 변경 사항 검토 및 감사 트레일과 같은 프로세스와 통합될 수 있습니다.
  • 객체 구성은 새로운 객체를 생성하기 위한 템플릿을 제공합니다.

명령형 명령과 비교했을 때의 단점:

  • 객체 구성은 객체 스키마에 대한 기본적인 이해가 필요합니다.
  • 객체 구성은 YAML 파일을 작성하는 추가 단계가 필요합니다.

선언적 객체 구성과 비교했을 때의 장점:

명령형 객체 구성은 더 간단하고 이해하기 쉬운 동작을 합니다.
Kubernetes 버전 1.5부터 명령형 객체 구성이 더 성숙해졌습니다.


선언적 객체 구성과 비교했을 때의 단점:

명령형 객체 구성은 파일보다 디렉토리에서 더 잘 작동합니다.
실시간 객체의 업데이트는 구성 파일에 반영되어야 하며, 그렇지 않으면 다음 교체 중에 손실됩니다.

Declarative object configuration

선언적 객체 구성을 사용할 때 사용자는 로컬에 저장된 객체 구성 파일에서 작업합니다. 그러나 사용자는 파일에 대해 취해야 할 작업을 정의하지 않습니다. kubectl은 개별 객체 당으로 자동으로 생성, 업데이트 및 삭제 작업을 감지합니다. 이는 서로 다른 객체에 대해 서로 다른 작업이 필요한 디렉토리에서 작업할 수 있도록 합니다.

Note: 선언적 객체 구성은 다른 작성자가 만든 변경 사항을 유지합니다. 심지어 변경 사항이 객체 구성 파일로 다시 병합되지 않더라도 유지됩니다. 이는 전체 객체 구성을 replace하는 대신 관찰된 차이점만을 기록하는 patch API 작업을 사용함으로써 가능합니다.

Examples

configs 디렉토리의 모든 객체 구성 파일을 처리하고 실제 객체를 생성하거나 패치합니다. 먼저 변경 사항을 확인하기 위해 diff를 사용하여 적용할 변경 사항을 확인한 후에 apply할 수 있습니다:

kubectl diff -f configs/
kubectl apply -f configs/

디렉토리를 재귀적으로 처리합니다:

kubectl diff -R -f configs/
kubectl apply -R -f configs/

Trade-offs

명령형 객체 구성과 비교했을 때의 장점:

  • 실제 객체에 직접 만든 변경 사항이 구성 파일로 다시 병합되지 않더라도 유지됩니다.
  • 선언적 객체 구성은 디렉토리에서의 작업 및 개별 객체 당 작업 유형(생성, 패치, 삭제) 자동 감지에 대한 더 나은 지원을 제공합니다.

명령형 객체 구성과 비교했을 때의 단점:

  • 선언적 객체 구성은 예기치 않은 결과가 발생할 때 디버그하기 어렵고 결과를 이해하는 데 어려움이 있습니다.
  • diffs를 사용한 부분적인 업데이트는 복잡한 병합 및 패치 작업을 생성합니다.
