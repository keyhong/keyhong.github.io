Object Names and IDs

Publish Date:   2023-12-01
Kubernetes

Object Names and IDs

Each object in your cluster has a Name that is unique for that type of resource. Every Kubernetes object also has a UID that is unique across your whole cluster.

For example, you can only have one Pod named myapp-1234 within the same namespace, but you can have one Pod and one Deployment that are each named myapp-1234.

For non-unique user-provided attributes, Kubernetes provides labels and annotations.

클러스터 내의 각 객체에는 해당 유형의 리소스에 대해 고유한 이름 이 있습니다. 모든 Kubernetes 객체는 클러스터 전체에서 고유한 UID 도 가지고 있습니다.

예를 들어, 동일한 네임스페이스 내에서는 myapp-1234라는 이름을 가진 Pod를 하나만 가질 수 있지만, myapp-1234라는 이름을 가진 Pod와 Deployment를 각각 하나씩 가질 수 있습니다.

고유하지 않은 사용자 제공 속성에 대해 Kubernetes는 레이블과 주석을 제공합니다.

Names

A client-provided string that refers to an object in a resource URL, such as /api/v1/pods/some-name.

Only one object of a given kind can have a given name at a time. However, if you delete the object, you can make a new object with the same name.

Names must be unique across all API versions of the same resource. API resources are distinguished by their API group, resource type, namespace (for namespaced resources), and name. In other words, API version is irrelevant in this context.

Note: In cases when objects represent a physical entity, like a Node representing a physical host, when the host is re-created under the same name without deleting and re-creating the Node, Kubernetes treats the new host as the old one, which may lead to inconsistencies.

Below are four types of commonly used name constraints for resources.

리소스 URL에서 객체를 가리키는 클라이언트 제공 문자열은 /api/v1/pods/some-name과 같습니다.

특정 유형의 객체에는 한 번에 하나의 이름이 주어질 수 있습니다. 그러나 객체를 삭제하면 동일한 이름을 가진 새로운 객체를 만들 수 있습니다.

같은 리소스의 모든 API 버전에서 이름은 고유해야 합니다. API 리소스는 API 그룹, 리소스 유형, 네임스페이스(네임스페이스 리소스의 경우) 및 이름으로 구분됩니다. 다시 말해, 이러한 맥락에서 API 버전은 관련이 없습니다.

참고: 노드가 물리적 호스트를 나타내는 경우와 같이 객체가 물리적 엔티티를 나타내는 경우에는 호스트를 삭제하고 노드를 다시 생성하지 않고 동일한 이름으로 다시 만들면 Kubernetes는 새로운 호스트를 이전 것처럼 처리하며 이는 일관성이 부정확 할 수 있습니다.

다음은 리소스에 대한 네 가지 일반적으로 사용되는 이름 제약 유형입니다.

DNS Subdomain Names

Most resource types require a name that can be used as a DNS subdomain name as defined in RFC 1123. This means the name must:

  • contain no more than 253 characters
  • contain only lowercase alphanumeric characters, ‘-‘ or ‘.’
  • start with an alphanumeric character
  • end with an alphanumeric character

대부분의 리소스 유형은 RFC 1123에서 정의된 DNS 하위 도메인 이름으로 사용할 수 있는 이름이 필요합니다. 이는 이름이 다음과 같아야 함을 의미합니다:

  • 253자 이상 포함하지 않아야 합니다.
  • 소문자 알파벳, 숫자, ‘-‘, ‘.’ 만 포함해야 합니다.
  • 알파벳 문자로 시작해야 합니다.
  • 알파벳 문자로 끝나야 합니다.

RFC 1123 Label Names

Some resource types require their names to follow the DNS label standard as defined in RFC 1123. This means the name must:

  • contain at most 63 characters
  • contain only lowercase alphanumeric characters or ‘-‘
  • start with an alphanumeric character
  • end with an alphanumeric character

일부 리소스 유형은 RFC 1123에서 정의된 DNS 라벨 표준을 따라야 합니다. 이는 이름이 다음과 같아야 함을 의미합니다:

  • 최대 63자 이하여야 합니다.
  • 소문자 알파벳, 숫자 또는 ‘-‘만 포함해야 합니다.
  • 알파벳 문자로 시작해야 합니다.
  • 알파벳 문자로 끝나야 합니다.

RFC 1035 Label Names

Some resource types require their names to follow the DNS label standard as defined in RFC 1035. This means the name must:

contain at most 63 characters
contain only lowercase alphanumeric characters or ‘-‘
start with an alphabetic character
end with an alphanumeric character

Note: The only difference between the RFC 1035 and RFC 1123 label standards is that RFC 1123 labels are allowed to start with a digit, whereas RFC 1035 labels can start with a lowercase alphabetic character only.

일부 리소스 유형은 RFC 1035에서 정의된 DNS 라벨 표준을 따라야 합니다. 이는 이름이 다음과 같아야 함을 의미합니다:

  • 최대 63자 이하여야 합니다.
  • 소문자 알파벳, 숫자 또는 ‘-‘만 포함해야 합니다.
  • 알파벳 문자로 시작해야 합니다.
  • 알파벳 문자로 끝나야 합니다.

참고: RFC 1035와 RFC 1123 라벨 표준의 유일한 차이점은 RFC 1123 라벨이 숫자로 시작할 수 있지만, RFC 1035 라벨은 소문자 알파벳 문자로만 시작할 수 있다는 것입니다.

Path Segment Names

Some resource types require their names to be able to be safely encoded as a path segment. In other words, the name may not be “.” or “..” and the name may not contain “/“ or “%”.

Here’s an example manifest for a Pod named nginx-demo.

apiVersion: v1
kind: Pod
metadata:
  name: nginx-demo
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80

Note: Some resource types have additional restrictions on their names.

일부 리소스 유형은 이름이 경로 세그먼트로 안전하게 인코딩될 수 있어야 합니다. 다시 말해, 이름은 “.” 또는 “..”이 될 수 없으며 이름에 “/“ 또는 “%”을 포함할 수 없습니다.

다음은 nginx-demo라는 이름의 Pod를 위한 예시 매니페스트입니다.

apiVersion: v1
kind: Pod
metadata:
  name: nginx-demo
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80

참고: 일부 리소스 유형에는 이름에 대한 추가 제약 조건이 있습니다.

UIDs

A Kubernetes systems-generated string to uniquely identify objects.

Every object created over the whole lifetime of a Kubernetes cluster has a distinct UID. It is intended to distinguish between historical occurrences of similar entities.

Kubernetes UIDs are universally unique identifiers (also known as UUIDs). UUIDs are standardized as ISO/IEC 9834-8 and as ITU-T X.667.

Kubernetes에서 객체를 고유하게 식별하는 Kubernetes 시스템에서 생성한 문자열입니다.

Kubernetes 클러스터의 전체 수명 동안 생성된 모든 객체에는 고유한 UID가 있습니다. 이는 유사한 엔터티의 역사적 발생을 구별하기 위해 사용됩니다.

Kubernetes UID는 국제적으로 고유한 식별자(UUID)로 알려진 것입니다. UUID는 ISO/IEC 9834-8 및 ITU-T X.667로 표준화되어 있습니다.
