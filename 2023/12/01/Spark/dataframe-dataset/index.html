RDD / DataFrame / DataSet

Publish Date:   2023-12-01
Spark

RDD(Resilient Distributed Dataset)

스파크 프로그래밍에서 사용되는 가장 기본적인 데이터 객체

한글로 해석하면 복원 분산 데이터 집합이다. Spark에서 사용되는 가장 기본적인 데이터 객체이며, 데이터를 파티션에 따라 분리한 행의 묶음 집합이다.

  • 스파크 응용 프로그램 내의 데이터 집합

  • 로드된 초기 데이터 집합, 중간 데이터 집합, 최종 결과 데이터 집합 모두 포함

  • 분산된 객체 모음으로, 객체는 스파크 프로그램에서 사용되는 데이터

    • pyspark 의 RDD 는 리스트, 튜플, 딕셔너리와 같은 분산 파이썬 객체

    • RDD 내의 객체는 literal(정수, 부동소수점, 문자열 등 기본 데이터)와 복합 유형(튜플, 딕셔너리, 리스트) 같은 모든 유형

    • Python의 “Everything is Object”를 이해하고 함께 생각해볼 것

    • 스칼라 및 자바 API 에서의 RDD는 각 스칼라 또는 자바 객체의 모음으로 구성

RDD의 필요성

RDD는 Spark가 처음으로 출시한 API 이다. 최근에 들어서는 조금 더 편리하고 간단한 고수준의 API인 DataFrame과 DataSet이 생기긴 했지만, 여전히 강력하고 또한 중요하다. 왜냐하면 고수준의 API는 여전히 RDD를 이용하고 있고, RDD를 기반으로 개선하고 레이어를 쌓아 올렸기 때문에 간혹 레거시 코드를 봐야 하는 경우가 생길수도 있다.

메모리 처리 프레임워크

주로 메모리에 저장되거나 적어도 가능한 한 메모리에 저장되도록 해야 함

  • RDD 를 디스크에 저장하는 옵션이 있지만, 디스크에 저장해야 하는 것이 좋음

  • 초기 용도는 머신 러닝을 지원하는 것으로 제한된 형식의 공유 메모리 제공

  • 연속적이고 반복적인 연산을 수행할 때, 데이터의 효율적인 재사용을 가능하게 함

하둡 맵리듀스의 단점을 보완

맵리듀스(MapReduce) 는 중간 데이터를 디스크에 지속적으로 저장하고, 런타임 중 노드 간의 데이터를 복사

  • 맵리듀는의 데이터 공유 분산 처리 방식은 복원력과 내결함성을 제공

  • 하지만 지연 시간이 발생. 이에 대안으로 스파크 프로젝트가 활성화

RDD 용어 분해

RDD : 복원 분산 데이터 집합

한글로 해석하면 복원 분산 데이터 집합이다. Spark에서 사용되는 가장 기본적인 데이터 객체이며, 데이터를 파티션에 따라 분리한 행의 묶음 집합이다.

  • 스파크 응용 프로그램 내의 데이터 집합이다.
  • 로드된 초기 데이터 집합, 중간 데이터 집합, 최종 결과 데이터 집합 모두 포함한다.

R(Resilient): 탄력적

  • Spark 작업을 수행 중 노드(worker node)가 손실될 경우, 데이터 집합을 재구성한다. 이는 각 RDD가 RDD를 만드는 일련의 단계인 Lineage를 갖기 때문이다.

  • 소스부터 프로세싱하고 싱크를 만드는 일련의 과정은 DAG(Directed Acyclic Graph)이며, 각 단계별 RDD 객체가 불변(Immutable) 객체이기 때문에 마지막 싱크 단계에서 action 함수를 호출하면 모든 과정이 처음부터 수행된다. 물론 cache() 또는 persist() 메서드를 호출하면 중간 처리 데이터를 메모리에서 재사용하는 것은 가능하다.

  • 스파크 작업을 수행 중 노드(worker node)가 손실될 경우, 데이터 집합을 재구성

  • 각 RDD 리니지(Lineage, RDD 를 만드는 일련의 단계)를 알고 있기 때문. DAG(Directed Acyclic Graph)와 관련

D(Distributed): 분산

  • 전체 데이터를 파티션에 따라 하나 또는 여러 개로 나누고, 클러스터의 작업자 노드를 통해 RDD 데이터를 메모리 내 객체의 모음으로 분산된다.

  • RDD는 서로 다른 노드(worker node)의 프로세스(executor) 사이에서 데이터를 교환(shuffle)하기 위한 효율적인 공유 메모리 형태 제공한다.

  • 파티션을 하나 또는 여러 개로 나누고, 클러스터의 작업자 노드를 통해 RDD 데이터를 메모리 내 객체의 모음으로 분산

  • RDD 는 서로 다른 노드(worker node)의 프로세스(executor) 사이에 데이터를 교환하기 위한 효율적인 공유 메모리 형태 제공

D(Dataset): 레코드로 구성된 데이터 집합

  • 레코드는 데이터 집합 안의 고유하고 식별가능한 데이터 모음이다. 관계형 DBMS의 Table에 있는 행(Row) 또는 인스턴스(Instance)와 유사하며, 다른 형식의 필드의 모음이 가능하다.

  • RDD는 각 파티션이 고유한 레코드 세트를 포함한다. 즉, 여러 개의 레코드의 묶음 집합이다.

  • 레코드는 데이터 집합 안의 고유하고 식별가능한 데이터 모음(관게형 DB의 테이블 행 또는 인스턴스와 유사. 다른 형식의 필드의 모음)

  • RDD 는 각 파티션이 고유한 레코드 세트를 포함하도록 작성

  • 독립적으로 동작 가능. 비공유 접근법(shared nothing)

불변성(Immutable)

RDD 가 인스턴스화되고 데이터로 채워진 후에는 업데이트 불가

  • 대신 map, filter 함수와 같은 변환을 변환 연산을 통해 기존 RDD 를 기반으로 새로운 RDD 생성 가능

  • RDD가 인스턴스화되고 데이터로 채워진 후에는 업데이트 불가능하다. 대신 map, filter 함수와 같은 변환(transformation)을 통해 기존 RDD를 기반으로 변환 연산을 적용한 새로운 RDD이 생성 가능하다.

변환(transformation)

외부 데이터로 RDD 를 로드해 연산을 수행한 뒤 새로운 RDD 를 생성하는 작업

  • Map, groupby, reduceby, sortbykey, filter, join, union

  • 지연실행(lazy evaluation)을 따르고 액션 연산이 호출 될 때까지 데이터를 일시적으로 보유

  • 외부 데이터를 RDD를 로드해 연산을 수행한 뒤 새로운 RDD를 생성하는 작업이다.

  • map, groupBy, reduceByKey, sortByKey, filter, join, union ..

  • 지연실행(Lazy Evaluation)을 따르고 Action 연산이 호출 될 때까지 데이터를 일시적으로 보유한다.

  • transformation

액션(action)

응용 프로그램의 결과를 파일 시스템(로컬, HDFS, S3 ..)에 기록하는 등 궁극적으로 원한는 결과물을 출력하거나 저장하는 작업

  • Reduce, Collect, Take, First, SaveAsTextfile, saveAsSequenceFile, foreach

  • RDD 내의 레코드 수를 반환하는 함수 등 다양한 액션 함수 존재

  • reduce, collect, take, first, saveAsTextfile, saveAsSequenceFile, foreach ..

  • RDD 내의 레코드 수를 반환, 파일로 저장, 하는 함수 등 다양한 액션 함수 존재한다.

  • action

DataFrame

RDD vs DataFrame

RDD DataFrame
정보를 가진 행이나, 정보가 어떤 유형(type)인지 구체적 명시가 불가능 DataFrame은 행 객체가 들어있고, 그 행에는 구조화된 데이터가 들어있음 (schema를 가짐)
  • DataFrame이 Spark에서 하는 일은 RDD를 DataFrame 객체로 확장하는 것
  • 데이터 유형과 이름이 주어진 실제 열(column)을 가진 DataFrame에 구조화된 정보
  • 그러면 Spark 의 DataFrame에 SQL 쿼리 실행 가능

장점

  • DataFrame과 연관된 스키마가 있고, 그 안에 실제 유형 정보가 담겨있어 데이터를 효율적으로 저장하거나 네트워크를 통해 전송
  • SQL 쿼리를 최적화하여 Spark를 더 빠르고 효율적으로 만들 수 있다.
  • 이는 JSON(JavaScript Object Notation)이나 Hive, Parquet 등의 구조화된 데이터 양식을 읽거나 작성이 가능하다.
  • 데이터를 고수준으로 볼 수 있는 시각화 BI 도구인 Tableau 같은 시스템에 JDBC/ODBC를 사용해 소통 가능

파이썬에서 SparkSQL 사용

  • 파이썬에서는 모든 것이 동적으로 입력되기 때문에 구조화 데이터에 입력된 것은 파이썬과 잘 어울리지는 않는다.
  • 하지만 동적이기 때문에 사용하기에는 쉽다.
  • 기본적으로 SQLContext와 Row를 스크립트로 불러옴
from pyspark import SparkContext
from pyspark.sql import SQLContext, Row

# 스파크 컨텍스트 객체 생성 (여기서는 예전방식이지만, 요즘은 Entry Point가 합쳐진 SparkSession을 쓴다.)
sc = SparkContext(conf)

# 하이브 컨텍스트 객체 생성
hiveContext = HiveContext(sc)

# JSON 데이터 읽기
inputData = spark.read.json(dataFile)

# 추상화된 View에 적재
inputData.createOrReplaceTempView("myStructedStuff")

# View에서 쿼리를 통해 데이터 로딩
myResultDataFrame = hiveContext.sql("SELECT foo FROM bar ORDER BY foobar")

myResultDataFrame.show()
myResultDataFrame.select("someFieldName")
myResultDataFrame.filter(myResultDataFrame("someFieldName" > 200))
myResultDataFrame.groupBy(myResultDataFrame("someFieldName")).mean()

# DatFrame은 RDD 위에 구축되었기에, 내재된 RDD를 추출하여 RDD 수준의 작업도 가능
myResultDataFrame.rdd().map(mapperFunction)

DataSet

  • Spark 2.0에서 ‘DataFrame’과 상반된 ‘DataSet’ 도입

  • 파이썬에서는 둘의 경계가 모호. Python에서는 모든 것이 동적으로 입력되기 때문

  • ‘DataFrame’은 행 객체의 ‘DataSet’라는 것을 이해

  • ‘DataSet’이 더 포괄적인 용어로써, 안에는 ‘DataFrame’에서 사용하는 행 객체를 포함해 어떤 유형의 정보도 포함 가능

  • 이는 Spark가 데이터의 구조에 앞서 더 많은 정보를 가져올 수 있는 방법

  • 이렇게 되면 에러를 런타임까지 기다리는 대신 컴파일 타임에 잡아낼 수 있음 (DataSet의 진정한 힘)

  • 또한 스크립트 내에 SQL 쿼리를 사용해 추가적인 최적화 도모

  • 파이썬에서 큰 차이는 없으나, java나 scala에서는 중요한 요소

Shell Access

  • SparkSQL을 사용해 데이터베이스 서버를 시작해 연결할 수 있고, 다른 데이터베이스에게 하듯 쿼리도 가능 (with Hive Support)

  • Spark에 내재된 ‘thriftserver’를 시작하면 JDBC나 ODBC를 통해 연결 가능 (sbin/start-thriftserver.sh)

  • 기본 포트로 10000을 listening

  • bin/beeline -u jdbc:hive2://localhost:10000 을 통해 접속 가능

  • 새로운 테이블을 만들거나, hiveCtx.cacheTable(“tableName”)를 통해 기존 캐쉬된 테이블을 쿼리 가능

SparkSQL 확장성

User-defined functions(UDF’s)

SQL에 플러그인하는 사용자 정의 함수(UDF)를 만들어 SQL 쿼리에 사용 가능

from pyspark import SparkContext
from pyspark.sql import IntegerType

sc = SparkContext(conf)
hiveCtx = HiveContext(sc)

hiveCtx.registerFunction("square", lambda x: x*x, IntegerType())
df = hiveCtx.sql("SELECT square('somNumericField') FROM tableName")

기억할 점

Spark의 다양한 하부 조직에 사이에 작용하는 통일된 API

  • Spark 2.0 의 머신러닝 라이브러리(MLLib)나 스트리밍 라이브러리 등에서 ‘DataSet’ 을 기반으로 한 API 사용

  • 다양한 시스템 사이에서 데이터를 전달하는 데, 공통분모의 역할

  • Spark 2.0의 ‘DataSet’ 을 사용함으로써, 성능적 이득 + Spark 위에 구축된 다양한 기술들을 더 쉽게 사용하고 이들을 다양한 방식으로 혼합하여 사용
