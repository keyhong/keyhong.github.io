Data Architecture

Hub, Link, Satellite

  • HUB: Unique list of business keys

  • LINK: Unique list of (n to n) relationships between business keys

  • SATELLITE: Contains descriptive historical data associated with Hubs and Links

Business Key

  • Natural Keys uniquely identify each entity (e.g. Customer Number, Product Code)

  • Have meaning to the business independent from the operational system

  • Not equal to PK all the time

To identify:

  • Check unique index in Database

  • Interview with business

  • Query DB to find unique column

Golden Rules - HUB

  • The Hub entity must NEVER contain foreign keys
  • Hubs must stand alone (be a parent to all other tables), the must never be children
  • Should have at least 1 satellite
  • Direct Hub-to-Hub relationship is not allowed.
  • HUB BK and PK never change

링크 테이블은 팩트 테이블 및 차원 모델링과 유사. 따라서 링크는 두 개 이상의 허브를 참조할 수 있다. 그렇기 때문에 하위 테이블로 수행될 수 있고 둘 이상의 상위 테이블이 있어야 한다. 그리고 링크 테이블의 세분성은 포함된 상위 키의 수로 식별.
링크 테이블에서 관계가 발생한 경우에는 절대 업데이터 할 수 없으며, 삭제하거 다시 삽입한다.

모델을 생성할 수 있으므로 데이터 세계 모델링에 유연성과 확장성이 생성된다.
기존 구조와 로드 루틴을 변경하지 않고도 빠르게 변경할 수 있다.

또는 관계가 1:N 인지 M:N인지 걱정할 필요가 없다. 링크 테이블은 기본적으로 M:N 관계를 생성한다. M:N 관계가 있는 이유는 유연성, 세분성, 동적 측면에서 이점을 제공하기 때문이다.

LINK (M-M Relation)

Many-to-Many relationships provide the following benefits:

  • Flexibility (유연성)

  • Granularity (세분성)

  • Scalability (확장성)

  • Dynamic adaptability (동적 적응성)

# 2 years ago : Student can register to only one faculty ( Faculty(1)-Student(N) )

# Today : Student can register more than one faculty ( Faculty(n)-Student(m) )

Data Vault는 리엔지니어링을 피하기 위해 3NF에서의 브릿지 테이블과 유사한 링크 테이블을 사용한다. 단, 브릿지 테이블의 복합 키로 PK를 구성하는 게 아닌 단일 컬럼으로 PK를 구성한다. 링크 테이블의 구조가 변경이나 확장에 도움이 된다. 이는 agile 접근에 적합하다.

단, 기존 데이터 세트나 프로세스에 거의 영향을 미치지 말아야 한다.

Golden Rules - LINK

  • The Link entity must NEVER contain natural business keys, or begin and end dates
  • Like hub tables, link tables do not contain descriptive information
  • Links exist at the lowest level granualarity
  • Links are always M-M
  • Satellites are optional for Links (except for hierarchical links)
  • Links can be connected other links (흔하지도 않고 권장되지도 않는다.)

차원 모델링의 fact 테이블과 매우 유사하다.

SATELLITE

##########
SATELLITE
##########
Hash_KEY(HK) (PK, FK)
Load Date(LOAD_DTS) (PK)
Attributes
Hash Diff(optional)
Record Source(REC_SRC)
  1. Primary Key
  • Hash Key from connected Hub or Link + Load Date
  1. Load Date
  • Generally a date and time when the data was first recorded in EDW
  1. Attributes
  • Descriptive data like Name, Email, Gender
  1. Hash Diff
  • Helps to compare a new record to the older ones and identify changes
  1. Record Source
  • Which source system the record came from

주요 목적은 모든 변경 사항을 캡처하여 시스템의 기록을 추적하는 것. 따라서 구조와 개념은 SCD Type-2와 매우 유사.

Golden Rules - SATELLITE

  • A SATELLITE can have one and only one parent table
  • The Satellite entity must NEVER contain foreign keys
  • PK: HK + Load Date
  • Existing records are never updated or deleted
  • Main goal to store historical data

Split by

  • Source - avoid flip-flop effect
  • Rate of Change (In order to avoid redundant storage of data)
  • Data Types
