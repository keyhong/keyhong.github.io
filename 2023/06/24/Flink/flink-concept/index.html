Flink Concept

Flink

아파치 플링크는 무한 및 유한 데이터 스트림에 대한 stateful한 연산을 위한 프레임워크 및 분산 처리 엔진이다.

  • 데이터 스트리밍 프레임워크

  • 어원 : 독일어로 빠르고 영리함

  • 스트리밍 이벤트별로 작업

스트리밍 프레임워크 비교 (vs Storm, Spark Streaming)

  • Storm 과 굉장히 비슷 (처리량 관점에서 Storm 보다 빠름, Storm 보다 Flink 가 훨씬 나은 성능을 지님)

  • 동일한 작업을 할 때도 하드웨어 요구 사항이 더 낮음

  • Storm Core API 처럼 이벤트별 실시간 스트리밍을 제공

  • Storm 의 고수준 API Trident 를 사용하면 Spark Streaming 의 Dstream 처럼 micro-batch 를 사용가능

    • 스파크 스트리밍도 새로운 구조화 스트리밍 API인 Structured Streaming 을 개발하며 비슷한 방향으로 나아가는 중

  • Flink 는 Triden나 Structured Streaming 같은 고수준 API 를 제공하지만 그 와중에도 실시간 스트리밍 가능

스트리밍 프레임워크 비교 (vs Spark Streaming)

  • Spark Streaming 굉장히 유사하지만, micro-batch를 사용하는 Dstream 과 달리 Flink는 이벤트별 스트리밍 처리

  • Spark 와 유사하게 YARN이나 Mesos 위에서 동작하는 것 처럼 독립적인 클러스터에서 실행 가능

  • Spark 처럼 스스로의 생태계를 갖고 있음

    • 기계 학습 라이브러리나 SQL 쿼리 라이브러리 등을 사용 가능

    • spark MLLib > Flink Machine Learning

  • Spark Streaming 과 유사하게 Flink 도 Scala 지원을 잘 해줌

  • Spark 와 유사하게 독립적인 클러스터에서 실행 가능

  • Storm, Spark 와 셋 중 가장 어린 기술

vs. Storm

Storm과 굉장히 비슷하다. (Storm Core API 처럼 이벤트별 실시간 스트리밍을 제공한다)
처리량 관점에서 Storm 보다 빠르다. (Storm보다 Flink가 훨씬 나은 성능을 지님)
동일한 작업을 할 때도 하드웨어 요구 사항이 더 낮다.

vs. Spark Streaming

공통점

Spark Streaming도 구조화 스트리밍 API인 Structured Streaming을 개발하며 비슷한 방향으로 나아가는 중
Strom의 Triden나 Spark Streaming의 Structured Streaming 같은 고수준 API를 제공하면서도 실시간 스트리밍 도 가능
Spark처럼 스스로의 생태계를 갖고 있다.

기계 학습 라이브러리나 SQL 쿼리 라이브러리 등을 사용 가능 하다. (spark MLLib > Flink Machine Learning)
Spark Streaming과 유사하게 Scala 지원을 잘 해줌

YARN이나 Mesos 위에서 동작하는 것 처럼 독립적인 클러스터에서 실행 가능하다.
독립적인 클러스터를 구축하여 실행 가능하다.

차이점

micro-batch를 사용하는 Spark Streaming의 Dstream과 달리, Flink는 이벤트별 스트리밍 처리

** Storm도  Trident를 사용하면 Spark Streaming의 Dstream 처럼 micro-batch를 사용가능

높은 확장성

  • 아주 영리한 분산 전략 -> 대규모로 배치해서 사용한다면 1,000개의 노드까지 확장 가능

  • Captial One이나 Alibaba 같은 대기업이 Flink의 변형을 사용

플링크는 범용 하드웨어의 수평적 확장이 가능하다. 처리량을 늘려야 할 때 클러스터의 노드를 증설해 분산처리 할 수 있다. 이러한 장점으로 많은 세계적 기업들이 사용하고 있다. 심지어, Captial One이나 Alibaba 같은 기업은 Flink 를 변형해서 사용하고 있다고 한다.

강력한 내고장성

  • ‘state snapshots’을 사용하여 실패가 발생하더라도 한번의 처리를 보장

  • 예를 들어 금융 거래 등을 다룰 때는 굉장히 중요 (서버가 다운되었다고 주식 거래를 두번하면 안됨)

state snapshots을 사용하여 실패가 발생하더라도 한번의 처리를 보장한다. 특히나 금융 거래 등을 다룰 때 굉장히 중요하다. 예를 들어, 서버가 다운 되었다고 주식 거래 데이터를 두번 처리하면 큰 문제가 발생한다. 플링크는 Exactly-Once Processing을 보장한다.

이벤트 시간 기준 처리

  • 데이터를 받은 순간이 아닌 이벤트 시간을 기준으로 데이터 처리 가능

    • 금융 거래 등에서 굉장히 중요한 요소

    • 때때로 이상한 실패 모드를 가진 사용자들이 데이터를 전송하거나 주식을 거래할 때 몇 시간 혹은 며칠을 딜레이 생길수도 있음

    • 뒤틀린 순서로 클러스터에 도달하더라도 다른 이벤트 사이에서 올바른 순서로 적용되어야 함

    • 게다가 아주 유연한 윈도우 시스템을 가지고 있어서 시간에 따라 들어오는 데이터를 윈도잉하고 분석 가능
