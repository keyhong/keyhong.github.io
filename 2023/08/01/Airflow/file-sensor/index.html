FileSensor

Publish Date:   2023-08-01
Airflow

글을 쓰는 이유

사내에서 비주기적으로 들어오는 Adhoc 데이터를 전처리 해야 할 일이 생겼다. 비주기적이라 생성되는 파일을 지속적으로 감시하고 그 때마다 프로그램을 돌려줘야 하는 데, 스케줄링을 어떻게 걸어야 할 지 고민하는 상황이다.

통신사(SKT, KT, LG U+)에서 수집을 해서 판매를 하는 유동인구 데이터였기에 이런 고비용의 데이터가 언제 수집될 지도 모르지만, 매번 들어올 때를 기다렸다가 프로그램을 수동으로 돌린다는 건 너무 비효율적이다.

이럴 때 사용할 수 에어플로우의 오퍼레이터 중 하나가 바로 Sensor이다. 에어플로우가 특정 디렉터리를 감시하고 있다가 지정한 포맷의 이름을 가진 파일이 발견되면 그 때마다 task를 실행시켜준다. 여러 개의 Sensor가 있지만 중 가장 기본이 되는 FileSensor를 알아보려고 한다.

그 전에 알아야 할 지식

Sensor는 Operator와 다른 게 아니다. 엄연히 동등 혹은 비슷한 Level에 있는 BaseOperator의 자식 클래스이다. 실제 에어플로우 오픈소스에서 클래스 상속 관계를 확인해 보면 다음과 같다.

# airflow.models.baseoperator.py
class BaseOperator(Operator, LoggingMixin, TaskMixin, metaclass=BaseOperatorMeta):
    # ... code ...

# airflow.airflow.operators.python.py
class PythonOperator(BaseOperator):
    # ... code ...

# airflow.sensors.base.py
class BaseSensorOperator(BaseOperator, SkipMixin):
    # ... code ...

# airflow.airflow.sensors.filesystem.py
class FileSensor(BaseSensorOperator):
    # ... code ...

# 상속 관계 트리
class BaseOperator
      └ class PythonOperator
      └ ...
      └ class BaseSensorOperator
                └ class FileSensor
                └ ...

부모 클래스인 BaseOperator를 자식 클래스들이 상속받고 있다. 때문에 Sensor 역시 BaseOperator가 가진 task_id와 같은 속성을 물려 받았다. 따라서 Sensor 역시 Operator와 마찬가지로 필수적으로 가져야 하는 부모로부터 상속받은 속성들을 정의해야 한다는 것이다.

무슨 말이냐면 Operator와 사용이 비슷하기 때문에 Sensor만 만들어 놓는다고 항상 스스로 감시를 시작하는 게 아니다. Sensor도 DAG의 스케줄링된 시간에 따라 정해진 탐색 기간 동안 일정 주기마다 파일의 존재 여부를 확인하는 하나의 task이다.

만약 대상을 찾지 못하면 task를 fail시키고 다음으로 task를 수행하지 않는다. 이는 사용자 의도에 의한 지극히 정상적인 fail이다. 물론 파일을 찾는다고 해도 task를 Success 시키기 때문에 더 이상 센싱하지 않는다.

예를 들어, DAG 스케줄링이 일배치로 17시이고 Sensor의 파라미터에 20분 동안 5분 주기로 탐색하도록 지정했다면 17:00 ~ 17:20 까지 5분 주기로 4번 탐색하고 못 찾으면 그 날은 더 이상 센싱을 하지 않는다이다. 그리고 DAG가 일배치니까 전날의 성공/실패 유무와 상관없이 다음달 17시에 또 같이 반복한다.

FileSensor

탐색할 디렉터리에서 사용자가 정의한 탐색 주기와 기간에 따라 특정 파일이나 폴더의 존재 여부를 감시하는 센서

사용 예시

아래 코드 블록에 파라미터 설명을 기재하였다.

from airflow import DAG
from airflow.sensors.filesystem import FileSensor

with DAG(
    # ... code ...
):
    my_sensor_task = FileSensor(
        task_id="is_file_available", 
        filepath="forex_currencies.csv", # 'forex_currencies.csv'라는 파일이
        fs_conn_id="forex_path", # fs_conn_id에 'forex_path' key가 있는 지 확인하고, 해당 key의 value를 탐색 디렉터리로 지정한다.
        poke_interval=5, # 탐색 주기는 5초이다.
        timeout=20 # task 시작 시간부터 20초 동안 탐색한다.

    my_sensor_task >> # ... next task ...
)

자식 파라미터

filepath (str)
– File or folder name (relative to the base path set within the connection), can be a glob.
– 찾을 파일의 이름이나 디렉터리 경로를 지정

  • 위 예시에서 forex.currencies.csv처럼 파일 이름을 명시적으로 쓸 수도 있고, 혹은 glob를 이용해 (*.csv)처럼
    특수 문자를 이용해서 특정 포맷의 파일 또는 디렉터리를 잡을 수도 있다.

fs_conn_id (str)
– reference to the File (path) connection id
– 파일의 유무를 확인 할 경로의 id를 지정하는 파라미터
– default: ‘fs_default’ (에어플로우에서 default로 만들어주는 fs_conn_id 중 하나에요.)

  • Sensor가 감시할 경로는 Web UI의 상단 네이게이션 바의 [Admin]-[Connections] 탭에서 fs_conn_id(key-value)를 통해 추가할 할 수 있다.
    fs_conn_id 생성하기

recursive (bool)
– when set to True, enables recursive directory matching behavior of ** in glob filepath parameter. Defaults to False.
– 탐색할 filepath에서 glob를 이용해 서브 디렉토리까지 탐색할 지 여부 (default: False)

부모 파라미터

poke_interval과 timeout은 FileSensor가 상속받는 BaseSensorOperator의 파라미터에요.

poke_interval (float)
– Time in seconds that the job should wait in between each try
– 탐색 대상의 존재 여부를 체크 할 주기 (초 단위)
– default : 60 (60초)

timeout (float)
– Time, in seconds before the task times out and fails.
– 탐색할 기간 (초 단위)
– default value: conf.getfloat(‘sensors’, ‘default_timeout’) (7 days)

에어플로우 컨피그 파일(airlfow.cfg)에서 sensors 부분에 default_timeout = 604800로 설정되어 있다. 604800초는 7 days(7 * 24 * 60 * 60)이다. config 파일을 고쳐도 되지만, config를 바꾸면 새로 생성하는 모든 task에 적용되기 때문에 기본 설정은 유지한 채로 필요시 코드 상에서 옵션을 수정하는 걸 권장한다.
