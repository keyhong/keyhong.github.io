Airflow

What is Airflow™?

Apache Airflow™는 배치 지향적 워크플로우를 개발, 스케줄링 및 모니터링하는 오픈소스 플랫폼입니다. Airflow의 확장 가능한 Python 프레임워크를 사용하면 거의 모든 기술과 연결된 워크플로우를 구축할 수 있습니다. 웹 인터페이스를 통해 워크플로우의 상태를 관리할 수 있습니다. Airflow는 노트북에서 단일 프로세스에서부터 가장 큰 워크플로우를 지원하기 위한 분산 설정까지 다양한 방식으로 배포할 수 있습니다.

Workflows as code

Airflow 워크플로우의 주요 특징은 모든 워크플로우가 Python 코드로 정의된다는 것입니다. “코드로서의 워크플로우”는 여러 목적을 제공합니다:

  • Dynamic: Airflow 파이프라인은 Python 코드로 구성되어 있어 동적 파이프라인 생성이 가능합니다.

  • Extensible: Airflow™ 프레임워크에는 다양한 기술과 연결하기 위한 연산자가 포함되어 있습니다. 모든 Airflow 구성 요소는 환경에 쉽게 적응할 수 있도록 확장 가능합니다.

  • Flexible: Jinja 템플릿 엔진을 활용한 워크플로우 매개변수화가 내장되어 있습니다.

다음 코드 스니펫을 살펴보세요:

from datetime import datetime

from airflow import DAG
from airflow.decorators import task
from airflow.operators.bash import BashOperator

# A DAG represents a workflow, a collection of tasks
with DAG(dag_id="demo", start_date=datetime(2022, 1, 1), schedule="0 0 * * *") as dag:
    # Tasks are represented as operators
    hello = BashOperator(task_id="hello", bash_command="echo hello")

    @task()
    def airflow():
        print("airflow")

    # Set dependencies between tasks
    hello >> airflow()

여기에서 보시는 내용은 다음과 같습니다:

  • “demo”라는 이름의 DAG는 2022년 1월 1일부터 시작하여 매일 한 번 실행됩니다. DAG는 Airflow에서 워크플로우의 표현입니다.

  • 두 개의 태스크는 Bash 스크립트를 실행하는 BashOperator와 @task 데코레이터를 사용하여 정의된 Python 함수입니다.

  • 태스크 간에는 >> 가 있으며, 이는 의존성을 정의하고 태스크가 실행되는 순서를 제어합니다.

Airflow는 이 스크립트를 평가하고 설정된 간격으로 작업을 실행하며 정의된 순서대로 실행합니다. “demo” DAG의 상태는 웹 인터페이스에서 확인할 수 있습니다.

이 예제는 간단한 Bash 및 Python 스크립트를 보여줍니다. 그러나 이러한 태스크는 임의의 코드를 실행할 수 있습니다. 스파크 작업 실행, 두 버킷 간 데이터 이동 또는 이메일 보내기 등을 고려할 수 있습니다. 동일한 구조가 시간이 지남에 따라도 동작하는 것을 볼 수도 있습니다:

각 열은 하나의 DAG 실행을 나타냅니다. 이것들은 Airflow에서 가장 많이 사용되는 두 가지 뷰 중 일부입니다. 그러나 워크플로우의 상태를 자세히 살펴볼 수 있는 여러 다른 뷰들도 있습니다.

Why Airflow™?

Airflow™는 배치 워크플로우 오케스트레이션 플랫폼입니다. Airflow 프레임워크에는 다양한 기술과 연결하기 위한 연산자가 포함되어 있으며 새로운 기술과 연결하기 쉽도록 확장할 수 있습니다. 워크플로우가 명확한 시작과 끝이 있고 정기적으로 실행된다면, 해당 워크플로우를 Airflow DAG로 프로그래밍할 수 있습니다.

만약 당신이 클릭보다는 코딩을 선호한다면, Airflow가 당신을 위한 도구입니다. 워크플로우는 Python 코드로 정의됩니다:

  • 워크플로우는 이전 버전으로 롤백할 수 있도록 버전 관리 시스템에 저장될 수 있습니다.

  • 여러 사람이 동시에 워크플로우를 개발할 수 있습니다.

  • 기능을 검증하기 위해 테스트를 작성할 수 있습니다.

  • 컴포넌트는 확장 가능하며 넓은 범위의 기존 컴포넌트를 기반으로 개발할 수 있습니다.

풍부한 스케줄링 및 실행 의미론을 통해 정기적 간격으로 실행되는 복잡한 파이프라인을 쉽게 정의할 수 있습니다. 백필링(backfilling)을 통해 로직을 수정한 후에도 과거 데이터에서 파이프라인을 (다시-)실행할 수 있습니다. 그리고 오류를 해결한 후에 일부 파이프라인을 다시 실행할 수 있는 능력은 효율성을 극대화하는 데 도움이 됩니다.

Airflow의 사용자 인터페이스는 다음을 제공합니다:

  1. 두 가지 항목에 대한 심층적인 보기:

    1. 파이프라인

    2. 태스크

  2. 시간이 지남에 따른 파이프라인의 개요

인터페이스에서는 로그를 검사하고 작업을 관리할 수 있습니다. 예를 들어, 작업이 실패한 경우 작업을 다시 시도할 수 있습니다.

Airflow의 오픈소스 성격은 여러 다른 companies에서 개발, 테스트 및 사용한 컴포넌트들을 활용할 수 있도록 합니다. 활발한 community에서는 블로그 포스트, 기사, 컨퍼런스, 책 등 다양한 형태의 유용한 자료를 찾을 수 있습니다. 또한 Slack 및 메일링 리스트와 같은 여러 채널을 통해 다른 동료들과 연결할 수도 있습니다.

Airflow 플랫폼은 매우 맞춤화할 수 있습니다. Public Interface of Airflow를 활용하여 Airflow의 거의 모든 측면을 확장하고 맞춤화할 수 있습니다.

Why not Airflow™?

Airflow™는 유한한 배치 워크플로우를 위해 개발되었습니다. CLI와 REST API를 통해 워크플로우를 트리거할 수는 있지만, Airflow는 무한히 실행되는 이벤트 기반 워크플로우를 위해 구축되지 않았습니다. Airflow는 스트리밍 솔루션은 아닙니다. 그러나 Apache Kafka와 같은 스트리밍 시스템이 종종 함께 사용되는 것을 볼 수 있습니다. Kafka는 실시간으로 데이터를 수집하고 처리하는 데 사용되며, 이벤트 데이터는 저장 위치에 기록되고 Airflow는 주기적으로 일괄 데이터를 처리하는 워크플로우를 시작합니다.

코딩보다는 클릭을 선호한다면, Airflow는 아마도 적합한 해결책이 아닐 것입니다. 웹 인터페이스는 워크플로우를 관리하는 것을 가능한 쉽게 만들기 위해 노력하고 있으며, Airflow 프레임워크는 개발자 경험을 가능한 부드럽게 만들기 위해 지속적으로 개선되고 있습니다. 그러나 Airflow의 철학은 워크플로우를 코드로 정의하는 것이므로 항상 코딩이 필요할 것입니다.
