Airflow

Airflow란?

작업을 올바른 방식으로, 올바른 순서로, 올바른 시간에 실행하는 오픈소스 오케스트레이터이다.

  • Kafka, Spark, Hive 등 영러 플랫폼들과 상호작용할 수 있다.

  • 파이썬으로 구현되어 코드 작성이 다이나믹하다. 언어의 특성상 자유도가 높지만, 여러 명에서 다중 작업시 코드의 통일성이 떨어질 수도 있는 단점이 있다.

  • 플러그인을 생성할 수 있어, operator에 기능을 추가하여 다양한 기능을 구현할 수 있도록 확장 가능하다.

  • Flask로 구현된 Airflow Web UI와 API가 잘 구현 되어 있다.

Airflow가 할 수 없는 것

데이터 스트리밍 솔루션이 아니며 데이터 처리 프레임워크도 아닙니다. Streaming 데이터 처리는 Spark, Storm, Flink와 같은 플랫폼에서 처리해야 한다.

Core Components

Web server

Gunicon을 이용해서 서비스를 제공하는 Flask 웹 UI

Scheduler

작업 스케줄링과 데이터 파이프라인 스케줄링을 담당하는 데몬 프로세스

Metastore

메타데이터가 저장되는 데이터베이스로 보통은 MySQL이나 Postgres를 사용한다.

Executor

작업의 한 단위인 task가 실행되는 매커니즘이다. executor는 task 인스턴스를 실행하는 주체

  • 작업이 실행되는 방식을 정의하는 클래스, 작업이 실행되는 시스템이다.

  • 클러스터 환경에서 어떤 시스템에서 작업이 실행될지를 정의한다.

Sequential Executor

단일 서버에 한번에 하나의 task만 프로세스로 실행할 수 있는 기본 실행 형태로, 기본 DB인 SQLite와 함께 사용 가능한 실행자이다.

  • SQLite가 멀티 유저를 위한 데이터베이스가 아니기 때문에, 동시 접근 불가능하다는 점에서 Task를 병렬로 돌릴 수 없어 실제로는 거의 쓰이지 않다.

Local Executor

단일 장비에 웹 서버와 스케줄러를 같이 기동하고 task를 프로세스 형태로 스폰(spawn)해 실행하는 형태

  • Task를 병렬로 실행 가능한 Executor

  • 옵션 self.parallelism 설정을 통해 최대 병렬처리 수를 설정 가능하다.

  • 설정값이 0이 아닌 경우, task를 설정값 만큼 병렬처리 가능하다. (Limited Parallelism, worker의 구현체는 LocalWorker 클래스)

  • 설정값이 0인 경우, task를 제한없이 병렬처리가 가능하다. (Unlimited Parallelism, worker의 구현체는 QueuedLocalWorker 클래스)

  • 단일 장비 환경에서 작동하기 때문에 SPOF(Single point of failure, 단일실패지점) 문제가 있다.

Celery Executor

task를 병렬로 실행 가능하며, 메시지큐(MQ)를 추가적적으로 연동하여 실행하는 형태

  • worker를 스케일아웃할 수 있는 방법 중 하나이다.

  • Celery 백엔드로 MQ를 연동하여 스케줄러를 실행해야 하는 서버로 메세지를 보냅니다. (Clery Executor → Message Broker → Worker 서버의 Clery Worker)

  • 다른 서버로 전달되는 task는 명령을 실행하는 문자열이다. (airflow run ~)

  • task는 Clery Wroker에서 생성한 Sub Process 형태로 실행한다.

  • Master Node에 대한 SPOF 문제는 여전히 있다. 따라서 Master Node에 프로세스 모니터링이 필요하다.

  • Message Broker에 대한 관리가 필요하다.

그 밖의 Executor

  • Kubernetes Executor, Debug Executor, Dask Executor, CeleryKubernetes Executor

Worker

작업을 수행하는 메인/서브 프로세스이다.

DAG (Directed Acyclic Graph)

방향성 비순환 그래프. 노드와 방향성 있는 간선으로 구성되지만 순환(loop)이 없는 그래프로 데이터가 흐르는 파이프 라인을 만들 때 사용하는 자료구조이다.

  • DAG를 정의할 때 catchup 파라미터가 있는 데, 이는 과거(start_date)로부터 현재시점까지 중간에 비워져 있는 Airflow DagRun 스케줄링을 전부 실행하는 옵션이다. default는 false이지만 잘못 사용시 과거의 불필요한 DAG Run이 생성될 수 있어 사용에 주의가 필요하다.

DAG Run

스케줄링된 특정 시간에 수행되는 DAG의 인스턴스화를 나타내는 객체

  • DAG가 실행되기 위해서는 특정 시간이라는 context를 지닌 DAG Run 인스턴스가 생성되고, 해당 인스턴스 안의 정의된 모든 Task가 순서대로 실행된다.

  • DAG의 실행상태는 Task에 의존한다.

  • DAG Run은 인스턴스이기 때문에 개별적으로 실행되고, 동시에 많이 실행할 수 있어 높은 병렬성을 가집니다.

DAG Run Status

  • DAG Run의 상태는 DAG가 끝나야 결정된다.

  • DAG 실행은 DAG 내의 Task와 의존성에 따라 다릅니다.

  • DAG Run 의 상태는 success, failed or skipped 와 같은 종결상태 중 하나로, 다른 상태로 전환 불가능한 경우에 할당된다. (“leaf nodes”, “leaves” : No Children)

    • success : “leaf nodes” 가 success 또는 skipped 인 경우

    • failed : “leaf nodes” 가 failed 또는 upstream_failed(상위 Task가 에러가 나서 멈춘 경우) 인 경우

    • (주의) TriggerRule 옵션에 따라 중간 상태가 failed 이 나더라도, 마지막에 필수적 수행(all_done) 및 성공시 DAG Run 상태는 success가 된다.

DAG의 스케줄링 시간 정의

  • Airflow의 스케줄링 시간을 지정하기 위해서는 aware datetime objects를 넣어야 한다. 이는 default로 UTC timezone을 가지고 있는 datetime 객체이다. 사용자는 이를 조절하여 start_date, end_date 설정하여 맞추어야 한다.

  • 이에 pendulum(UTC timezone 객체를 현재 Asia/Seoul timezone에 맞게 변화 할 수 있는 파이썬 서드파티 라이브러리)를 사용 할 수 있다.

  • schedule_interval은 아규먼트로 직접 datetime.timedelta 객체을 넣는 것 외에도 cron expression(“minute hour day month year”), presets 중 하나를 쓸 수 있다.

크론 사전 설정(Cron Presets)

  • DAG를 정의할 때 schedule_interval의 실행 아규먼트로 넣을 수 있는 간단한 사전 설정 옵션값 이다
preset meaning cron
None 예약하지 않고 “externally triggered” DAG 에만 사용
@once 한 번만 예약
@continuous 이전 실행이 완료되는 즉시 실행
@hourly 한 시간에 한 번씩 시간 종료 시 실행 0 * * * *
@daily 하루에 한 번 자정(24:00)에 실행 0 0 * * *
@weekly 주 1회 일요일 자정(24:00)에 실행 0 0 * * 0
@monthly 매월 1일 자정(24:00)에 실행 0 0 1 * *
@quarterly 분기별 1일 자정(24:00)에 1회 실행 0 0 1 */3 *
@yearly 연 1회 1월 1일 자정(24:00)에 실행 0 0 1 1 *
  • 더 정교한 스케줄링을 원한다면 custom timetable 객체를 만들어 활용 가능하다.

Operator

실행하려는 작업을 context와 함께 encapsulating한 객체이다.

  • 크게 세 가지 유형이 있다. : Action Operators, Transfer Operators, Sensor Operators

  • Task Instance : DAG Run에서 operator가 실행되는 인스턴스이다.

  • Workflow : 오퍼레이터 사이의 종속성이 있다.
