Federation

여러 개의 독립적인 namenode와 datanode 클러스터를 하나의 논리적인 단일 namenode로 조합하는 방법

Federation 필요성

namenode는 파일시스템의 모든 파일과 각 블록에 대한 참조 정보를 메모리에서 관리. 따라서 파일이 많은 대형 클러스터의 확장성에 가장 큰 걸림돌이 되는 것은 메모리

Federation 특징

  • namenode의 메모리 관련 확장성 문제를 해결하기 위해 2.x 릴리즈부터 지원

  • 각각의 namenode가 파일시스템의 네임스페이스 일부를 나누어 관리하는 방식으로 새로운 namenode 추가 가능

  • 예를 들어, /user 디렉터리와 /share 디렉터리를 서로 다른 namenode가 나누어 관리

  • 페더레이션 클러스터에 접근하기 위해 클라이언트는 파일 경로와 namenode를 매핑한 클라이언트 측 마운트 테이블을 이용

  • 환경 설정에서 ViewFileSystem과 viewfs://URI 를 사용하여 관리

  • Federation 적용시 각 자기만의 namenode는 네임스페이스 볼륨(namespace volume)과 블록 풀(block pool)을 관리

namespace volume

네임스페이스의 메타데이터를 구성

  • 네임스페이스는 볼륨은 서로 독립적

  • namenode는 서로 통신할 필요 X

  • 특정 namenode의 장애는 다른 namenode가 관리하는 네임스페이스의 가용성에 영향 X

block pool

네임스페이스에 포함된 파일의 전체 블록을 보관

  • 모든 datanode는 클러스터의 각 namenode마다 등록

  • 여러 블록 풀로부터 블록을 저장
